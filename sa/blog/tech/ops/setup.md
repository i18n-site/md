## गुप्तशब्द-रहित-प्रवेशं विन्यस्यताम्

कतिपयनिमेषेभ्यः अनन्तरं सर्वराः सर्वे सक्रियताम् अवाप्तवन्तः ।

प्रथमं वस्तु IP स्थानीययन्त्रस्य `~/.ssh/config` सम्पादयित्वा उपनाम सेट् कर्तुं, ततः गुप्तशब्द-रहित-प्रवेशं विन्यस्तुं `ssh-copy-id root@c2` उपयोगं कुर्वन्तु ।

![](https://i-01.eu.org/1718287198.avif)

आरम्भविन्यासं शीघ्रं पूर्णं कर्तुं भवान् एतां [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) परिवर्तयितुं शक्नोति ।

गुप्तशब्द-रहित-विन्यासस्य समाप्तेः अनन्तरं भवान् [pdsh](https://github.com/chaos/pdsh) उपयोगं कर्तुं शक्नोति It can be operated in batchs, which is convenient.

प्रथमं ipv6 इत्येतत् मैन्युअल् रूपेण सक्षमं कृत्वा परीक्षणं करणीयम्

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

ततः एकेन क्लिक् करणेन प्रणालीं ubuntu 24 इत्यत्र उन्नयनं कृत्वा सामान्यतया प्रयुक्तं सॉफ्टवेयरं संस्थापयन्तु ।

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 इत्यस्य अर्थः भाषां चीनीभाषायां, समयक्षेत्रं बीजिंगभाषायां परिवर्तनं च ।

पूर्वनिर्धारितरूपेण संस्थापितं सॉफ्टवेयरं मम व्यक्तिगत-उपयोग-अभ्यासानां अनुसारं विन्यस्तं भवति, यथा [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) , इत्यादयः आवश्यके सति [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) गोदामस्य प्रतिलिपिं कृत्वा आवश्यकतानुसारं परिवर्तनं कर्तुं शक्नुवन्ति

एतत् पदं बहुकालं यावत् भवति, ततः परं सामान्यतया प्रयुक्तस्य सॉफ्टवेयरस्य उन्नयनं, संस्थापनं च सफलं भविष्यति । भवान् निम्नलिखित-आदेशेन एतत् पुष्टिं कर्तुं शक्नोति ।

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## मोश वेगं करोति ssh

यूरोपतः चीनपर्यन्तं जालम् स्थिरं [mosh](//mosh.org) `ssh` .

स्थानीयनिर्देशिकायां `~/.bin` (मुख्यतया वातावरणचर `PATH` मध्ये योजितम्) मध्ये `c0` स्क्रिप्ट् निम्नलिखितरूपेण रचयन्तु ।

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

निर्माणानन्तरं कार्यान्वयनीय-अनुमतिः, सॉफ्ट-लिङ्क् `c1` तथा `c2` प्रदातुम् ।

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

भविष्ये `mosh` उच्चगतिप्रत्यक्षसंयोजनस्य उपयोगाय प्रत्यक्षतया `c0` तथा `c1` प्रविष्टुं शक्नुवन्ति तथा च [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## ext4 तः btrfs मध्ये विभाजनस्वरूपं परिवर्तयन्तु

[btrfs](https://wikipedia.org/wiki/Btrfs) सुलभबैकअपार्थं डिस्कस्नैपशॉट् ग्रहीतुं शक्नोति।

प्रथमं उद्धारव्यवस्थां प्रविशन्तु

![](https://i-01.eu.org/1718300448.avif)

Debian Rescue इति चिनोतु

![](https://i-01.eu.org/1718300753.avif)

सर्वरस्य पुनः आरम्भं प्रतीक्ष्यताम्

ssh अङ्गुलिचिह्नपरिवर्तनं भविष्यति, तत् स्वच्छं कर्तुं `ssh-keygen -R ip` उपयोगं कुर्वन्तु

ततः विभाजनं परिवर्तयितुं निम्नलिखितस्क्रिप्ट् चालयन्तु

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

परिवर्तनस्य सफलतायाः अनन्तरं स्क्रिप्ट् स्वयमेव प्रणालीं पुनः आरभेत ।

### btrfs संपीडन अनुपातं पश्यन्तु

पूर्वनिर्धारितरूपेण स्वयमेव संपीडयिष्यति ।

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### मैनुअल संपीडन

यदि भवान् सिद्धिवादी अस्ति तर्हि पूर्ण डिस्क संपीडनं स्वहस्तेन अपि कर्तुं शक्नोति आदेशः निम्नलिखितरूपेण अस्ति ।

```bash
btrfs filesystem defragment -r -czstd /
```

यथा अधोलिखिते चित्रे दर्शितं, भवन्तः द्रष्टुं शक्नुवन्ति यत् नूतने प्रणाल्यां हस्तचलितसंपीडनस्य अनन्तरं प्रायः 100MB अधिकं संपीडितम् आसीत् ।

## उद्धारव्यवस्थायाः विवरणम्

उद्धारप्रणालीं 22 पोर्ट् [tcping](https://github.com/paradiseduo/tcping) अभिगम्यमानस्य अनन्तरं 2-3 निमेषान् यावत् सेट् गुप्तशब्देन तत् अभिगन्तुं न शक्नोति केवलं क्षणं प्रतीक्ष्यताम् ।

```
tcping ip 22 -c 9999
```

पोर्ट्-सुलभतायाः निरीक्षणं कर्तुं शक्यते ।

### Rescue प्रविष्टस्य अनन्तरं हार्डड्राइव् कथं प्रवेशः भवति Restore Snapshot &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

भवन्तः स्नैपशॉट् द्रष्टुं शक्नुवन्ति, उदाहरणार्थम्

```
snapper -c etc list
```

स्नैपशॉट् सञ्चिकापरिवर्तनं पश्यन्तु

```
snapper -c etc status 5..6
```

विशिष्टं diff पाठं पश्यन्तु

```
snapper -c etc diff 3..5
```

स्नैपशॉट् पुनःस्थापयन्तु: उदाहरणार्थं, स्नैपशॉट् 5 इत्यस्य अनन्तरं सर्वाणि परिवर्तनानि पूर्ववत् कुर्वन्तु

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### लॉग स्थायित्वं सक्षमं कुर्वन्तु

उद्धारप्रणाल्याः अन्तर्गतं लॉग-स्थायित्वं सक्षमीकरणाय भवता तत्सम्बद्धं निर्देशिकां स्वहस्तेन निर्मातुं आवश्यकं भवति तथा च केचन अतिरिक्तपदानि कर्तुं आवश्यकं भवति, यतः भवतः समीपे systemd सेवा प्रचलति नास्ति अत्र भवन्तः प्रयतितुं शक्नुवन्ति सोपानानि सन्ति:

प्रथमं चोत्, तर्हि

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

स्वस्य उबण्टु-प्रणाल्यां पुनः आरम्भं कुर्वन्तु: उपर्युक्तं परिवर्तनं कृत्वा, भवन्तः स्वस्य यन्त्रं सामान्यतया पुनः आरभणीयम्, उद्धार-प्रणाल्याः स्थाने स्वस्य उबण्टु-प्रणाल्यां च पुनः आरभणीयाः ।

एवं प्रकारेण systemd-journald नूतनविन्यासेन आरभ्य हार्डडिस्कमध्ये लॉग्स् निरन्तरं संग्रहीतुं आरभेत ।

उपर्युक्तानि कार्याणि उद्धारप्रणाल्यां क्रियन्ते इति मनसि धारयन्तु, अतः ते परिवर्तनानि यावत् प्रणाली पुनः आरभ्यते, उद्धारविधाने न भवति तावत् प्रभावं न प्राप्नुयुः

अस्य अर्थः अस्ति यत् भवन्तः तत्क्षणमेव विद्यमान-लॉग्स्-मध्ये किमपि परिवर्तनं न पश्यन्ति ये स्थायित्वं प्राप्नुवन्ति, परन्तु तदनन्तरं उत्पन्नाः सर्वे लॉग्-पत्राणि स्थायित्वं प्राप्नुयुः ।

एवं भवन्तः ssh सेवायाः लॉग्स् द्रष्टुं शक्नुवन्ति

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## कठिनजटिलरोगाणां विषये टिप्पण्यानि

### विचित्रसमस्यायाः समाधानं कुर्वन्तु: सर्वरः ssh कर्तुं न शक्नोति

मया पूर्वं स्वचालितसंकलनं परिनियोजनस्क्रिप्ट् च कर्तुं github क्रियायाः उपयोगः कृतः, परन्तु परिनियोजनसेवा ssh न भवितुम् अर्हति इति ज्ञातम् ।

उद्धारप्रणालीं प्रविश्य लॉग्स् पश्यन् अनुमतिविषयः अस्ति:

`Missing privilege separation directory: /run/sshd`

भवान् निम्नलिखित आदेशेन एतां निर्देशिकां निर्मातुम् अर्हति

```
chown root:root /
systemd-tmpfiles --create
```

पुनः पुनः अन्वेषणस्य अनन्तरं github action tar इत्यस्य User `uid` `1001` अस्ति

tar `--no-same-owner` `uid` `1001`

फलतः, rsync मूलनिर्देशिकायाः उपयोक्तृसमूहं परिवर्तयिष्यति, ततः पुनः आरभ्य ssh मध्ये प्रवेशं कर्तुं न शक्ष्यति ।
