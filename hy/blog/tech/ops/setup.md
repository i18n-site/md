## Կարգավորել Առանց Գաղտնաբառի Մուտքը

Մի քանի րոպե անց բոլոր սերվերները ակտիվացան:

Առաջին բանը տեղական մեքենայի `~/.ssh/config` խմբագրելն IP ` անուն դնելու համար, այնուհետև օգտագործել `ssh-copy-id root@c2` ՝ առանց գաղտնաբառի մուտքը կարգավորելու համար:

![](https://i-01.eu.org/1718287198.avif)

Դուք կարող եք փոփոխել այս [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) ՝ սկզբնավորման կոնֆիգուրացիան ավելի արագ ավարտելու համար:

Առանց գաղտնաբառի կազմաձևումն ավարտվելուց հետո կարող եք օգտագործել [pdsh](https://github.com/chaos/pdsh) խմբաքանակով, ինչը հարմար է:

Առաջինը ipv6-ը ձեռքով միացնելն ու փորձարկելն է

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

Այնուհետև մեկ սեղմումով արդիականացրեք համակարգը մինչև ubuntu 24 և տեղադրեք սովորաբար օգտագործվող ծրագրակազմը:

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 նշանակում է լեզուն անցում չինարենի, իսկ ժամային գոտին՝ Պեկին:

Լռելյայն տեղադրված [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) կազմաձևված է ըստ իմ անձնական օգտագործման սովորությունների, ինչպիսիք [direnv](https://direnv.net/) [mise](https://github.com/jdx/mise)

Այս քայլը երկար ժամանակ է պահանջում Կես ժամ սպասելուց հետո սովորաբար օգտագործվող ծրագրաշարի թարմացումը և տեղադրումը հաջողությամբ կանցնեն: Դուք կարող եք դա հաստատել հետևյալ հրամանով.

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## Մոշը Արագացնում Է Սշ

Ցանցը Եվրոպայից մինչև Չինաստան կայուն չէ: Խորհուրդ է տրվում օգտագործել `ssh` [mosh](//mosh.org)

Ստեղծեք `c0` սկրիպտ տեղական գրացուցակում `~/.bin` (հիմնականում ավելացված է շրջակա միջավայրի փոփոխական `PATH` ին) հետևյալ կերպ.

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

Ստեղծվելուց հետո տրամադրեք գործարկվող թույլտվություններ և soft-link `c1` և `c2` :

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

Ապագայում կարող եք ուղղակիորեն մուտքագրել `c0` և `c1` `mosh` գերարագ ուղիղ միացում օգտագործելու համար և մուտքագրել [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## Փոխարկել Բաժանման Ձևաչափը Ext4-Ից btrfs

Կարող [btrfs](https://wikipedia.org/wiki/Btrfs) նկարահանել սկավառակի լուսանկարները հեշտ կրկնօրինակման համար:

Նախ մտեք փրկարարական համակարգ

![](https://i-01.eu.org/1718300448.avif)

Ընտրեք Debian Rescue-ը

![](https://i-01.eu.org/1718300753.avif)

Սպասեք, մինչև սերվերը վերագործարկվի

ssh մատնահետքերի փոփոխություններ, այն մաքրելու համար օգտագործեք `ssh-keygen -R ip`

Այնուհետև գործարկեք հետևյալ սկրիպտը՝ բաժանումը փոխարկելու համար

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

Փոխակերպումը հաջողությամբ ավարտվելուց հետո սկրիպտը ավտոմատ կերպով կվերագործարկի համակարգը:

### Դիտեք btrfs Սեղմման Հարաբերակցությունը

Այն ավտոմատ կերպով կսեղմվի լռելյայնորեն Սեղմման հարաբերակցությունը ստուգելու հրամանը հետևյալն է.

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### Ձեռքով Սեղմում

Եթե դուք պերֆեկցիոնիստ եք, կարող եք նաև ձեռքով կատարել սկավառակի ամբողջական սեղմում Հրամանը հետևյալն է.

```bash
btrfs filesystem defragment -r -czstd /
```

Ինչպես ցույց է տրված ստորև նկարում, դուք կարող եք տեսնել, որ նոր համակարգում ձեռքով սեղմելուց հետո սեղմվել է ավելի քան 100 ՄԲ:

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### Ստեղծեք Btrfs Snapshot-Ի Կրկնօրինակում Snapper-Ով

Btrfs կրկնօրինակման լավագույն գործընկերն [snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html) :

Նախաստորագրման սցենարը կստեղծի `~/ubuntu` գրացուցակը, պարզապես գործարկեք հետևյալ հրամանը.

```
~/ubuntu/snapper/init.sh
```

Կհանդիպենք սցենարի բովանդակության մեջ [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

Օգտագործեք `snapper list-configs` ՝ դիտելու ենթածավալները, որոնց համար դրված են ակնթարթները:

![](https://i-01.eu.org/1718302689.avif)

Օգտագործեք `snapper -c mnt list` ենթահատորի պատկերը դիտելու համար

`/etc/snapper/configs/mnt` կարող է դիտել այս ենթահատորի հատուկ պահուստավորման կարգավորումները

Լրացուցիչ հղումներ.

[Հեշտությամբ խաղացեք Btrfs -ի snapshot ֆունկցիայի հետ snapper](https://zhuanlan.zhihu.com/p/31082518)

## Փրկարարական Համակարգի Մանրամասները

Մուտքագրեք փրկարարական համակարգը 22 Այն բանից հետո, երբ նավահանգիստը հասանելի [tcping](https://github.com/paradiseduo/tcping) , դուք չեք կարողանա մուտք գործել սահմանված գաղտնաբառով 2-3 րոպե:

```
tcping ip 22 -c 9999
```

Նավահանգստի հասանելիությունը կարող է վերահսկվել:

### Ինչպես Մուտք Գործել Կոշտ Սկավառակ & Փրկարարական Ծառայություն Մտնելուց Հետո

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Օրինակ, կարող եք դիտել նկարը

```
snapper -c etc list
```

Դիտեք նկարի ֆայլի փոփոխությունները

```
snapper -c etc status 5..6
```

Դիտեք հատուկ տարբերվող տեքստը

```
snapper -c etc diff 3..5
```

Վերականգնել պատկերը. Օրինակ՝ հետարկել բոլոր փոփոխությունները 5-րդ նկարից հետո

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Միացնել Գրանցամատյանի Կայունությունը

Փրկարարական համակարգում գրանցամատյանների կայունության ակտիվացումը պահանջում է, որ դուք ձեռքով ստեղծեք համապատասխան գրացուցակը և կատարեք որոշ լրացուցիչ քայլեր, քանի որ հնարավոր է, որ դուք չունեք գործող systemd ծառայություն: Ահա քայլեր, որոնք կարող եք փորձել.

Նախ chroot, ապա

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Վերագործարկեք ձեր Ubuntu համակարգում. վերը նշված փոփոխությունները կատարելուց հետո դուք պետք է վերաբեռնեք ձեր մեքենան սովորական ռեժիմով և ձեր Ubuntu համակարգում փրկարարական համակարգի փոխարեն:

Այսպիսով, systemd-journald-ը կսկսի նոր կոնֆիգուրացիայից և կսկսի կոշտ սկավառակի վրա համառորեն պահել տեղեկամատյանները:

Հիշեք, որ վերը նշված գործողությունները կատարվում են փրկարարական համակարգի վրա, ուստի այդ փոփոխություններն ուժի մեջ չեն մտնի մինչև համակարգը չվերագործարկվի և այլևս չհայտնվի փրկարարական ռեժիմում:

Սա նշանակում է, որ դուք անմիջապես չեք տեսնի գոյություն ունեցող տեղեկամատյանների որևէ փոփոխություն, որը դառնում է մշտական, բայց դրանից հետո ստեղծված բոլոր տեղեկամատյանները կպահպանվեն:

Այս կերպ Դուք կարող եք դիտել ssh ծառայության տեղեկամատյանները

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Ծանոթագրություններ Բարդ ԵՒ Բարդ Հիվանդությունների Մասին

### Լուծեք Տարօրինակ Խնդիրը. Սերվերը Չի Կարող ssh

Նախկինում ես օգտագործել էի github գործողությունը՝ ավտոմատ հավաքման և տեղակայման սցենար պատրաստելու համար, բայց պարզեցի, որ տեղակայման ծառայությունը չի կարող լինել ssh:

Փրկարարական համակարգ մտնելը և տեղեկամատյանները դիտելը թույլտվության խնդիր է.

`Missing privilege separation directory: /run/sshd`

Դուք կարող եք ստեղծել այս գրացուցակը հետևյալ հրամանով

```
chown root:root /
systemd-tmpfiles --create
```

Կրկնվող հետազոտություններից հետո, քանի github action tar `uid` `1001` է

Ապափաթեթավորումը պետք է լինի tar եթե չկա պարամետր `--no-same-owner` , ապա այն կհանվի `uid` ի `1001` օգտագործողի համար:

Արդյունքում, rsync-ը կփոխի արմատային գրացուցակի օգտատերերի խումբը, այնուհետև վերագործարկումից հետո դուք չեք կարողանա մուտք գործել ssh:

Փոփոխեք ապակոմպրեսիոն սցենարը հետևյալ կերպ.

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
