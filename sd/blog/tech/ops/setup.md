## پاسورڊ مفت لاگ ان کي ترتيب ڏيو

ڪجھ منٽن کان پوء، سرورز سڀني کي چالو ڪيو ويو.

پهرين شيء کي ايڊٽ ڪرڻ IP مقامي مشين جي `~/.ssh/config` کي عرف سيٽ ڪرڻ لاءِ، ۽ پوءِ استعمال ڪريو `ssh-copy-id root@c2` کي ترتيب ڏيڻ لاءِ پاسورڊ فري لاگ ان.

![](https://i-01.eu.org/1718287198.avif)

توھان ھن [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) ترميم ڪري سگھوٿا شروعاتي ترتيب کي تيزيءَ سان مڪمل ڪرڻ لاءِ.

پاسورڊ کان سواءِ ٺاھ جوڙ مڪمل ٿيڻ کان پوءِ، توھان ان [pdsh](https://github.com/chaos/pdsh) استعمال ڪري سگھو ٿا ان کي بيچ ۾ هلائي سگھجي ٿو، جيڪو آسان آھي.

پهريون دستي طور تي ipv6 ۽ ٽيسٽ کي فعال ڪرڻ آهي

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

پوءِ سسٽم کي اپ گريڊ ڪريو ubuntu 24 تي هڪ ڪلڪ سان ۽ انسٽال ڪريو عام طور تي استعمال ٿيندڙ سافٽ ويئر.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN = 1 جو مطلب آھي ٻولي کي چيني ڏانھن ۽ ٽائيم زون کي بيجنگ ڏانھن.

ڊفالٽ طور تي نصب ڪيل سافٽ ويئر منهنجي ذاتي استعمال جي عادتن مطابق ترتيب ڏنل آهي، جهڙوڪ [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) , وغيره. جيڪڏهن ضروري هجي ته، توهان [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) گودام کي نقل ڪري سگهو ٿا ۽ ضرورت مطابق ان کي تبديل ڪري سگهو ٿا.

اهو قدم هڪ ڊگهو وقت وٺندو آهي، اڌ ڪلاڪ جي انتظار کان پوء، عام طور تي استعمال ٿيل سافٽ ويئر جي اپڊيٽ ۽ انسٽال ڪرڻ ڪامياب ٿي ويندي. توھان ھيٺ ڏنل حڪم سان ھن جي تصديق ڪري سگھو ٿا:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh تيز ڪري ٿو ssh

يورپ کان چين تائين نيٽ ورڪ کي استعمال ڪرڻ جي سفارش ڪئي وئي آهي `ssh` [mosh](//mosh.org)

مقامي ڊاريڪٽري `~/.bin` ۾ `c0` اسڪرپٽ ٺاھيو (بنيادي طور تي ماحوليات جي متغير `PATH` ۾ شامل ڪيو ويو) ھيٺ ڏنل آھي:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

ٺاھڻ کان پوء، قابل عمل اجازت ڏيو ۽ نرم لنڪ `c1` ۽ `c2` .

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

مستقبل ۾، توھان سڌو سنئون داخل ڪري سگھو ٿا `c0` ۽ `c1` استعمال ڪرڻ لاءِ `mosh` تيز رفتار سڌو ڪنيڪشن ۽ داخل ڪريو [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## ورهاڱي جي فارميٽ کي ext4 کان btrfs ۾ تبديل ڪريو

آسان بيڪ اپ لاءِ ڊسڪ اسنيپ شاٽ وٺي سگھي [btrfs](https://wikipedia.org/wiki/Btrfs) .

پهرين بچاء واري نظام ۾ داخل ڪريو

![](https://i-01.eu.org/1718300448.avif)

ڊيبين ریسکيو چونڊيو

![](https://i-01.eu.org/1718300753.avif)

سرور کي ٻيهر شروع ڪرڻ لاء انتظار ڪريو

ssh فنگر پرنٽ تبديليون ٿينديون، ان کي صاف ڪرڻ لاء `ssh-keygen -R ip` استعمال ڪريو

ورهاڱي کي تبديل ڪرڻ لاءِ هيٺ ڏنل اسڪرپٽ هلايو

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

رسم الخط خودڪار طريقي سان سسٽم کي ٻيهر شروع ڪندو جڏهن تبديلي ڪامياب ٿي ويندي آهي.

### ڏسو btrfs ڪمپريشن تناسب

اهو خودڪار طور تي ڊفالٽ طور ڪمپريشن جي تناسب کي جانچڻ جو حڪم هن ريت آهي:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### دستي کمپريشن

جيڪڏهن توهان هڪ تڪميل پسند آهيو، توهان پڻ مڪمل ڊسڪ ڪمپريشن دستي طور تي ڪم ڪري سگهو ٿا حڪم هن ريت آهي:

```bash
btrfs filesystem defragment -r -czstd /
```

جيئن هيٺ ڏنل شڪل ۾ ڏيکاريل آهي، توهان ڏسي سگهو ٿا ته نئين سسٽم ۾ دستي ڪمپريشن کان پوء، تقريبا 100 ايم بي وڌيڪ دٻايو ويو.

### سنيپر سان گڏ هڪ Btrfs سنيپ شاٽ بيڪ اپ ٺاهيو

Btrfs بيڪ اپ لاءِ بهترين پارٽنر آهي [snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html)

شروعاتي اسڪرپٽ `~/ubuntu` ڊاريڪٽري ٺاهي ويندي، صرف ھيٺ ڏنل حڪم هلائي.

```
~/ubuntu/snapper/init.sh
```

توهان کي اسڪرپٽ مواد ۾ ڏسو [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

ذيلي حجم ڏسڻ لاءِ `snapper list-configs` استعمال ڪريو جن لاءِ سنيپ شاٽ سيٽ ڪيا ويا آھن.

![](https://i-01.eu.org/1718302689.avif)

استعمال ڪريو `snapper -c mnt list` ھڪڙي ذيلي حجم جي سنيپ شاٽ ڏسڻ لاء

`/etc/snapper/configs/mnt` هن ذيلي حجم جي مخصوص بيڪ اپ سيٽنگون ڏسي سگھن ٿا

## بچاء واري نظام جي تفصيل

داخل ڪريو بچاءُ واري 22 کان پوءِ پورٽ [tcping](https://github.com/paradiseduo/tcping) پهچ، توهان ان تائين رسائي نٿا ڪري سگهو 2-3 منٽ لاءِ انتظار ڪريو ۽ اهو ٺيڪ ٿي ويندو.

```
tcping ip 22 -c 9999
```

پورٽ جي رسائي جي نگراني ڪري سگهجي ٿو.

### ريسٽور سنيپ شاٽ داخل ڪرڻ کان پوءِ هارڊ ڊرائيو تائين ڪيئن رسائي حاصل ڪجي &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

توھان سنيپ شاٽ ڏسي سگھو ٿا، مثال طور

```
snapper -c etc list
```

سنيپ شاٽ فائل ۾ تبديليون ڏسو

```
snapper -c etc status 5..6
```

ڏسو مخصوص فرق متن

```
snapper -c etc diff 3..5
```

سنيپ شاٽ بحال ڪريو: مثال طور، سنيپ شاٽ 5 کان پوءِ سڀ تبديليون واپس ڪريو

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### لاگ تسلسل کي فعال ڪريو

بچاءُ واري نظام جي تحت لاگ تسلسل کي فعال ڪرڻ لاءِ توهان کي دستي طور تي لاڳاپيل ڊاريڪٽري ٺاهڻ ۽ ڪجهه اضافي قدمن کي انجام ڏيڻ جي ضرورت آهي، ڇاڪاڻ ته توهان وٽ شايد هلندڙ سسٽم ڊي سروس نه هجي. هتي اهي قدم آهن جيڪي توهان ڪوشش ڪري سگهو ٿا:

پهرين ڪروٽ، پوءِ

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

توهان جي Ubuntu سسٽم ۾ ريبوٽ ڪريو: مٿيون تبديليون ڪرڻ کان پوء، توهان کي پنهنجي مشين کي عام طور تي ريبوٽ ڪرڻ جي ضرورت آهي ۽ بچاء واري نظام جي بدران توهان جي Ubuntu سسٽم ۾.

اهڙيءَ طرح، systemd-journald نئين ترتيب سان شروع ڪندو ۽ هارڊ ڊسڪ تي لڳاتار لاگن کي محفوظ ڪرڻ شروع ڪندو.

ذهن ۾ رکو ته مٿين عملن کي بچاء واري نظام تي ڪيو ويو آهي، تنهنڪري اهي تبديليون اثر انداز نه ٿينديون جيستائين سسٽم ريبوٽ نه ڪيو وڃي ۽ هاڻي بچاء واري موڊ ۾ نه آهي.

ان جو مطلب اهو آهي ته توهان فوري طور تي موجود لاگز ۾ ڪا به تبديلي نه ڏسندا جيڪي مستقل ٿي ويندا آهن، پر ان کان پوءِ ٺاهيل سڀئي لاگز برقرار رهندا.

هن طريقي سان توهان ssh سروس جا لاگ ڏسي سگهو ٿا

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## ڏکيو ۽ پيچيده بيمارين تي نوٽس

### عجيب مسئلو حل ڪريو: سرور ssh نٿو ڪري سگھي

مون استعمال ڪيو github عمل هڪ خودڪار تاليف ۽ ترتيب ڏيڻ واري اسڪرپٽ ٺاهڻ لاءِ اڳي، پر ڏٺم ته ڊيپلائيشن سروس ssh نه ٿي سگهي.

بچاء واري نظام ۾ داخل ٿيڻ ۽ لاگز کي ڏسڻ جي اجازت جو مسئلو آهي:

`Missing privilege separation directory: /run/sshd`

توھان ھي ڊاريڪٽري ٺاھي سگھوٿا ھيٺ ڏنل حڪم سان

```
chown root:root /
systemd-tmpfiles --create
```

بار بار تحقيق ڪرڻ کان پوء، اهو آهي ڇو github action صارف `uid` جو tar `1001` آهي

Unpacking هجڻ ضروري آهي جيڪڏهن ڪو به پيراميٽر `--no-same-owner` نه آهي، اهو `1001` استعمال ڪندڙن لاءِ `uid` تي ڊمپپريشن ڪيو ويندو tar

نتيجي طور، rsync روٽ ڊاريڪٽري جي استعمال ڪندڙ گروپ کي تبديل ڪندو، ۽ پوء توهان ٻيهر شروع ڪرڻ کان پوء ssh ۾ لاگ ان ٿيڻ جي قابل نه هوندا.

ھيٺ ڏنل ڊمپپريشن اسڪرپٽ کي تبديل ڪريو:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
