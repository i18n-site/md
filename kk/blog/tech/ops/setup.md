## Құпия Сөзсіз Кіруді Конфигурациялаңыз

Бірнеше минуттан кейін барлық серверлер іске қосылды.

Бірінші нәрсе - жергілікті құрылғының `~/.ssh/config` үшін бүркеншік атын орнату IP содан кейін құпия сөзсіз кіруді конфигурациялау үшін `ssh-copy-id root@c2` пайдаланыңыз.

<img alt="" src="https://i-01.eu.org/1718287198.avif">

Баптандыру конфигурациясын тезірек аяқтау үшін осы [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) өзгертуге болады.

Құпия сөзсіз конфигурациялау аяқталғаннан кейін оны пайдалануға болады, бұл ыңғайлы [pdsh](https://github.com/chaos/pdsh)

Біріншісі - ipv6-ны қолмен қосу және тексеру

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

<img alt="" src="https://i-01.eu.org/1718287969.avif">

Содан кейін бір рет басу арқылы жүйені ubuntu 24 нұсқасына жаңартыңыз және жиі қолданылатын бағдарламалық құралды орнатыңыз.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 тілді қытай тіліне және уақыт белдеуін Бейжіңге ауыстыруды білдіреді.

Әдепкі бойынша орнатылған бағдарламалық құрал , [direnv](https://direnv.net/) және т.б. сияқты жеке пайдалану әдеттеріме сәйкес конфигурацияланған [mise](https://github.com/jdx/mise) Қажет болса, [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) қоймасын көшіріп, қажетінше өзгертуге болады.

Бұл қадам көп уақытты алады, жарты сағат күткеннен кейін жиі қолданылатын бағдарламалық құралды жаңарту және орнату сәтті болады. Оны келесі пәрмен арқылы растауға болады:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh ssh Жылдамдатады

[mosh](//mosh.org) `ssh` дейінгі желі тұрақты емес.

Жергілікті `~/.bin` (негізінен `PATH` орта айнымалысына қосылады) келесідей `c0` сценарийін жасаңыз:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

Жасалғаннан кейін орындалатын рұқсаттарды және жұмсақ сілтемені беріңіз `c1` `c2` .

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

Болашақта тікелей `c0` , `c1` енгізе аласыз және сіз `mosh` жоғары жылдамдықты тікелей қосылымды пайдалана аласыз және енгізе аласыз [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## Құтқару Жүйесінің Мәліметтері

Құтқару жүйесіне кіріңіз 22 Портқа [tcping](https://github.com/paradiseduo/tcping) қолжетімді болғаннан кейін, сіз оған орнатылған құпия сөзбен 2-3 минут күте алмайсыз.

```
tcping ip 22 -c 9999
```

Порттың қолжетімділігін бақылауға болады.

### & Суретін Енгізгеннен Кейін Қатты Дискіге Қалай Қол Жеткізуге Болады

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Мысалы, суретті көруге болады

```
snapper -c etc list
```

Сурет файлының өзгерістерін қараңыз

```
snapper -c etc status 5..6
```

Арнайы айырмашылық мәтінін қараңыз

```
snapper -c etc diff 3..5
```

Суретті қалпына келтіру: Мысалы, сурет 5-тен кейінгі барлық өзгерістерді болдырмау

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Журнал Тұрақтылығын Қосыңыз

Құтқару жүйесінде журнал тұрақтылығын қосу сізге сәйкес каталогты қолмен жасауды және кейбір қосымша қадамдарды орындауды талап етеді, себебі сізде іске қосылған systemd қызметі болмауы мүмкін. Мына қадамдарды орындауға болады:

Алдымен chroot, содан кейін

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Ubuntu жүйесінде қайта жүктеңіз: Жоғарыда көрсетілген өзгертулерді жасағаннан кейін, құрылғыңызды қалыпты жағдайда және құтқару жүйесінің орнына Ubuntu жүйесінде қайта жүктеуіңіз керек.

Осылайша, systemd-journald жаңа конфигурациядан басталады және журналдарды қатты дискіде тұрақты түрде сақтай бастайды.

Жоғарыда көрсетілген әрекеттер құтқару жүйесінде орындалатынын есте сақтаңыз, сондықтан бұл өзгерістер жүйе қайта жүктелгеннен кейін ғана күшіне енеді және құтқару режимінде болмайды.

Бұл бұрыннан бар журналдардағы өзгерістердің тұрақты болатынын бірден көрмейсіз, бірақ одан кейін жасалған барлық журналдар сақталады дегенді білдіреді.

Осылайша сіз ssh қызметінің журналдарын көре аласыз

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Күрделі Және Күрделі Аурулар Туралы Ескертулер

### Біртүрлі Мәселені Шешіңіз: Сервер ssh Жасай Алмайды

Мен бұрын автоматты жинақтау және орналастыру сценарийін жасау үшін github әрекетін қолдандым, бірақ орналастыру қызметі ssh бола алмайтынын анықтадым.

Құтқару жүйесіне кіру және журналдарды қарау рұқсат мәселесі болып табылады:

`Missing privilege separation directory: /run/sshd`

Бұл каталогты келесі пәрмен арқылы жасауға болады

```
chown root:root /
systemd-tmpfiles --create
```

Қайталанатын зерттеулерден кейін, бұл github action tar `1001` салған пайдаланушы `uid`

tar Орамнан шығару параметрлерсіз орындалуы керек `--no-same-owner` ол `1001` үшін `uid` пайдаланушыға шығарылады

Нәтижесінде rsync түбірлік каталогтың пайдаланушылар тобын өзгертеді, содан кейін қайта іске қосқаннан кейін ssh жүйесіне кіре алмайсыз.

Декомпрессия сценарийін келесідей өзгертіңіз:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
