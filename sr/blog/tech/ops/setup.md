## Конфигуришите Пријављивање Без Лозинке

Након неколико минута, сви сервери су били активирани.

Прва ствар IP да уредите `~/.ssh/config` локалне машине да поставите псеудоним, а затим користите `ssh-copy-id root@c2` да конфигуришете пријаву без лозинке.

![](https://i-01.eu.org/1718287198.avif)

Можете да измените ову [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) да бисте брже завршили конфигурацију иницијализације.

Након што је конфигурација без лозинке завршена, можете је користити у групама, што је згодно [pdsh](https://github.com/chaos/pdsh)

Први је да ручно омогућите ипв6 и тестирате

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

Затим надоградите систем на убунту 24 једним кликом и инсталирајте софтвер који се обично користи.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

ЦН=1 значи пребацивање језика на кинески и временске зоне на Пекинг.

Софтвер који је подразумевано инсталиран је конфигурисан у складу са мојим личним навикама коришћења, као што су [mise](https://github.com/jdx/mise) [direnv](https://direnv.net/) , итд. Ако је потребно, можете копирати [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) варехоусе и изменити га по потреби!

Овај корак траје дуго Након чекања од пола сата, надоградња и инсталација често коришћеног софтвера ће бити успешна. Ово можете потврдити следећом командом:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

![](https://i-01.eu.org/1718290346.avif)

## Мосх Убрзава Ссх

Мрежа од Европе до Кине није стабилна. Препоручује [mosh](//mosh.org) употреба `ssh` .

Креирајте `c0` скрипте у локалном директоријуму `~/.bin` (углавном додате променљивој окружења `PATH` ) на следећи начин:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

Након креирања, доделите извршне дозволе и софт-линк `c1` и `c2` .

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

У будућности можете директно да унесете `c0` и `c1` да бисте користили `mosh` директне везе велике брзине и унесите [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## Конвертујте Формат Партиције Из Ект4 У Бтрфс

Може [btrfs](https://wikipedia.org/wiki/Btrfs) прави снимке диска ради лакшег прављења резервних копија.

Прво уђите у систем спасавања

![](https://i-01.eu.org/1718300448.avif)

Изаберите Дебиан Ресцуе

![](https://i-01.eu.org/1718300753.avif)

Сачекајте да се сервер поново покрене

ssh промена отиска прста, користите `ssh-keygen -R ip` да га очистите

Затим покрените следећу скрипту да бисте конвертовали партицију

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

Скрипта ће аутоматски поново покренути систем након што конверзија буде успешна.

### Погледајте Компресију Бтрфс

Подразумевано ће се аутоматски компримовати Команда за проверу компресије је следећа:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### Ручна Компресија

Ако сте перфекциониста, можете и ручно да извршите комплетну компресију диска: Команда је следећа.

```bash
btrfs filesystem defragment -r -czstd /
```

Као што је приказано на слици испод, можете видети да је након ручне компресије у новом систему компримовано још скоро 100МБ.

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### Направите Резервну Копију Бтрфс Снимака Помоћу Снаппер-А

```

```

Видимо се у садржају сценарија [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

## Детаљи О Систему Спасавања

Уђите у систем за 22 Након што је порт [tcping](https://github.com/paradiseduo/tcping) , нећете моћи да му приступите са постављеном лозинком 2-3 минута.

```
tcping ip 22 -c 9999
```

Приступачност порта се може пратити.

### Како Приступити Чврстом Диску Након Уласка У Ресторе Снапсхот &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Можете погледати снимак, на пример

```
snapper -c etc list
```

Прегледајте измене датотеке снимка

```
snapper -c etc status 5..6
```

Погледајте одређени текст разлике

```
snapper -c etc diff 3..5
```

Врати снимак: На пример, поништите све промене након снимка 5

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Омогућите Постојаност Дневника

Омогућавање постојаности евиденције у систему за спасавање захтева да ручно креирате одговарајући директоријум и извршите неке додатне кораке, јер можда немате покренут системд сервис. Ево корака које можете испробати:

Прво цхроот, онда

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Поново покрените свој Убунту систем: Након што извршите горенаведене измене, потребно је да поново покренете машину нормално и у свој Убунту систем уместо у систем за спасавање.

На овај начин, системд-јоурналд ће почети са новом конфигурацијом и почети да стално складишти евиденције на чврстом диску.

Имајте на уму да се горе наведене операције изводе на систему за спасавање, тако да те промене неће ступити на снагу све док се систем поново не покрене и више не буде у режиму спасавања.

То значи да нећете одмах видети никакве промене у постојећим евиденцијама које постану трајне, али ће сви дневники генерисани након тога бити постојани.

На овај начин можете прегледати евиденцију ссх услуге

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Напомене О Тешким И Компликованим Болестима

### Решите Чудан Проблем: Сервер Не Може Ссх

Раније сам користио гитхуб акцију да направим аутоматску компилацију и скрипту за примену, али сам открио да услуга постављања не може бити ссх.

Улазак у систем спасавања и гледање дневника је проблем са дозволом:

`Missing privilege separation directory: /run/sshd`

Можете креирати овај директоријум помоћу следеће команде

```
chown root:root /
systemd-tmpfiles --create
```

Након поновљених истраживања, то је зато github action је корисник `uid` tar `1001`

Распакивање мора бити ако не постоји параметар `--no-same-owner` , биће декомпримован на `uid` за `1001` корисника tar

Као резултат тога, рсинц ће променити корисничку групу основног директоријума и тада нећете моћи да се пријавите на ссх након поновног покретања.

Измените скрипту за декомпресију на следећи начин:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
