## पासवर्ड-रहित लगइन कन्फिगर गर्नुहोस्

केहि मिनेट पछि, सर्भरहरू सबै सक्रिय थिए।

पहिलो कुरा भनेको स्थानीय मेसिनको लागि एक उपनाम सेट गर्नुहोस् `~/.ssh/config` र त्यसपछि पासवर्ड-रहित लगइन कन्फिगर गर्न `ssh-copy-id root@c2` प्रयोग गर्नुहोस् IP

<img alt="" src="https://i-01.eu.org/1718287198.avif">

प्रारम्भिक कन्फिगरेसन छिटो पूरा गर्न तपाईँले यो [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) परिमार्जन गर्न सक्नुहुन्छ।

पासवर्ड-रहित कन्फिगरेसन पूरा भएपछि, तपाईंले यसलाई ब्याचहरूमा सञ्चालन गर्न सक्नुहुन्छ, जुन सुविधाजनक छ [pdsh](https://github.com/chaos/pdsh)

पहिलो म्यानुअल रूपमा ipv6 र परीक्षण सक्षम गर्न हो

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

<img alt="" src="https://i-01.eu.org/1718287969.avif">

त्यसपछि प्रणालीलाई एक क्लिकमा ubuntu 24 मा अपग्रेड गर्नुहोस् र सामान्यतया प्रयोग हुने सफ्टवेयर स्थापना गर्नुहोस्।

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 भनेको भाषालाई चिनियाँ र समय क्षेत्र बेइजिङमा बदल्नु हो।

पूर्वनिर्धारित रूपमा स्थापित सफ्टवेयर मेरो व्यक्तिगत प्रयोग बानीहरू अनुसार कन्फिगर गरिएको छ, जस्तै [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) , आदि। आवश्यक भएमा, तपाईंले [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) वेयरहाउसलाई प्रतिलिपि गर्न सक्नुहुन्छ र आवश्यकता अनुसार परिमार्जन गर्न सक्नुहुन्छ।

यो चरणले लामो समय लिन्छ, आधा घण्टाको प्रतीक्षा पछि, सामान्यतया प्रयोग हुने सफ्टवेयरको स्तरवृद्धि र स्थापना सफल हुनेछ। तपाइँ यसलाई निम्न आदेश संग पुष्टि गर्न सक्नुहुन्छ:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh ssh को गति बढाउँछ

युरोप देखि चीन को नेटवर्क यो गति गर्न को `ssh` सिफारिश गरिएको छ [mosh](//mosh.org)

स्थानीय मेसिनमा `c0` स्क्रिप्ट सिर्जना `~/.bin` (मुख्य रूपमा वातावरण चर `PATH` मा थपियो) निम्नानुसार:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

सिर्जना पछि, कार्यान्वयन योग्य अनुमतिहरू र `c2` -लिङ्क दिनुहोस् `c1`

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

भविष्यमा, तपाइँ सिधै प्रविष्ट गर्न सक्नुहुन्छ `c0` `c1` र तपाइँ `mosh` जडान गर्न र प्रविष्ट गर्न सक्नुहुन्छ [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## विभाजन ढाँचालाई ext4 बाट btrfs मा रूपान्तरण गर्नुहोस्

सजिलो ब्याकअपको लागि डिस्क स्न्यापसट लिन सक्छ [btrfs](https://wikipedia.org/wiki/Btrfs)

पहिले उद्धार प्रणाली प्रविष्ट गर्नुहोस्

<img alt="" src="https://i-01.eu.org/1718300448.avif">

डेबियन उद्धार छान्नुहोस्

<img alt="" src="https://i-01.eu.org/1718300753.avif">

सर्भर पुन: सुरु हुनको लागि पर्खनुहोस्

फिंगरप्रिन्ट परिवर्तन हुनेछ ssh यसलाई सफा गर्न `ssh-keygen -R ip`

त्यसपछि विभाजन रूपान्तरण गर्न निम्न स्क्रिप्ट चलाउनुहोस्

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

रूपान्तरण सफल भएपछि स्क्रिप्ट स्वचालित रूपमा प्रणाली पुन: सुरु हुनेछ।

### btrfs कम्प्रेसन अनुपात हेर्नुहोस्

यो स्वतः पूर्वनिर्धारित रूपमा कम्प्रेस हुनेछ कम्प्रेसन अनुपात जाँच गर्न आदेश निम्नानुसार छ:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### म्यानुअल कम्प्रेसन

यदि तपाइँ पूर्णतावादी हुनुहुन्छ भने, तपाइँ म्यानुअल रूपमा पूर्ण डिस्क कम्प्रेसन गर्न सक्नुहुन्छ आदेश निम्नानुसार छ

```bash
btrfs filesystem defragment -r -czstd /
```

तलको चित्रमा देखाइए अनुसार, तपाईले देख्न सक्नुहुन्छ कि नयाँ प्रणालीमा म्यानुअल कम्प्रेसन पछि, लगभग 100MB थप कम्प्रेस गरिएको थियो।

### स्न्यापरसँग Btrfs स्न्यापसट ब्याकअप सिर्जना गर्नुहोस्

Btrfs ब्याकअपको लागि उत्तम साझेदार हो [snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html)

प्रारम्भिक स्क्रिप्टले `~/ubuntu` सिर्जना गर्नेछ, केवल निम्न आदेश चलाउनुहोस्:

```
~/ubuntu/snapper/init.sh
```

स्क्रिप्ट सामग्रीमा भेटौंला [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

उपभोल्युमहरू हेर्न प्रयोग `snapper list-configs` जसको लागि स्न्यापसटहरू सेट गरिएका छन्।

`/etc/snapper/configs/mnt` यो सबभोल्युमको विशिष्ट ब्याकअप सेटिङहरू हेर्न सक्नुहुन्छ

## उद्धार प्रणाली विवरण

उद्धार प्रणाली प्रविष्ट गर्नुहोस् 22 पोर्ट [tcping](https://github.com/paradiseduo/tcping) भएपछि, तपाईंले 2-3 मिनेटको लागि यसलाई पहुँच गर्न सक्नुहुन्न र यो ठीक हुनेछ।

```
tcping ip 22 -c 9999
```

पोर्ट पहुँच अनुगमन गर्न सकिन्छ।

### रिस्टोर स्न्यापसटमा प्रवेश & हार्ड ड्राइभमा कसरी पहुँच गर्ने

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

तपाईं स्न्यापसट हेर्न सक्नुहुन्छ, उदाहरणका लागि

```
snapper -c etc list
```

स्न्यापसट फाइल परिवर्तनहरू हेर्नुहोस्

```
snapper -c etc status 5..6
```

विशिष्ट भिन्न पाठ हेर्नुहोस्

```
snapper -c etc diff 3..5
```

स्न्यापसट पुनर्स्थापित गर्नुहोस्: उदाहरणका लागि, स्न्यापसट 5 पछि सबै परिवर्तनहरू पूर्ववत गर्नुहोस्

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### लग दृढता सक्षम गर्नुहोस्

एक उद्धार प्रणाली अन्तर्गत लग दृढता सक्षम गर्नाले तपाईंलाई म्यानुअल रूपमा सम्बन्धित डाइरेक्टरी सिर्जना गर्न र केही थप चरणहरू गर्न आवश्यक छ, किनभने तपाईंसँग चलिरहेको systemd सेवा नहुन सक्छ। यहाँ तपाईंले प्रयास गर्न सक्ने चरणहरू छन्:

पहिले क्रोट, त्यसपछि

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

तपाईंको Ubuntu प्रणालीमा रिबुट गर्नुहोस्: माथिका परिवर्तनहरू गरेपछि, तपाईंले आफ्नो मेसिनलाई सामान्य रूपमा र उद्धार प्रणालीको सट्टा तपाईंको Ubuntu प्रणालीमा रिबुट गर्न आवश्यक छ।

यस तरिकाले, systemd-journald नयाँ कन्फिगरेसनसँग सुरु हुनेछ र हार्ड डिस्कमा लगहरूलाई निरन्तर रूपमा भण्डारण गर्न थाल्नेछ।

ध्यान राख्नुहोस् कि माथिका अपरेसनहरू उद्धार प्रणालीमा गरिन्छन्, त्यसैले ती परिवर्तनहरू प्रणाली रिबुट भएपछि मात्र प्रभावकारी हुनेछन् र अब उद्धार मोडमा छैनन्।

यसको मतलब तपाईले अवस्थित लगहरूमा कुनै पनि परिवर्तनहरू स्थिर भएको देख्नुहुने छैन, तर यस पछि उत्पन्न भएका सबै लगहरू निरन्तर रहनेछन्।

यसरी तपाईले ssh सेवाको लगहरू हेर्न सक्नुहुन्छ

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## कठिन र जटिल रोगहरूमा टिप्पणीहरू

### अनौठो समस्या समाधान गर्नुहोस्: सर्भरले ssh गर्न सक्दैन

मैले पहिले स्वचालित संकलन र डिप्लोइमेन्ट स्क्रिप्ट बनाउन github कार्य प्रयोग गरें, तर फेला पारे कि डिप्लोयमेन्ट सेवाले ssh सकेन।

उद्धार प्रणालीमा प्रवेश र लगहरू हेर्दै अनुमति मुद्दा हो:

`Missing privilege separation directory: /run/sshd`

तपाइँ निम्न आदेश संग यो डाइरेक्टरी सिर्जना गर्न सक्नुहुन्छ

```
chown root:root /
systemd-tmpfiles --create
```

बारम्बार अनुसन्धान पछि, `1001` tar अपलोड गर्ने प्रयोगकर्ता `uid` github action

अनप्याकिङ प्यारामिटरहरू बिना `--no-same-owner` tar `1001` प्रयोगकर्ताको लागि `uid` मा डिकम्प्रेस गरिनेछ

नतिजाको रूपमा, rsync ले रूट डाइरेक्टरीको प्रयोगकर्ता समूह परिवर्तन गर्नेछ, र त्यसपछि तपाइँ पुन: सुरु गरेपछि ssh मा लग इन गर्न सक्षम हुनुहुने छैन।

निम्नानुसार decompression लिपि परिमार्जन गर्नुहोस्:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
