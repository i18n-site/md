## पासवर्ड-मुक्त लॉगिन कॉन्फिगर करा

काही मिनिटांनंतर, सर्व सर्व्हर सक्रिय झाले.

पहिली गोष्ट म्हणजे स्थानिक मशीनचे `~/.ssh/config` साठी उपनाव सेट करा IP नंतर पासवर्ड-मुक्त लॉगिन कॉन्फिगर करण्यासाठी `ssh-copy-id root@c2` वापरा.

<img alt="" src="https://i-01.eu.org/1718287198.avif">

आरंभीकरण कॉन्फिगरेशन जलद पूर्ण करण्यासाठी तुम्ही या [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) बदल करू शकता.

पासवर्ड-मुक्त कॉन्फिगरेशन पूर्ण झाल्यानंतर, तुम्ही ते वापरू शकता [pdsh](https://github.com/chaos/pdsh) ते बॅचमध्ये ऑपरेट केले जाऊ शकते, जे सोयीस्कर आहे.

प्रथम ipv6 आणि चाचणी व्यक्तिचलितपणे सक्षम करणे आहे

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

<img alt="" src="https://i-01.eu.org/1718287969.avif">

नंतर एका क्लिकने सिस्टमला ubuntu 24 वर अपग्रेड करा आणि सामान्यतः वापरले जाणारे सॉफ्टवेअर इन्स्टॉल करा.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 म्हणजे भाषा चिनी आणि टाइम झोन बीजिंगमध्ये बदलणे.

डीफॉल्टनुसार स्थापित केलेले सॉफ्टवेअर माझ्या वैयक्तिक वापराच्या सवयींनुसार कॉन्फिगर केले आहे, जसे की [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) , इ. आवश्यक असल्यास, तुम्ही [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) गोदाम कॉपी करू शकता आणि आवश्यकतेनुसार त्यात बदल करू शकता.

या चरणात बराच वेळ लागतो अर्धा तास प्रतीक्षा केल्यानंतर, सामान्यतः वापरल्या जाणाऱ्या सॉफ्टवेअरचे अपग्रेड आणि इंस्टॉलेशन यशस्वी होईल. आपण खालील आदेशासह याची पुष्टी करू शकता:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh ssh वेग वाढवते

युरोप ते चीन पर्यंतचे नेटवर्क स्थिर नाही `ssh` वापरण्याची शिफारस केली जाते [mosh](//mosh.org)

खालीलप्रमाणे स्थानिक `~/.bin` (प्रामुख्याने पर्यावरण व्हेरिएबल `PATH` मध्ये जोडलेले) `c0` स्क्रिप्ट तयार करा:

```bash
#!/usr/bin/env बॅश

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

निर्मितीनंतर, एक्झिक्युटेबल परवानग्या आणि सॉफ्ट-लिंक द्या `c1` `c2` .

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

भविष्यात, आपण थेट प्रविष्ट करू शकता `c0` `c1` आणि आपण `mosh` उच्च-गती थेट कनेक्शन वापरू शकता आणि प्रविष्ट करू शकता [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## विभाजन स्वरूप ext4 वरून btrfs मध्ये रूपांतरित करा

सहज बॅकअपसाठी डिस्क स्नॅपशॉट घेऊ [btrfs](https://wikipedia.org/wiki/Btrfs) .

प्रथम बचाव प्रणाली प्रविष्ट करा

<img alt="" src="https://i-01.eu.org/1718300448.avif">

डेबियन बचाव निवडा

<img alt="" src="https://i-01.eu.org/1718300753.avif">

सर्व्हर रीस्टार्ट होण्याची प्रतीक्षा करा

फिंगरप्रिंट बदल होतील ssh ते स्वच्छ करण्यासाठी `ssh-keygen -R ip`

नंतर विभाजन रूपांतरित करण्यासाठी खालील स्क्रिप्ट चालवा

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

रूपांतरण यशस्वी झाल्यानंतर स्क्रिप्ट स्वयंचलितपणे सिस्टम रीस्टार्ट करेल.

### btrfs कॉम्प्रेशन रेशो पहा

हे डिफॉल्टनुसार आपोआप कॉम्प्रेस होईल.

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### मॅन्युअल कॉम्प्रेशन

जर तुम्ही परिपूर्णतावादी असाल, तर तुम्ही पूर्ण डिस्क कॉम्प्रेशन मॅन्युअली देखील करू शकता

```bash
btrfs filesystem defragment -r -czstd /
```

खालील आकृतीमध्ये दर्शविल्याप्रमाणे, तुम्ही पाहू शकता की नवीन प्रणालीमध्ये मॅन्युअल कॉम्प्रेशन केल्यानंतर, जवळजवळ 100MB अधिक संकुचित केले गेले.

### स्नॅपरसह Btrfs स्नॅपशॉट बॅकअप तयार करा

Btrfs बॅकअपसाठी सर्वोत्तम भागीदार [snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html) .

आरंभीकरण स्क्रिप्ट `~/ubuntu` तयार करेल, फक्त खालील आदेश चालवा:

```
~/ubuntu/snapper/init.sh
```

स्क्रिप्ट सामग्रीमध्ये भेटू [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

ज्यासाठी स्नॅपशॉट सेट केले आहेत ते सबव्हॉल्यूम पाहण्यासाठी `snapper list-configs` .

<img alt="" src="https://i-01.eu.org/1718302689.avif">

सबव्हॉल्यूमचा स्नॅपशॉट पाहण्यासाठी `snapper -c mnt list`

`/etc/snapper/configs/mnt` या सबव्हॉल्यूमची विशिष्ट बॅकअप सेटिंग्ज पाहू शकता

अधिक संदर्भ:

## बचाव प्रणाली तपशील

रेस्क्यू सिस्टम एंटर करा 22 पोर्ट ॲक्सेसेबल झाल्यानंतर, तुम्ही 2-3 मिनिटांसाठी थांबा [tcping](https://github.com/paradiseduo/tcping) ते ठीक होईल.

```
tcping ip 22 -c 9999
```

पोर्ट ऍक्सेसिबिलिटीचे परीक्षण केले जाऊ शकते.

### & रीस्टोर स्नॅपशॉटमध्ये प्रवेश केल्यानंतर हार्ड ड्राइव्हवर कसे प्रवेश करावे

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

तुम्ही स्नॅपशॉट पाहू शकता, उदाहरणार्थ

```
snapper -c etc list
```

स्नॅपशॉट फाइल बदल पहा

```
snapper -c etc status 5..6
```

विशिष्ट फरक मजकूर पहा

```
snapper -c etc diff 3..5
```

स्नॅपशॉट पुनर्संचयित करा: उदाहरणार्थ, स्नॅपशॉट 5 नंतर सर्व बदल पूर्ववत करा

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### लॉग चिकाटी सक्षम करा

रेस्क्यू सिस्टीम अंतर्गत लॉग पर्सिस्टन्स सक्षम करण्यासाठी तुम्हाला संबंधित डिरेक्ट्री स्वहस्ते तयार करावी लागेल आणि काही अतिरिक्त पायऱ्या कराव्या लागतील, कारण तुमच्याकडे कदाचित चालू असलेली systemd सेवा नसेल. तुम्ही प्रयत्न करू शकता अशा पायऱ्या येथे आहेत:

प्रथम chroot, नंतर

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

तुमच्या उबंटू सिस्टममध्ये रीबूट करा: वरील बदल केल्यानंतर, तुम्हाला तुमचे मशीन सामान्यपणे आणि रेस्क्यु सिस्टमऐवजी तुमच्या उबंटू सिस्टममध्ये रीबूट करावे लागेल.

अशा प्रकारे, systemd-journald नवीन कॉन्फिगरेशनसह सुरू होईल आणि हार्ड डिस्कवर सतत लॉग संग्रहित करणे सुरू करेल.

लक्षात ठेवा की वरील ऑपरेशन्स रेस्क्यू सिस्टमवर केली जातात, त्यामुळे ते बदल फक्त एकदाच प्रभावी होतील जेव्हा सिस्टम रीबूट होईल आणि यापुढे रेस्क्यू मोडमध्ये नसेल.

याचा अर्थ असा की तुम्हाला विद्यमान नोंदींमध्ये कोणतेही बदल ताबडतोब स्थिर होताना दिसणार नाहीत, परंतु यानंतर व्युत्पन्न केलेले सर्व लॉग कायम राहतील.

अशा प्रकारे तुम्ही ssh सेवेचे लॉग पाहू शकता

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## कठीण आणि गुंतागुंतीच्या आजारांवरील नोट्स

### विचित्र समस्या सोडवा: सर्व्हर ssh करू शकत नाही

मी आधी स्वयंचलित संकलन आणि उपयोजन स्क्रिप्ट करण्यासाठी गिथब क्रिया वापरली, परंतु उपयोजन सेवा ssh असू शकत नाही असे आढळले.

बचाव प्रणालीमध्ये प्रवेश करणे आणि लॉग पाहणे ही परवानगी समस्या आहे:

`Missing privilege separation directory: /run/sshd`

तुम्ही खालील कमांडद्वारे ही डिरेक्टरी तयार करू शकता

```
chown root:root /
systemd-tmpfiles --create
```

वारंवार तपासल्यानंतर, `1001` tar अपलोड केलेला वापरकर्ता `uid` github action

पॅरामीटर्सशिवाय अनपॅक करणे आवश्यक `--no-same-owner` tar `1001` वापरकर्त्यासाठी `uid` मध्ये विघटित केले जाईल

परिणामी, rsync रूट डिरेक्ट्रीचा वापरकर्ता गट बदलेल, आणि नंतर तुम्ही रीस्टार्ट केल्यानंतर ssh मध्ये लॉग इन करू शकणार नाही.

खालीलप्रमाणे डीकंप्रेशन स्क्रिप्ट सुधारित करा:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
