## サーバーレス全文検索ソリューションの概観

ドキュメントや個人のブログなど、中小規模の純粋に静的な Web サイトの場合、独自に構築した全文検索バックエンドを構築するのは重すぎるため、サービス不要の全文検索がより一般的な選択肢となります。

サーバーレス全文検索ソリューションは大きく二つのカテゴリに分類されます:

まず、[algolia.com](//algolia.com)などのサードパーティ検索サービスプロバイダーが、全文検索用のフロントエンドコンポーネントを提供しています。

このようなサービスは検索量に応じて料金が発生し、ウェブサイトの規制やコンプライアンスの問題により、中国本土のユーザーが利用できないことが多いです。

オフラインでの使用やイントラネットでの使用が不可能で、大きな制限があります。ここでは詳細に議論しません。

次に、純粋なフロントエンドの全文検索があります。

現在一般的な純粋なフロントエンド全文検索には、[lunrjs](//lunrjs.com)や[ElasticLunr.js](//github.com/weixsong/elasticlunr.js)(`lunrjs`をベースに二次開発されたもの)などがあります。

`lunrjs`のインデックス構築には二つの方法があり、それぞれに問題があります。

1. 事前に構築されたインデックスファイル

   インデックスにはすべてのドキュメントの単語が含まれているため、ファイルサイズが大きくなります。
   ドキュメントが追加や変更されるたびに、新しいインデックスファイルをロードする必要があります。
   ユーザーの待ち時間が長くなり、大量の帯域幅が消費されます。

2. ドキュメントをロードし、その場でインデックスを構築します

   インデックスの構築は計算負荷が高いため、アクセスごとにインデックスを再構築すると、明らかな遅延が発生し、ユーザーエクスペリエンスが低下します。

---

`lunrjs`以外にも、他の全文検索ソリューションもあります

[fusejs](//www.fusejs.io) は、文字列間の相似度を計算して検索を行います。

[TinySearch](//stackoverflow.com/questions/70984437/fuse-js-takes-10-seconds-with-semi-long-queries)は、ブルームフィルタを使用して検索しますが、接頭辞検索(例えば「goo」を入力して「good」や「google」を検索)には対応せず、自動補完のような効果も実現できません。

[TinySearch](//github.com/tinysearch/tinysearch)は、ブルームフィルターを用いて検索を行いますが、接頭辞検索(例えば、`goo`を入力し、`good`や`google`を検索)には対応せず、自動補完のような効果も得られません。

現行のソリューションにはそれぞれの欠点があり、`i18n.site`はその問題を解決するために、以下の特徴を持つ新たな純粋なフロントエンド全文検索ソリューションを自社開発しました

1. 多言語対応で、ファイルサイズも小さく、`gzip`圧縮後のサイズはわずか`6.9KB`です(比較のために、`lunrjs`のサイズは`25KB`です)
1. `indexedb`を基盤にした転置インデックスを構築し、メモリ消費が少なく、高速です
1. ドキュメントが追加/変更された場合、追加または変更されたドキュメントのみが再インデックスされるため、計算量が削減されます
1. オフラインでも使用可能です
1. オフライン利用可能

## 多言語対応の分詞

以下は分詞の`coffeescript`コードの例です。

![](//p.3ti.site/1727667759.avif)

以下は、分詞に用いる`coffeescript`のコードです

```coffee
SEG = new Intl.Segmenter 0, granularity: "word"

seg = (txt) =>
  r = []
  for {segment} from SEG.segment(txt)
    for i from segment.split('.')
      i = i.trim()
      if i and !'|`'.includes(i) and !/\p{P}/u.test(i)
        r.push i
  r

export default seg

export segqy = (q) =>
  seg q.toLocaleLowerCase()
```

その中で:

* `/\p{P}/` 、句読点に一致する正規表現です。 `! " # $ % & ' ( ) * + , - . / : ; < = > ? @ [ \ ] ^ _ ` { | } 〜。 `。</p><ul><li> `split('.')`は、 `Firefox`ブラウザの単語分割では`.`分割しないためです。</li>


## インデックスの構築

`word` : id - 単語

* `word` : id - ワード
* `docWord` : ドキュメントid - ワードidの配列
* `prefix` : プレフィックス - ワードidの配列
* `rindex` : ワードid - ドキュメントid : 行番号の配列
* `rindex` : ワードid - ドキュメントid : 行番号の配列

ドキュメントの`url`とバージョン番号`ver`の配列を渡し、`doc`テーブルにドキュメントが存在するかどうかを確認します。存在しない場合には、逆転置インデックスを作成し、渡されなかったドキュメントの逆転置インデックスを削除します。

この方法により、インクリメンタルなインデックス作成が可能となり、計算負荷を削減します。

フロントエンドのインタラクションでは、初回のロード時の遅延を避けるため、インデックスのロード進行状況バーを表示します。詳細は、「単一のprogress + 綺麗なCSSで実装したアニメーション付きの進行状況バー」[英文](//dev.to/i18n-site/a-single-progress-uses-pure-css-to-achieve-animation-effects-2oo) / [中国語](//juejin.cn/post/7413586285954154522)を参照してください。

### IndexedDBの高并发書き込み

このプロジェクトは、IndexedDBの非同期処理をカプセル化する[idb](//www.npmjs.com/package/idb)をベースに開発されています。

IndexedDBの読み取りと書き込みは非同期で、インデックス作成時には、ドキュメントが同時に読み込まれ、インデックスが作成されます。

競合書き込みによるデータの一部損失を避けるためには、以下の`coffeescript`のコードを参照し、読み取りと書き込みの間に`ing`キャッシュを追加して競合を防ぎます。

```coffee
pusher = =>
  ing = new Map()
  (table, id, val)=>
    id_set = ing.get(id)
    if id_set
      id_set.add val
      return

    id_set = new Set([val])
    ing.set id, id_set
    pre = await table.get(id)
    li = pre?.li or []

    loop
      to_add = [...id_set]
      li.push(...to_add)
      await table.put({id,li})
      for i from to_add
        id_set.delete i
      if not id_set.size
        ing.delete id
        break
    return

rindexPush = pusher()
prefixPush = pusher()
```

## プレフィックスリアルタイム検索

ユーザーが入力中に検索結果をリアルタイムで表示するため、例えば`wor`と入力すると、`words`や`work`など`wor`で始まる単語が表示されます。

![](//p.3ti.site/1727684944.avif)

検索エンジンは、分詞後の最後の単語を基に`prefix`テーブルを使用して、その単語で始まるすべての単語を検索し、順次検索します。

フロントエンドのインタラクションでは、ユーザー入力の頻度を減らし、計算負荷を軽減するために、デバウンス機能`debounce`を使用します(以下のように実装されています)。

```js
export default (wait, func) => {
  var timeout;
  return function(...args) {
    clearTimeout(timeout);
    timeout = setTimeout(func.bind(this, ...args), wait);
  };
}
```

## 精度と完全性

検索では、まずユーザーが入力したキーワードを分詞します。

分詞後のワードが`N`個あると仮定すると、結果はまずすべてのキーワードを含む結果から始まり、次に`N-1`、`N-2`、…、`1`個のキーワードを含む結果が順次表示されます。

最初に表示される検索結果によってクエリの正確さが保証され、その後読み込まれる結果 ([さらに読み込む] ボタンをクリック) によって再現率が保証されます。

![](//p.3ti.site/1727684564.avif)

## オフライン利用可能

インデックステーブルには元のテキストは保存されず、単語のみが保存されるため、ストレージの容量が削減されます。

同時に、`service worker`がすべての記事をキャッシュするため、ユーザーが検索を実行すると、検索を含むウェブサイト全体がオフラインでも利用可能になります。

さらに、`service worker`がすべての記事をキャッシュするため、ユーザーが一度検索を行った後は、検索を含むウェブサイト全体がオフラインでも利用可能になります。

## まとめ

純粋なフロントエンドで実装された逆転置全文検索は、サーバー無しで実行可能であり、ドキュメントや個人のブログなどの中小規模のウェブサイトに非常に適しています。

`i18n.site`がオープンソースで自社開発した純粋なフロントエンド検索は、ファイルサイズが小さく、応答が速く、現在の純粋なフロントエンド全文検索の欠点を解消し、より良いユーザーエクスペリエンスを提供します。
