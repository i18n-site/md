## مروری بر راه‌حل‌های جستجوی متن کامل بدون سرور

برای وبسایتهای کوچک مانند اسناد/وبلاگهای شخصی که صرفاً ثابت هستند، بدون شک ساختن پشتیبان جستجوی تمام متن توسط خودتان بسیار سنگین است و جستجوی تمام متن بدون خدمات بدون شک وزن بهتری است.

راه‌حل‌های جستجوی متن کامل بدون سرور به دو دسته کلی تقسیم می‌شوند.

یکی از این دسته‌ها ارائه‌دهنده خدمات جستجوی شخص ثالث مشابه [algolia.com](//algolia.com) است که اجزای جستجوی متن کامل جلویی را ارائه می‌دهد.

این خدمات نیاز به پرداخت دارند و به دلیل مسائل مربوط به انطباق وب‌سایت، برای کاربران در سرزمین اصلی چین در دسترس نیستند.

این راه‌حل‌ها قابل استفاده به صورت آفلاین نیستند و نمی‌توان از آنها در اینترانت استفاده کرد، بنابراین محدودیت‌های زیادی دارند. در این مقاله به طور مفصل به آنها پرداخته نمی‌شود.

دسته دوم، جستجوی متن کامل جلویی خالص است.

جست و جوهای تمام متنی خالص و شناخته شده [lunrjs](/0) شامل [ ElasticLunr.js ] [https://github.com/weixsong/elasticlunr.js](%E5%9F%BA%E4%BA%8E%60lunrjs%60%E4%BA%8C%E6%AC%A1%E5%BC%80%E5%8F%91) هستند.

`lunrjs` دو روش برای ایجاد ایندکس دارد، اما هر دو مشکلات خاص خود را دارند.

1. فایل‌های شاخص از پیش ساخته شده

   از آنجایی که ایندکس شامل کلمات از همه اسناد است، اندازه آن بزرگ است.
   هر زمان که سندی اضافه یا تغییر می‌کند، باید یک فایل ایندکس جدید بارگذاری شود.
   این کار زمان انتظار کاربر را افزایش می‌دهد و پهنای باند زیادی مصرف می‌کند.

2. بارگذاری اسناد و ایجاد ایندکس در حین اجرا

   ایجاد ایندکس یک کار محاسباتی فشرده است.

علاوه بر `lunrjs`، راه‌حل‌های جستجوی کامل متن دیگری نیز وجود دارد، مانند:

[fusejs](https://www.fusejs.io) که شباهت بین رشته‌ها را برای جستجو محاسبه می‌کند.

این راه‌حل عملکرد بسیار ضعیفی دارد و نمی‌توان از آن برای جستجوی متن کامل استفاده کرد (مطالعه کنید [Fuse.js جستجوی طولانی بیش از 10 ثانیه طول می‌کشد، چگونه آن را بهینه کنیم؟](https://stackoverflow.com/questions/70984437/fuse-js-takes-10-seconds-with-semi-long-queries)).

[TinySearch](https://github.com/tinysearch/tinysearch) که از فیلتر بلوم برای جستجو استفاده می‌کند، نمی‌توان از آن برای جستجوی پیشوند استفاده کرد (مثلاً جستجوی `goo`، نتایج `good` و `google` را نشان نمی‌دهد) و نمی‌توان به اثر تکمیل خودکار مشابه دست یافت.

به دلیل نارضایتی از کاستی‌های راه‌حل‌های موجود، `i18n.site` راه‌حل جستجوی کامل متنی جلویی جدیدی را توسعه داده است که ویژگی‌های زیر را دارد:

1. پشتیبانی از جستجوی چند زبانه، حجم کم، حجم فشرده‌سازی شده `gzip` به `6.9KB` (در مقایسه با `lunrjs` که حجم `25KB` دارد)
1. ایجاد شاخص معکوس بر اساس `indexedb` که استفاده از حافظه کمتری دارد و سریع است
1. هنگامی که اسناد اضافه/اصلاح می شوند، فقط اسناد اضافه شده یا اصلاح شده مجدداً نمایه می شوند و میزان محاسبات را کاهش می دهد
1. پشتیبانی از جستجوی پیشوند و نمایش نتایج جستجو در حین تایپ کاربر
1. قابلیت استفاده آفلاین

در ادامه، جزئیات اجرایی فنی `i18n.site` به تفصیل معرفی خواهد شد.

## تقسیم‌بندی کلمات چند زبانه

تقسیم‌بندی کلمه از تقسیم‌بندی کلمه بومی مرورگر `Intl.Segmenter` استفاده می‌کند و همه مرورگرهای اصلی از این رابط پشتیبانی می‌کنند.

![](https://p.3ti.site/1727667759.avif)

کد تقسیم‌بندی کلمه به زبان `coffeescript` به شرح زیر است:

```coffee
SEG = new Intl.Segmenter 0, granularity: "word"

seg = (txt) =>
  r = []
  for {segment} from SEG.segment(txt)
    for i from segment.split('.')
      i = i.trim()
      if i and !'|`'.includes(i) and !/\p{P}/u.test(i)
        r.push i
  r

export default seg

export segqy = (q) =>
  seg q.toLocaleLowerCase()
```

در:

* `/\p{P}/` یک عبارت منظم است که با علائم نگارشی مطابقت دارد: `! " # $ % & ' ( ) * + , - . / : ; < = > ? @ [ \ ] ^ _ ` { | } `.</p><ul style=";text-align:right;direction:rtl"><li style=";text-align:right;direction:rtl"> `split('.')` به این دلیل است که `Firefox` بخش بندی کلمه مرورگر `.` را تقسیم نمی کند.</li>


## ساختمان ایندکس

پنج جدول ذخیره شی در `IndexedDB` ایجاد شده‌اند:

* `word` کلمه : id -
* `doc` : id - url - شماره نسخه سند
* `docWord` آرایه سند id - id :
* `prefix` : از پیشوند - کلمه id
* `rindex` : id - سند id : آرایه اعداد خط

از آرایه سند `url` و نسخه شماره `ver` عبور کنید و جستجو کنید که آیا سند در جدول `doc` وجود دارد یا خیر. اگر وجود نداشت، یک ایندکس معکوس ایجاد کنید. در عین حال، ایندکس معکوس را برای اسنادی که وارد نشده‌اند حذف کنید.

به این ترتیب می‌توان به ایندکس‌سازی افزایشی دست یافت و میزان محاسبات را کاهش داد.

در تعامل جلویی [،](https://dev.to/i18n-site/a-single-progress-uses-pure-css-to-achieve-animation-effects-2oo) نوار [پیشرفت](https://juejin.cn/post/7413586285954154522) بارگیری نمایه / می توان css جلوگیری از تاخیر در هنگام بارگیری برای اولین progress + نمایش داد.

### نوشتن همزمان IndexedDB بالا

این پروژه بر اساس کپسوله‌بندی ناهمزمان [idb](https://www.npmjs.com/package/idb) بر روی IndexedDB توسعه یافته است.

خواندن و نوشتن در IndexedDB ناهمزمان است. هنگام ایجاد شاخص، اسناد به صورت همزمان برای ایجاد شاخص بارگذاری می‌شوند.

برای جلوگیری از از دست دادن جزئی داده‌های ناشی از نوشتن رقابتی، می‌توانید به کد `coffeescript` زیر مراجعه کنید و یک حافظه `ing` بین خواندن و نوشتن برای رهگیری نوشته‌های رقیب اضافه کنید.

```coffee
pusher = =>
  ing = new Map()
  (table, id, val)=>
    id_set = ing.get(id)
    if id_set
      id_set.add val
      return

    id_set = new Set([val])
    ing.set id, id_set
    pre = await table.get(id)
    li = pre?.li or []

    loop
      to_add = [...id_set]
      li.push(...to_add)
      await table.put({id,li})
      for i from to_add
        id_set.delete i
      if not id_set.size
        ing.delete id
        break
    return

rindexPush = pusher()
prefixPush = pusher()
```

## جستجوی پیشوند بلادرنگ

برای نمایش نتایج جستجو در حین تایپ کاربر، به عنوان مثال، زمانی که `wor` وارد می‌شود، کلماتی که با پیشوند `wor` مانند `words` و `work` هستند نمایش داده می‌شوند.

![](https://p.3ti.site/1727684944.avif)

هسته جستجو از جدول `prefix` برای آخرین کلمه پس از تقسیم‌بندی کلمه استفاده می‌کند تا همه کلمات پیشوند با آن را پیدا کند و به ترتیب جستجو کند.

عملکرد ضد لرزش `debounce` نیز در تعامل جلویی (به شرح زیر اجرا می شود) برای کاهش فرکانس جستجوهای آغازگر ورودی کاربر و کاهش میزان محاسبه استفاده می شود.

```js
export default (wait, func) => {
  var timeout;
  return function(...args) {
    clearTimeout(timeout);
    timeout = setTimeout(func.bind(this, ...args), wait);
  };
}
```

## دقت و یادآوری

جستجو ابتدا کلمات کلیدی وارد شده توسط کاربر را تقسیم می‌کند.

فرض کنید `N` کلمه بعد از تقسیم‌بندی کلمه وجود دارد. در هنگام بازگرداندن نتایج، ابتدا نتایج حاوی تمام کلمات کلیدی و سپس نتایج حاوی `N-1`، `N-2`،...، `1` کلمه کلیدی بازگردانده می‌شود.

نتایج جستجوی نمایش داده شده ابتدا صحت پرس و جو را تضمین می‌کند و نتایج بارگذاری شده متعاقبا (با کلیک روی دکمه بارگذاری بیشتر) نرخ فراخوان را تضمین می‌کند.

![](https://p.3ti.site/1727684564.avif)

## بارگذاری بر اساس درخواست

برای بهبود سرعت پاسخگویی، جستجو از ژنراتور `yield` برای اجرای بارگذاری بر اساس درخواست استفاده می‌کند و `limit` بار که یک نتیجه پرس و جو می‌شود، بازگردانده می‌شود.

توجه داشته باشید که هر بار که بعد از `yield` دوباره جستجو می‌کنید، باید تراکنش پرس و جوی `IndexedDB` را دوباره باز کنید.

## قابلیت استفاده آفلاین

جدول ایندکس، متن اصلی را ذخیره نمی‌کند، فقط کلمات را ذخیره می‌کند که میزان ذخیره‌سازی را کاهش می‌دهد.

برجسته کردن نتایج جستجو مستلزم بارگذاری مجدد متن اصلی است و تطبیق `service worker` می‌تواند از درخواست‌های مکرر شبکه جلوگیری کند.

در عین حال، به دلیل اینکه `service worker` همه مقالات را در حافظه پنهان نگه می‌دارد، پس از انجام جستجوی کاربر، کل وب‌سایت، از جمله جستجو، به صورت آفلاین در دسترس است.
