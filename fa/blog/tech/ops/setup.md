# راه‌اندازی سرور (استفاده از پارتیشن BTRFS، عکس‌های فوری برنامه‌ریزی شده snapper)

## خرید سرور از contabo

contabo یک ارائه‌دهنده VPS ارزان در آلمان با سابقه طولانی (بیش از 20 سال) و کاربران بسیار زیاد (بیش از 200000) است، بنابراین نیازی به نگرانی در مورد فرار نیست.

VPS خرید [contabo.com](//contabo.com) به یاد داشته باشید که قیمت یورو را انتخاب کنید، که ارزان‌تر است.

![](https://i-01.eu.org/1718285782.avif)

contabo هزینه نصب یک‌بار مصرف دارد.
با این حال، هر ماه چند روز وجود دارد که هزینه نصب رایگان است، فقط ثبت‌نام کنید و به ایمیل‌های تبلیغاتی توجه کنید.

علاوه بر هزینه منطقه آزاد در اروپا، سایر مناطق دارای سطوح متفاوتی از افزایش قیمت هستند.

من یک سرور اروپایی را انتخاب کردم، زیرا پس از استفاده از Cloudflare، تفاوت زیادی بین سرورها وجود ندارد.

پیکربندی نهایی به شرح زیر است: 16 گیگابایت حافظه، 6 CPU، 200 گیگابایت NVMe، ماهانه 9.50 یورو، معادل 74.43 یوان چین.

![](https://i-01.eu.org/1718286008.avif)

**این قیمت تقریباً یک دهم قیمت آلی‌کلاود و آمازون است**.

![](https://i-01.eu.org/1718356956.avif)

من از [Yet-Another-Bench-Script](https://github.com/masonr/yet-another-bench-script) برای اجرای عملکرد و پارامترها استفاده کردم:

```
curl -sL yabs.sh | bash -s -- -s https://www.vpsbenchmarks.com/yabs/upload
```

نمایش مدل CPU `AMD EPYC 9224 24-Core Processor` و امتیاز عملکرد تک هسته‌ای 1285
، امتیاز چند هسته‌ای 4277. نتایج بیشتر را در [vpsbenchmarks.com](https://www.vpsbenchmarks.com/yabs/contabo-6c-16gb-3fb7d8) مشاهده کنید

این می‌تواند اطمینان حاصل کند که سرورها در یک دیتاسنتر قرار دارند و برای ساختارهای بعدی با قابلیت اطمینان بالا مناسب هستند.

سرورهای contabo گهگاه قطع می‌شوند، اما با استفاده از قابلیت اطمینان بالا، دسترسی به وب‌سایت تضمین می‌شود.

کاربران جدید هنگام خرید باید کارت شناسایی خود را برای احراز هویت واقعی ارسال کنند.

## پیکربندی ورود بدون رمز عبور

بعد از چند دقیقه، همه سرورها فعال شدند

اولین کار، ویرایش فایل `~/.ssh/config` برای تنظیم نام مستعار IP و سپس استفاده از `ssh-copy-id root@c2` برای پیکربندی ورود بدون رمز عبور است.

![](https://i-01.eu.org/1718287198.avif)

شما می‌توانید این اسکریپت [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) را تغییر دهید تا پیکربندی اولیه را سریعتر انجام دهید.

پس از تکمیل پیکربندی بدون رمز عبور، می‌توانید از [pdsh](https://github.com/chaos/pdsh) برای عملیات دسته‌ای استفاده کنید، که بسیار راحت است.

ابتدا به صورت دستی ipv6 را فعال کرده و تست کنید

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

سپس با یک کلیک سیستم را به اوبونتو 24 ارتقاء دهید و نرم‌افزارهای رایج را نصب کنید.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 به معنای تغییر زبان به چینی و تنظیم منطقه زمانی به پکن است.

نرم‌افزارهای پیش‌فرض نصب شده بر اساس عادات استفاده شخصی من پیکربندی شده‌اند، مانند [mise](https://github.com/jdx/mise)، [direnv](https://direnv.net/) و غیره. در صورت نیاز می‌توانید مخزن [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) را کپی کرده و تغییرات مورد نظر خود را اعمال کنید.

این مرحله ممکن است مدت زمان بیشتری طول بکشد، پس از نیم ساعت انتظار، ارتقاء و نصب نرم‌افزارهای رایج با موفقیت انجام می‌شود. می‌توانید با دستور زیر این موضوع را تایید کنید:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

![](https://i-01.eu.org/1718290346.avif)

## mosh سرعت ssh را افزایش می‌دهد

شبکه از اروپا به چین پایدار نیست، توصیه می‌شود از [mosh](//mosh.org) برای افزایش سرعت `ssh` استفاده کنید.

اسکریپت `c0` را در دایرکتوری محلی `~/.bin` (که باید به متغیر محیطی `PATH` اضافه شود) به شرح زیر ایجاد کنید:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

پس از ایجاد، مجوزهای اجرایی را اعطا کنید و لینک‌های نرم‌افزاری `c1` و `c2` را ایجاد کنید.

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

در آینده می‌توانید مستقیماً `c0` و `c1` را وارد کنید تا از `mosh` برای اتصال مستقیم پرسرعت و ورود به [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html) استفاده کنید.

## تبدیل فرمت پارتیشن از ext4 به btrfs

[btrfs](https://wikipedia.org/wiki/Btrfs) می‌تواند عکس‌های فوری از دیسک تهیه کند که برای پشتیبان‌گیری مفید است.

ابتدا وارد سیستم نجات شوید

![](https://i-01.eu.org/1718300448.avif)

Debian Rescue را انتخاب کنید

![](https://i-01.eu.org/1718300753.avif)

منتظر بمانید تا سرور راه‌اندازی مجدد شود

در هنگام ورود به ssh، تغییرات اثر انگشت مشاهده می‌شود، از دستور `ssh-keygen -R ip` برای پاک کردن آن استفاده کنید

سپس اسکریپت زیر را برای تبدیل پارتیشن اجرا کنید:

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

اسکریپت پس از موفقیت آمیز شدن تبدیل، سیستم را به طور خودکار راه‌اندازی مجدد می‌کند.

### مشاهده نسبت فشرده‌سازی btrfs

به طور پیش‌فرض به طور خودکار فشرده می‌شود. دستور بررسی نسبت فشرده‌سازی به شرح زیر است:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### فشرده‌سازی دستی

اگر یک کمال‌گرا هستید، می‌توانید فشرده‌سازی کامل دیسک را به صورت دستی نیز انجام دهید

```bash
btrfs filesystem defragment -r -czstd /
```

همانطور که در شکل زیر نشان داده شده است، مشاهده می‌کنید که پس از فشرده‌سازی دستی در سیستم جدید، نزدیک به 100 مگابایت بیشتر فشرده شده است.

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### با استفاده از snapper عکس‌های فوری Btrfs ایجاد کنید

[snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html) بهترین شریک برای پشتیبان‌گیری Btrfs است.

اسکریپت اولیه دایرکتوری `~/ubuntu` را ایجاد می‌کند، فقط دستور زیر را اجرا کنید:

```
~/ubuntu/snapper/init.sh
```

محتوای اسکریپت را در [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh) مشاهده کنید

از دستور `snapper list-configs` برای مشاهده حجم‌های فرعی که عکس‌های فوری برای آنها تنظیم شده‌اند استفاده کنید

![](https://i-01.eu.org/1718302689.avif)

از دستور `snapper -c mnt list` برای مشاهده عکس‌های فوری یک زیرجلد خاص استفاده کنید

از فایل `/etc/snapper/configs/mnt` برای مشاهده تنظیمات پشتیبان خاص این زیرجلد استفاده کنید

مراجع بیشتر:

[به راحتی با عکس‌های فوری Btrfs بازی کنید snapper](https://zhuanlan.zhihu.com/p/31082518)

## عکس‌های فوری سرور contabo

contabo عملکرد عکس‌های فوری سرور را ارائه می‌دهد، اما پشتیبانی از عکس‌های فوری برنامه‌ریزی شده را ندارد. من یک [github action](https://github.com/i18n-cron/cron/actions/workflows/contabo.snapshot.yml) برای ایجاد عکس‌های فوری روزانه نوشته‌ام.

![](https://i-01.eu.org/1718304206.avif)

برای بازیابی عکس‌های فوری سرور، ابتدا در منوی مدیریت روی Crontol کلیک کنید

![](https://i-01.eu.org/1718304275.avif)

سپس روی Snapshots کلیک کنید تا بازیابی انجام شود.

![](https://i-01.eu.org/1718304063.avif)

با این حال، جزئیات این عکس‌های فوری بسیار زیاد است، پس از بازیابی، داده‌های تازه اضافه شده به طور کامل از بین خواهند رفت، بنابراین لطفاً با احتیاط از آن استفاده کنید.

معمولاً استفاده از عکس‌های فوری btrfs راحت‌تر است.

## جزئیات سیستم نجات

وارد سیستم نجات شوید، پس از اینکه پورت 22 با [tcping](https://github.com/paradiseduo/tcping) در دسترس قرار گرفت، ممکن است به مدت 2-3 دقیقه با رمز عبور تعیین شده نتوانید به آن دسترسی پیدا کنید، لطفاً صبر کنید.

```
tcping ip 22 -c 9999
```

دسترسی به بندر قابل نظارت است.

### چگونه به هارد دیسک دسترسی پیدا کنید و عکس‌های فوری را بازیابی کنید

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

می‌توانید عکس‌های فوری را مشاهده کنید، به عنوان مثال:

```
snapper -c etc list
```

مشاهده تغییرات فایل لحظه‌ای

```
snapper -c etc status 5..6
```

مشاهده متن متفاوت خاص

```
snapper -c etc diff 3..5
```

بازیابی عکس‌های فوری: به عنوان مثال، تمام تغییرات بعد از عکس‌های فوری 5 را لغو کنید

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### فعال کردن ماندگاری گزارش

فعال کردن ماندگاری گزارش در سیستم نجات نیاز به ایجاد دستی دایرکتوری مربوطه و انجام مراحل اضافی دارد، زیرا ممکن است سرویس systemd در حال اجرا نداشته باشید. در اینجا مراحلی وجود دارد که می‌توانید امتحان کنید:

ابتدا chroot کنید، سپس

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

راه‌اندازی مجدد در سیستم اوبونتو: پس از انجام تغییرات بالا، باید دستگاه خود را به طور عادی راه‌اندازی مجدد کنید و به جای سیستم نجات، در سیستم اوبونتو خود راه‌اندازی کنید.

به این ترتیب، systemd-journald با پیکربندی جدید شروع به کار می‌کند و شروع به ذخیره‌سازی لاگ‌ها به طور مداوم روی دیسک سخت می‌کند.

به خاطر داشته باشید که عملیات فوق در سیستم نجات انجام می‌شود، بنابراین تا زمانی که سیستم راه‌اندازی مجدد نشود و دیگر در حالت نجات نباشد، این تغییرات اعمال نمی‌شوند.

این بدان معناست که شما فوراً هیچ تغییری را در لاگ‌های موجود مشاهده نمی‌کنید که پایدار می‌شوند، اما همه لاگ‌هایی که پس از آن ایجاد می‌شوند، باقی خواهند ماند.

به این ترتیب می‌توانید لاگ‌های سرویس ssh را مشاهده کنید

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## نکاتی در مورد مشکلات پیچیده

### حل مشکل عجیب: سرور نمی‌تواند ssh

من قبلاً از اکشن github برای ایجاد یک اسکریپت کامپایل و استقرار خودکار استفاده کردم، اما متوجه شدم که سرویس استقرار نمی‌تواند ssh باشد.

ورود به سیستم نجات و مشاهده لاگ‌ها نشان می‌دهد که مسئله مجوز است:

`Missing privilege separation directory: /run/sshd`

از دستور زیر می‌توانید این دایرکتوری را ایجاد کنید:

```
chown root:root /
systemd-tmpfiles --create
```

پس از بررسی‌های مکرر، متوجه شدم که کاربر `uid` از tar در github action `1001` است

باز کردن بسته‌بندی باید انجام شود اگر پارامتر `--no-same-owner` وجود نداشته باشد، برای کاربر `uid` `1001` کاهش می‌یابد tar

در نتیجه rsync گروه کاربری دایرکتوری ریشه را تغییر می‌دهد و پس از راه‌اندازی مجدد نمی‌توانید وارد ssh شوید.

اسکریپت فشرده‌سازی را به صورت زیر تغییر دهید:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
