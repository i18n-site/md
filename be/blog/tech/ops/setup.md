## Наладзьце Ўваход Без Пароля

Праз некалькі хвілін усе серверы былі актываваныя.

Першае, што трэба адрэдагаваць на лакальнай машыне `~/.ssh/config` усталяваць псеўданім для IP а затым выкарыстоўваць `ssh-copy-id root@c2` , каб наладзіць уваход без пароля.

<img alt="" src="https://i-01.eu.org/1718287198.avif">

Вы можаце змяніць гэты [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) , каб хутчэй завяршыць канфігурацыю ініцыялізацыі.

Пасля завяршэння канфігурацыі без пароля вы можаце выкарыстоўваць [pdsh](https://github.com/chaos/pdsh) пакетна, што зручна.

Першы - уручную ўключыць ipv6 і праверыць

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

<img alt="" src="https://i-01.eu.org/1718287969.avif">

Затым абнавіце сістэму да ubuntu 24 адным пстрычкай мышы і ўсталюйце часта выкарыстоўванае праграмнае забеспячэнне.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 азначае пераключэнне мовы на кітайскую і часавы пояс на Пекін.

Праграмнае забеспячэнне, усталяванае па змаўчанні, наладжана ў адпаведнасці з маімі асабістымі звычкамі выкарыстання, напрыклад [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) і г.д. Пры неабходнасці вы можаце скапіяваць [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) warehouse і змяніць яго па меры неабходнасці.

Гэты крок займае шмат часу. Пасля чакання на працягу паўгадзіны абнаўленне і ўстаноўка часта выкарыстоўванага праграмнага забеспячэння будуць паспяховымі. Вы можаце пацвердзіць гэта з дапамогай наступнай каманды:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh Паскарае ssh

Сетка з Еўропы ў Кітай не стабільная. Прыходзьце `ssh` каб паскорыць [mosh](//mosh.org)

Стварыце скрыпт `c0` у `~/.bin` лакальнай машыны (галоўным чынам дададзены ў зменную асяроддзя `PATH` ) наступным чынам:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

Пасля стварэння дайце дазволы на выкананне і праграмную `c2` `c1`

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

У далейшым вы можаце напрамую ўвесці `c0` , `c1` і вы можаце выкарыстоўваць `mosh` высакахуткаснае прамое злучэнне і ўвесці [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

Дачакайцеся перазапуску сервера

Затым запусціце наступны сцэнар, каб пераўтварыць раздзел

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &

```

Сцэнар аўтаматычна перазапусціць сістэму пасля паспяховага пераўтварэння.

### Прагляд Каэфіцыента Сціску btrfs

Па змаўчанні ён будзе аўтаматычна сціскацца, каб праверыць ступень сціску:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### Ручное Сцісканне

Калі вы перфекцыяніст, вы таксама можаце выканаць поўнае сцісканне дыска ўручную

```bash
btrfs filesystem defragment -r -czstd /
```

Як паказана на малюнку ніжэй, вы бачыце, што пасля ручнога сціску ў новай сістэме было сціснута яшчэ амаль на 100 МБ.

### Як Атрымаць Доступ Да Жорсткага Дыска Пасля Ўваходу Ў Rescue &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Вы можаце праглядзець здымак, напрыклад

```
snapper -c etc list
```

Праглядзіце змены файла здымка

```
snapper -c etc status 5..6
```

Прагляд канкрэтнага тэксту розніцы

```
snapper -c etc diff 3..5
```

Аднавіць здымак: напрыклад, адмяніць усе змены пасля здымка 5

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Уключыць Захаванне Журнала

Каб уключыць захаванне журнала ў сістэме выратавання, вы павінны ўручную стварыць адпаведны каталог і выканаць некаторыя дадатковыя крокі, таму што ў вас можа не быць запушчанай службы systemd. Вось крокі, якія вы можаце паспрабаваць:

Спачатку chroot, затым

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Перазагрузіце сістэму Ubuntu: пасля ўнясення вышэйзгаданых змяненняў вам трэба перазагрузіць машыну ў звычайным рэжыме і ў сістэму Ubuntu замест сістэмы выратавання.

Такім чынам, systemd-journald запусціцца з новай канфігурацыяй і пачне пастаянна захоўваць журналы на цвёрдым дыску.

Майце на ўвазе, што апісаныя вышэй аперацыі выконваюцца ў выратавальнай сістэме, таму гэтыя змены ўступяць у сілу толькі пасля перазагрузкі сістэмы, а не ў рэжыме выратавання.

Гэта азначае, што вы не адразу ўбачыце, што змены ў існуючых журналах стануць пастаяннымі, але ўсе журналы, створаныя пасля гэтага, будуць захоўвацца.

Такім чынам можна праглядаць журналы службы ssh

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Нататкі Аб Цяжкіх І Складаных Захворваннях

### Вырашыце Дзіўную Праблему: Сервер Не Можа Выкарыстоўваць ssh

Раней я выкарыстоўваў дзеянне github для аўтаматычнай кампіляцыі і сцэнарыя разгортвання, але выявіў, што служба разгортвання не можа выкарыстоўваць ssh.

Уваход у сістэму выратавання і прагляд журналаў - гэта праблема дазволу:

`Missing privilege separation directory: /run/sshd`

Вы можаце стварыць гэты каталог з дапамогай наступнай каманды

```
chown root:root /
systemd-tmpfiles --create
```

Пасля неаднаразовых даследаванняў гэта таму github action што карыстальнік, які загрузіў `1001` tar `uid`

tar Распакоўка павінна выконвацца без параметраў `--no-same-owner` яна будзе распакавана ў `uid` для карыстальніка `1001`

У выніку rsync зменіць групу карыстальнікаў каранёвага каталога, і тады вы не зможаце ўвайсці ў ssh пасля перазапуску.

Змяніце скрыпт дэкампрэсіі наступным чынам:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
