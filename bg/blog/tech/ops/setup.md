## Конфигурирайте Влизане Без Парола

След няколко минути всички сървъри бяха активирани.

Първото нещо е да редактирате локалната машина `~/.ssh/config` Задайте псевдоним за IP и след това използвайте `ssh-copy-id root@c2` , за да конфигурирате влизане без парола.

<img alt="" src="https://i-01.eu.org/1718287198.avif">

Можете да промените този [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) , за да завършите по-бързо конфигурацията за инициализация.

След като конфигурацията без парола е завършена, можете да я използвате [pdsh](https://github.com/chaos/pdsh) Може да се работи на партиди, което е удобно.

Първият е ръчно активиране на ipv6 и тестване

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

<img alt="" src="https://i-01.eu.org/1718287969.avif">

След това надстройте системата до ubuntu 24 с едно кликване и инсталирайте често използван софтуер.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 означава превключване на езика на китайски и часовата зона на Пекин.

Инсталираният по подразбиране софтуер е конфигуриран според личните ми навици на използване, като [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) и т.н. Ако е необходимо, можете да копирате [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) warehouse и да го промените според нуждите.

Тази стъпка отнема много време, след като изчакате половин час, надстройката и инсталирането на често използван софтуер ще бъдат успешни. Можете да го потвърдите със следната команда:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh Ускорява ssh

Мрежата от Европа до Китай не е стабилна. Препоръчително е да `ssh` използвате [mosh](//mosh.org)

Създайте `c0` скрипт в директорията `~/.bin` на локалната машина (главно добавена към променливата на средата `PATH` ), както следва:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

След създаването дайте разрешения за изпълним файл и мека връзка `c1` `c2` .

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

В бъдеще можете директно да въведете `c0` , `c1` и можете да използвате `mosh` за да се свържете директно и да въведете [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## Конвертиране На Формат На Дял От ext4 В btrfs

[btrfs](https://wikipedia.org/wiki/Btrfs) Може да прави моментни снимки на диска за лесно архивиране.

Първо влезте в системата за спасяване

<img alt="" src="https://i-01.eu.org/1718300448.avif">

Изберете Debian Rescue

<img alt="" src="https://i-01.eu.org/1718300753.avif">

Изчакайте сървърът да се рестартира

Промените на пръстовия отпечатък ще се появят `ssh-keygen -R ip` когато ssh

След това изпълнете следния скрипт, за да конвертирате дяла

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

Скриптът автоматично ще рестартира системата след успешно преобразуване.

### Вижте Коефициента На Компресия На btrfs

Той автоматично ще компресира по подразбиране, за да провери съотношението на компресия, както следва:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### Ръчна Компресия

Ако сте перфекционист, можете също така да извършите пълна компресия на диска, както следва

```bash
btrfs filesystem defragment -r -czstd /
```

Както е показано на фигурата по-долу, можете да видите, че след ръчно компресиране в новата система бяха компресирани още близо 100MB.

### Създайте Резервно Копие На Моментна Снимка На Btrfs Със snapper

[snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html) е най-добрият партньор за Btrfs backup.

Скриптът за инициализиране ще създаде директорията `~/ubuntu` , просто изпълнете следната команда:

Ще се видим в съдържанието на сценария [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

## Подробности За Спасителната Система

Влезте в системата за спасяване 22 След като портът е [tcping](https://github.com/paradiseduo/tcping) , няма да можете да получите достъп до него с зададената парола за 2-3 минути.

```
tcping ip 22 -c 9999
```

Достъпността на портовете може да се наблюдава.

### Как Да Получите Достъп До Твърдия Диск След Влизане В Rescue & Restore Snapshot

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Можете например да видите моментната снимка

```
snapper -c etc list
```

Преглед на промените във файла за моментна снимка

```
snapper -c etc status 5..6
```

Преглед на конкретен текст на разликата

```
snapper -c etc diff 3..5
```

Възстановяване на моментна снимка: Например отмяна на всички промени след моментна снимка 5

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Активиране На Постоянството На Регистрационния Файл

Разрешаването на постоянство на журнала под спасителна система изисква ръчно да създадете съответната директория и да изпълните някои допълнителни стъпки, тъй като може да нямате работеща услуга systemd. Ето стъпките, които можете да опитате:

Първо chroot, тогава

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Рестартирайте вашата Ubuntu система: След като направите горните промени, трябва да рестартирате нормално вашата машина и във вашата Ubuntu система вместо системата за спасяване.

По този начин systemd-journald ще започне с новата конфигурация и ще започне постоянно да съхранява регистрационни файлове на твърдия диск.

Имайте предвид, че горните операции се извършват на спасителна система, така че тези промени ще влязат в сила само след като системата се рестартира и вече не е в спасителен режим.

Това означава, че няма да видите незабавно промените в съществуващите регистрационни файлове да станат постоянни, но всички регистрационни файлове, генерирани след това, ще бъдат запазени.

По този начин можете да видите регистрационните файлове на ssh услугата

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Бележки За Трудни И Сложни Заболявания

### Решете Странния Проблем: Сървърът Не Може Да Използва ssh

Използвах github действие, за да направя автоматична компилация и скрипт за внедряване преди, но открих, че услугата за внедряване не може да ssh.

Влизането в системата за спасяване и разглеждането на регистрационните файлове е проблем с разрешение:

`Missing privilege separation directory: /run/sshd`

Можете да създадете тази директория със следната команда

```
chown root:root /
systemd-tmpfiles --create
```

След многократни проучвания, това е `1001` github action защото потребителят, който е качил tar `uid`

tar Разопаковането трябва да се извърши без параметри `--no-same-owner` то ще бъде декомпресирано в `uid` за `1001` потребител

В резултат на това rsync ще промени потребителската група на основната директория и след това няма да можете да влезете в ssh след рестартиране.

Променете скрипта за декомпресия, както следва:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
