## mosh Ускорява ssh

Мрежата от Европа до Китай не е стабилна. Препоръчително е да използвате `ssh` [mosh](//mosh.org)

Създайте `c0` скрипта в директорията `~/.bin` на локалната машина (главно добавена към променливата на средата `PATH`), както следва:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

След създаването на скрипта, дайте му изпълними права и създайте меки връзки `c1` и `c2`.

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

В бъдеще можете директно да използвате `c0` и `c1`, за да се свържете с помощта на `mosh` и да влезете в [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html).

## Подробности за Спасителната Система

След като влезете в системата за спасяване и портът 22 е [tcping](https://github.com/paradiseduo/tcping), няма да можете да получите достъп до него с зададената парола за 2-3 минути.

Достъпността на портовете може да се наблюдава.

### Можете например да видите моментните снимки:

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Преглед на промените във файла за моментна снимка

```
snapper -c etc list
```

Преглед на конкретния текст на разликите

```
snapper -c etc status 5..6
```

Преглед на конкретен текст на разликата

```
snapper -c etc diff 3..5
```

Активирайте постоянството на журнала

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Активиране на постоянството на регистрацията на файла

Първо изпълнете chroot, след това

Първо chroot, тогава

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Така systemd-journald ще започне с новата конфигурация и ще започне постоянно да съхранява регистрационни файлове на твърдия диск.

Имайте предвид, че горните операции се извършват в системата за спасяване, така че тези промени няма да влязат в сила, докато системата не се рестартира и вече не е в спасителен режим.

Това означава, че няма веднага да видите промени в съществуващите регистрационни файлове, които стават постоянни, но всички генерирани след това регистрационни файлове ще бъдат запазени.

Така можете да видите регистрационните файлове на услугата SSH.

Бележки за трудни и сложни проблеми

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Решаване на странен проблем: Сървърът не може да използва SSH

### Използвах github action за автоматична компилация и внедряване на скрипт, но открих, че услугата за внедряване не може да използва SSH

След влизане в системата за спасяване и разглеждане на регистрационните файлове, установих, че има проблем с разрешението:

Можете да създадете тази директория със следната команда:

`Missing privilege separation directory: /run/sshd`

Можете да създадете тази директория с командата:

```
chown root:root /
systemd-tmpfiles --create
```

След многократни проучвания това е така, github action потребител `uid` на tar е `1001`

tar Разопаковането трябва да бъде, ако няма параметър `--no-same-owner` , той ще бъде декомпресиран до `uid` за `1001` потребители

Променете скрипта за разопаковане, както следва:

Променете скрипта за разопаковане, както следва:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
