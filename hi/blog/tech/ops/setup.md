## पासवर्ड-मुक्त लॉगिन कॉन्फ़िगर करें

कुछ मिनटों के बाद, सभी सर्वर सक्रिय हो गए।

पहली बात IP कि उपनाम सेट करने के लिए स्थानीय मशीन के `~/.ssh/config` संपादित करें, और फिर पासवर्ड-मुक्त लॉगिन को कॉन्फ़िगर करने के लिए `ssh-copy-id root@c2` का उपयोग करें।

![](https://i-01.eu.org/1718287198.avif)

आरंभीकरण कॉन्फ़िगरेशन को तेज़ी से पूरा करने के लिए आप इस [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) संशोधित कर सकते हैं।

पासवर्ड-मुक्त कॉन्फ़िगरेशन पूरा होने के बाद, आप इसका उपयोग कर सकते हैं [pdsh](https://github.com/chaos/pdsh) इसे बैचों में संचालित किया जा सकता है, जो सुविधाजनक है।

पहला है IPv6 को मैन्युअल रूप से सक्षम करना और परीक्षण करना

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

फिर एक क्लिक से सिस्टम को ubuntu 24 में अपग्रेड करें और आमतौर पर उपयोग किए जाने वाले सॉफ़्टवेयर इंस्टॉल करें।

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 का अर्थ है भाषा को चीनी और समय क्षेत्र को बीजिंग में बदलना।

डिफ़ॉल्ट रूप से इंस्टॉल किया गया सॉफ़्टवेयर मेरी व्यक्तिगत उपयोग की आदतों के अनुसार कॉन्फ़िगर किया गया है, जैसे [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) , आदि। यदि आवश्यक हो, तो आप [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) वेयरहाउस की प्रतिलिपि बना सकते हैं और आवश्यकतानुसार इसे संशोधित कर सकते हैं।

इस चरण में लंबा समय लगता है। आधे घंटे के इंतजार के बाद आमतौर पर उपयोग किए जाने वाले सॉफ़्टवेयर का अपग्रेड और इंस्टॉलेशन सफल हो जाएगा। आप निम्न आदेश से इसकी पुष्टि कर सकते हैं:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh ssh को गति देता है

यूरोप से चीन तक का नेटवर्क स्थिर नहीं है। `ssh` गति बढ़ाने के लिए इसका उपयोग करने की अनुशंसा की जाती है [mosh](//mosh.org)

स्थानीय निर्देशिका `~/.bin` में `c0` स्क्रिप्ट बनाएं (मुख्य रूप से पर्यावरण चर `PATH` में जोड़ा गया) निम्नानुसार:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

निर्माण के बाद, निष्पादन योग्य अनुमतियाँ और सॉफ्ट-लिंक `c1` और `c2` प्रदान करें।

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

भविष्य में, आप `mosh` हाई-स्पीड डायरेक्ट कनेक्शन का उपयोग करने के लिए सीधे `c0` और `c1` दर्ज कर सकते हैं और दर्ज कर सकते हैं [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## विभाजन प्रारूप को ext4 से btrfs में बदलें

आसान बैकअप के लिए डिस्क स्नैपशॉट ले सकते हैं [btrfs](https://wikipedia.org/wiki/Btrfs)

सबसे पहले बचाव प्रणाली में प्रवेश करें

![](https://i-01.eu.org/1718300448.avif)

डेबियन रेस्क्यू चुनें

![](https://i-01.eu.org/1718300753.avif)

सर्वर के पुनरारंभ होने की प्रतीक्षा करें

फ़िंगरप्रिंट परिवर्तन ssh , इसे साफ़ करने के लिए `ssh-keygen -R ip` उपयोग करें

फिर विभाजन को परिवर्तित करने के लिए निम्न स्क्रिप्ट चलाएँ

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

रूपांतरण सफल होने के बाद स्क्रिप्ट स्वचालित रूप से सिस्टम को पुनः आरंभ करेगी।

### btrfs संपीड़न अनुपात देखें

यह डिफ़ॉल्ट रूप से स्वचालित रूप से संपीड़ित हो जाएगा। संपीड़न अनुपात की जांच करने का आदेश इस प्रकार है:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### मैनुअल संपीड़न

यदि आप पूर्णतावादी हैं, तो आप मैन्युअल रूप से पूर्ण डिस्क संपीड़न भी कर सकते हैं: आदेश इस प्रकार है:

```bash
btrfs filesystem defragment -r -czstd /
```

जैसा कि नीचे दिए गए चित्र में दिखाया गया है, आप देख सकते हैं कि नए सिस्टम में मैन्युअल संपीड़न के बाद, लगभग 100 एमबी अधिक संपीड़ित किया गया था।

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### स्नैपर के साथ Btrfs स्नैपशॉट बैकअप बनाएं

Btrfs बैकअप के लिए सबसे अच्छा भागीदार है [snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html)

आरंभीकरण स्क्रिप्ट `~/ubuntu` निर्देशिका बनाएगी, बस निम्नलिखित कमांड चलाएँ:

```
~/ubuntu/snapper/init.sh
```

स्क्रिप्ट सामग्री में मिलते हैं [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

उन सबवॉल्यूम को देखने के लिए `snapper list-configs` उपयोग करें जिनके लिए स्नैपशॉट सेट किए गए हैं।

![](https://i-01.eu.org/1718302689.avif)

किसी सबवॉल्यूम का स्नैपशॉट देखने के लिए `snapper -c mnt list` उपयोग करें

`/etc/snapper/configs/mnt` इस सबवॉल्यूम की विशिष्ट बैकअप सेटिंग्स देख सकता है

अधिक संदर्भ:

[Btrfs के स्नैपशॉट फ़ंक्शन के साथ आसानी से खेलें snapper](https://zhuanlan.zhihu.com/p/31082518)

## बचाव प्रणाली का विवरण

बचाव प्रणाली दर्ज करें 22 पोर्ट [tcping](https://github.com/paradiseduo/tcping) पहुंच योग्य होने के बाद, आप इसे 2-3 मिनट के लिए सेट पासवर्ड के साथ एक्सेस नहीं कर पाएंगे। बस एक क्षण प्रतीक्षा करें और यह ठीक हो जाएगा।

```
tcping ip 22 -c 9999
```

बंदरगाह पहुंच की निगरानी की जा सकती है।

### बचाव दर्ज करने के बाद हार्ड ड्राइव तक कैसे पहुंचें & स्नैपशॉट पुनर्स्थापित करें

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

उदाहरण के लिए, आप स्नैपशॉट देख सकते हैं

```
snapper -c etc list
```

स्नैपशॉट फ़ाइल परिवर्तन देखें

```
snapper -c etc status 5..6
```

विशिष्ट भिन्न पाठ देखें

```
snapper -c etc diff 3..5
```

स्नैपशॉट पुनर्स्थापित करें: उदाहरण के लिए, स्नैपशॉट 5 के बाद सभी परिवर्तनों को पूर्ववत करें

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### लॉग दृढ़ता सक्षम करें

एक बचाव प्रणाली के तहत लॉग दृढ़ता को सक्षम करने के लिए आपको मैन्युअल रूप से संबंधित निर्देशिका बनाने और कुछ अतिरिक्त कदम उठाने की आवश्यकता होती है, क्योंकि आपके पास एक चालू सिस्टमडी सेवा नहीं हो सकती है। यहां वे चरण दिए गए हैं जिन्हें आप आज़मा सकते हैं:

पहले चिरौट, फिर

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

अपने उबंटू सिस्टम में रीबूट करें: उपरोक्त परिवर्तन करने के बाद, आपको बचाव प्रणाली के बजाय अपनी मशीन को सामान्य रूप से और अपने उबंटू सिस्टम में रीबूट करना होगा।

इस तरह, सिस्टमड-जर्नल नए कॉन्फ़िगरेशन के साथ शुरू होगा और हार्ड डिस्क पर लगातार लॉग संग्रहीत करना शुरू कर देगा।

ध्यान रखें कि उपरोक्त ऑपरेशन एक बचाव प्रणाली पर किए जाते हैं, इसलिए वे परिवर्तन तब तक प्रभावी नहीं होंगे जब तक कि सिस्टम रिबूट न हो जाए और बचाव मोड में न हो जाए।

इसका मतलब यह है कि आप मौजूदा लॉग में तुरंत कोई परिवर्तन नहीं देखेंगे जो लगातार बने रहेंगे, लेकिन उसके बाद उत्पन्न सभी लॉग बने रहेंगे।

इस तरह आप ssh सेवा के लॉग देख सकते हैं

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## कठिन और जटिल रोगों पर नोट्स

### अजीब समस्या का समाधान करें: सर्वर ssh नहीं कर सकता

मैंने पहले स्वचालित संकलन और परिनियोजन स्क्रिप्ट बनाने के लिए जीथब एक्शन का उपयोग किया था, लेकिन पाया कि परिनियोजन सेवा ssh नहीं हो सकी।

बचाव प्रणाली में प्रवेश करना और लॉग देखना एक अनुमति समस्या है:

`Missing privilege separation directory: /run/sshd`

आप इस निर्देशिका को निम्न आदेश से बना सकते हैं

```
chown root:root /
systemd-tmpfiles --create
```

बार-बार जांच के बाद, ऐसा इसलिए है github action tar का उपयोगकर्ता `uid` `1001` है

यदि कोई पैरामीटर `--no-same-owner` नहीं है tar अनपैकिंग होनी चाहिए, इसे `1001` उपयोगकर्ताओं के लिए `uid` पर विघटित किया जाएगा

परिणामस्वरूप, rsync रूट निर्देशिका के उपयोगकर्ता समूह को बदल देगा, और फिर आप पुनरारंभ करने के बाद ssh में लॉग इन नहीं कर पाएंगे।

डीकंप्रेसन स्क्रिप्ट को इस प्रकार संशोधित करें:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
