## Сырсөзсүз Логинди Конфигурациялаңыз

Бир нече мүнөттөн кийин серверлердин бардыгы активдешти.

Биринчиси, лакап ат коюу үчүн жергиликтүү машинанын `~/.ssh/config` IP , анан сырсөзсүз кирүү конфигурациялоо үчүн `ssh-copy-id root@c2` колдонуңуз.

![](https://i-01.eu.org/1718287198.avif)

Сиз инициализация конфигурациясын тезирээк бүтүрүү үчүн бул [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) өзгөртө аласыз.

Сырсөзсүз конфигурация аяктагандан кийин, аны колдонсоңуз болот, бул ыңгайлуу [pdsh](https://github.com/chaos/pdsh)

Биринчиси, ipv6-ны кол менен иштетүү жана сыноо

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

Андан кийин бир чыкылдатуу менен системаны ubuntu 24кө жаңыртыңыз жана көп колдонулган программаны орнотуңуз.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 тилди кытай тилине жана убакыт алкагын Пекинге которууну билдирет.

Демейки боюнча орнотулган программа менин жеке колдонуу адаттарыма жараша конфигурацияланган, мисалы [mise](https://github.com/jdx/mise) [direnv](https://direnv.net/) , ж [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu)

Бул кадам көп убакытты талап кылат, жарым саат күткөндөн кийин, кеңири колдонулган программалык камсыздоону жаңыртуу жана орнотуу ийгиликтүү болот. Муну төмөнкү буйрук менен ырастасаңыз болот:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh ssh Ылдамдатат

[mosh](//mosh.org) Кытайга болгон тармак туруктуу эмес `ssh` .

Жергиликтүү `~/.bin` каталогунда `c0` скрипт түзүңүз (негизинен чөйрө өзгөрмөсүнө `PATH` кошулду) төмөнкүдөй:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

Түзүлгөндөн кийин, аткарылуучу уруксаттарды жана `c1` жана `c2` жумшак шилтемени бериңиз.

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

Келечекте `mosh` жогорку ылдамдыктагы түз байланышты колдонуу үчүн `c0` жана `c1` түз киргизип, киргизсеңиз болот [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## Куткаруу Тутумунун Чоо-Жайы

Куткаруу тутумуна кириңиз 22 Порт [tcping](https://github.com/paradiseduo/tcping) жеткиликтүү болгондон кийин, сиз ага коюлган сырсөз менен 2-3 аз күтө албайсыз.

```
tcping ip 22 -c 9999
```

Порттун жеткиликтүүлүгүн көзөмөлдөөгө болот.

### Rescue Киргенден Кийин Катуу Дискке Кантип Кирүү Керек &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Мисалы, сиз сүрөттү көрө аласыз

```
snapper -c etc list
```

Снапшот файлынын өзгөрүүлөрүн көрүү

```
snapper -c etc status 5..6
```

Белгилүү бир айырмачылык текстти көрүү

```
snapper -c etc diff 3..5
```

Сүрөттү калыбына келтирүү: Мисалы, 5-сүрөттөн кийин бардык өзгөртүүлөрдү жокко чыгарыңыз

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Журналдын Туруктуулугун Иштетүү

Куткаруу тутумунун алкагында журналдын туруктуулугун иштетүү сизден тиешелүү каталогду кол менен түзүп, бир нече кошумча кадамдарды аткарууну талап кылат, анткени сизде иштеп жаткан systemd кызматы жок болушу мүмкүн. Бул жерде сиз аракет кыла турган кадамдар:

Биринчи chroot, анан

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Ubuntu тутумуңузга кайра жүктөө: Жогорудагы өзгөртүүлөрдү киргизгенден кийин, сиз куткаруу тутумунун ордуна машинаңызды кадимкидей жана Ubuntu тутумуңузга кайра жүктөшүңүз керек.

Ошентип, systemd-journald жаңы конфигурациядан башталып, журналдарды катуу дискте туруктуу сактай баштайт.

Жогорудагы операциялар куткаруу тутумунда аткарыларын эстен чыгарбаңыз, андыктан ал өзгөртүүлөр система кайра жүктөлмөйүнчө жана куткаруу режиминде болбой калмайынча күчүнө кирбейт.

Бул сиз дароо туруктуу болуп турган учурдагы журналдарга эч кандай өзгөртүүлөрдү көрбөйсүз, бирок андан кийин түзүлгөн бардык журналдар сакталып кала берет дегенди билдирет.

Ушундай жол менен сиз ssh кызматынын журналдарын көрө аласыз

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

### Кызык Маселени Чечиңиз: Сервер ssh Кыла Албайт

Мен мурун автоматтык компиляция жана жайылтуу скриптин түзүү үчүн github аракетин колдонгом, бирок жайылтуу кызматы ssh болушу мүмкүн эмес экенин байкадым.

Куткаруу тутумуна кирүү жана журналдарды карап чыгуу - бул уруксат маселеси:

`Missing privilege separation directory: /run/sshd`

Бул каталогду төмөнкү буйрук менен түзө аласыз

```
chown root:root /
systemd-tmpfiles --create
```

Кайталанма иликтөөлөрдөн кийин, себеби github action tar `uid` колдонуучусу `1001`

`--no-same-owner` параметри жок болсо, таңгактан чыгаруу керек, ал `1001` колдонуучу үчүн `uid` чейин ачылат tar

Натыйжада, rsync түпкү каталогдун колдонуучу тобун өзгөртөт, анан кайра күйгүзгөндөн кийин ssh кире албай каласыз.

Декомпрессия сценарийин төмөнкүдөй өзгөртүңүз:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
