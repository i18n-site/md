## Პაროლის Გარეშე Შესვლის Კონფიგურაცია

რამდენიმე წუთის შემდეგ ყველა სერვერი გააქტიურდა.

უპირველეს ყოვლისა IP შეცვალეთ ადგილობრივი აპარატის `~/.ssh/config` , რომ დააყენოთ მეტსახელი და შემდეგ გამოიყენოთ `ssh-copy-id root@c2` პაროლის გარეშე შესვლის კონფიგურაციისთვის.

![](https://i-01.eu.org/1718287198.avif)

თქვენ შეგიძლიათ შეცვალოთ ეს [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) , რათა დაასრულოთ ინიციალიზაციის კონფიგურაცია უფრო სწრაფად.

პაროლის გარეშე კონფიგურაციის დასრულების შემდეგ, შეგიძლიათ [pdsh](https://github.com/chaos/pdsh) იგი სერიულად, რაც მოსახერხებელია.

პირველი არის ხელით ჩართოთ ipv6 და შეამოწმოთ

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

შემდეგ განაახლეთ სისტემა ubuntu 24-ზე ერთი დაწკაპუნებით და დააინსტალირეთ ხშირად გამოყენებული პროგრამული უზრუნველყოფა.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 ნიშნავს ენის შეცვლას ჩინურზე და დროის ზონაში პეკინზე.

ნაგულისხმევად დაინსტალირებული პროგრამული [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) კონფიგურირებულია ჩემი პირადი გამოყენების ჩვევების მიხედვით, როგორიცაა [mise](https://github.com/jdx/mise) [direnv](https://direnv.net/) და ა.შ.

ამ ნაბიჯს დიდი დრო სჭირდება ნახევარი საათის ლოდინის შემდეგ, ჩვეულებრივ გამოყენებული პროგრამული უზრუნველყოფის განახლება და ინსტალაცია წარმატებული იქნება. ამის დადასტურება შეგიძლიათ შემდეგი ბრძანებით:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh Აჩქარებს ssh

ქსელი ევროპიდან ჩინეთში არ არის სტაბილური, რეკომენდებულია `ssh` გამოყენება [mosh](//mosh.org)

შექმენით `c0` სკრიპტი ადგილობრივ დირექტორიაში `~/.bin` (ძირითადად დამატებულია გარემოს ცვლადში `PATH` ) შემდეგნაირად:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

შექმნის შემდეგ, მიანიჭეთ შესრულებადი ნებართვები და რბილი ბმული `c1` და `c2` .

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

მომავალში, შეგიძლიათ პირდაპირ შეიყვანოთ `c0` და `c1` რომ გამოიყენოთ `mosh` მაღალსიჩქარიანი პირდაპირი კავშირი და შეიყვანოთ [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## Სამაშველო Სისტემის Დეტალები

შედით სამაშველო სისტემაში მას შემდეგ 22 რაც პორტი იქნება [tcping](https://github.com/paradiseduo/tcping) წვდომა, თქვენ ვერ შეძლებთ მასზე წვდომას დაყენებული პაროლით მხოლოდ 2-3 წუთით და ყველაფერი კარგად იქნება.

```
tcping ip 22 -c 9999
```

პორტის ხელმისაწვდომობის მონიტორინგი შესაძლებელია.

### Როგორ Შეხვიდეთ Მყარ Დისკზე Სამაშველოში & Შემდეგ

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

თქვენ შეგიძლიათ ნახოთ სნეპშოტი, მაგალითად

```
snapper -c etc list
```

Snapshot ფაილის ცვლილებების ნახვა

```
snapper -c etc status 5..6
```

იხილეთ კონკრეტული განსხვავებული ტექსტი

```
snapper -c etc diff 3..5
```

სნეპშოტის აღდგენა: მაგალითად, გააუქმეთ ყველა ცვლილება სნეპშოტის 5-ის შემდეგ

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Ჟურნალის Მდგრადობის Ჩართვა

სამაშველო სისტემაში ჟურნალის მდგრადობის ჩართვა მოითხოვს, რომ ხელით შექმნათ შესაბამისი დირექტორია და შეასრულოთ რამდენიმე დამატებითი ნაბიჯი, რადგან შეიძლება არ გქონდეთ გაშვებული systemd სერვისი. აქ არის ნაბიჯები, რომლებიც შეგიძლიათ სცადოთ:

ჯერ chroot, მერე

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

გადატვირთეთ თქვენს Ubuntu სისტემაში: ზემოაღნიშნული ცვლილებების განხორციელების შემდეგ, თქვენ უნდა გადატვირთოთ თქვენი მანქანა ჩვეულებრივ და თქვენს Ubuntu სისტემაში სამაშველო სისტემის ნაცვლად.

ამ გზით, systemd-journald დაიწყებს ახალი კონფიგურაციით და დაიწყებს ჟურნალების მუდმივ შენახვას მყარ დისკზე.

გაითვალისწინეთ, რომ ზემოაღნიშნული ოპერაციები შესრულებულია სამაშველო სისტემაზე, ამიტომ ეს ცვლილებები ძალაში არ შედის, სანამ სისტემა არ გადაიტვირთება და აღარ იქნება სამაშველო რეჟიმში.

ეს ნიშნავს, რომ თქვენ დაუყოვნებლივ ვერ იხილავთ რაიმე ცვლილებას არსებულ ჟურნალებში, რომლებიც გახდება მუდმივი, მაგრამ ყველა ჟურნალი, რომელიც გენერირებულია ამის შემდეგ, შენარჩუნდება.

ამ გზით შეგიძლიათ ნახოთ ssh სერვისის ჟურნალები

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Შენიშვნები Რთულ Და Რთულ Დაავადებებზე

### Მოაგვარეთ Უცნაური Პრობლემა: Სერვერს Არ Შეუძლია ssh

ადრე გამოვიყენე github მოქმედება ავტომატური შედგენისა და განლაგების სკრიპტის შესაქმნელად, მაგრამ აღმოვაჩინე, რომ განლაგების სერვისი არ შეიძლება იყოს ssh.

სამაშველო სისტემაში შესვლა და ჟურნალების დათვალიერება ნებართვის საკითხია:

`Missing privilege separation directory: /run/sshd`

თქვენ შეგიძლიათ შექმნათ ეს დირექტორია შემდეგი ბრძანებით

```
chown root:root /
systemd-tmpfiles --create
```

განმეორებითი გამოკვლევების შემდეგ, ეს იმიტომ ხდება github action რომ tar ის მომხმარებელი `uid` არის `1001`

შეფუთვა უნდა მოხდეს, თუ არ არის პარამეტრი `--no-same-owner` , ის დეკომპრესირდება `uid` მდე `1001` მომხმარებლისთვის tar

შედეგად, rsync შეცვლის root დირექტორიას მომხმარებელთა ჯგუფს და შემდეგ გადატვირთვის შემდეგ ვერ შეძლებთ ssh-ში შესვლას.

შეცვალეთ დეკომპრესიის სკრიპტი შემდეგნაირად:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
