# تهيئة الخادم (استخدم قسم BTRFS، لقطة snapper المجدولة)

## شراء خادم من Contabo

Contabo هو مزود VPS رخيص في ألمانيا ويتمتع بتاريخ طويل (أكثر من 20 عامًا) ويعتمد عليه عدد كبير من المستخدمين (أكثر من 200,000)، لذا لا داعي للقلق بشأن الهروب.

شراء VPS من [contabo.com](//contabo.com) وتذكر اختيار التسعير باليورو، فهو أرخص.

![](https://i-01.eu.org/1718285782.avif)

يوجد رسوم تثبيت لمرة واحدة عند استخدام Contabo.
ومع ذلك، هناك أيام في كل شهر يتم فيها تقديم رسوم التثبيت مجانًا، فقط تحتاج إلى التسجيل والانتباه إلى رسائل البريد الإلكتروني الترويجية بعد التسجيل.

بالإضافة إلى رسوم المنطقة المجانية في أوروبا، هناك مناطق أخرى بها زيادات مختلفة في الأسعار.

لقد اخترت خادمًا أوروبيًا، حيث لا يهم موقع الخادم بعد استخدام Cloudflare.

التكوين النهائي هو كما يلي: 16 جيجابايت من الذاكرة، 6 وحدات معالجة مركزية، 200 جيجابايت NVMe، بسعر 9.50 يورو شهريًا، أي ما يعادل 74.43 يوان صيني.

## تكوين تسجيل الدخول بدون كلمة مرور

بعد بضع دقائق، تم تفعيل جميع الخوادم

أول شيء هو تحرير `~/.ssh/config` على الجهاز المحلي لتعيين اسم مستعار للـ IP، ثم استخدام `ssh-copy-id root@c2` لتكوين تسجيل الدخول بدون كلمة مرور.

![](https://i-01.eu.org/1718287198.avif)

يمكنك تعديل هذا البرنامج النصي [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) لإكمال تكوين التهيئة بشكل أسرع.

بعد اكتمال التكوين الخالي من كلمة المرور، يمكنك استخدام [pdsh](https://github.com/chaos/pdsh) للتحكم بالجماعات، مما يسهل العمل.

الأول هو تمكين IPv6 يدويًا وتجربته

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

ثم قم بترقية النظام إلى Ubuntu 24 بنقرة واحدة وقم بتثبيت البرامج الشائعة الاستخدام.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 يعني تبديل اللغة إلى اللغة الصينية والمنطقة الزمنية إلى بكين.

يتم تكوين البرامج المثبتة افتراضيًا وفقًا لعادات الاستخدام الشخصية، مثل [mise](https://github.com/jdx/mise) و[direnv](https://direnv.net/) وما إلى ذلك. إذا لزم الأمر، يمكنك نسخ المستودع [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) وتعديله حسب الحاجة.

تستغرق هذه الخطوة وقتًا طويلاً، وبعد الانتظار لمدة نصف ساعة، ستتم ترقية البرامج الشائعة الاستخدام وتثبيتها بنجاح. يمكنك تأكيد ذلك باستخدام الأمر التالي:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

![](https://i-01.eu.org/1718290346.avif)

## تسريع SSH باستخدام Mosh

الشبكة من أوروبا إلى الصين غير مستقرة، لذا يُنصح باستخدام [mosh](//mosh.org) لتسريع `ssh`.

قم بإنشاء برنامج نصي `c0` في الدليل المحلي `~/.bin` (يُضاف بشكل أساسي إلى متغير البيئة `PATH`) كما يلي:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

بعد الإنشاء، قم بمنح الأذونات القابلة للتنفيذ وإنشاء روابط ناعمة لـ `c1` و `c2`.

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

في المستقبل، يمكنك إدخال `c0` و `c1` مباشرةً لاستخدام `mosh` للدخول بسرعة إلى [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html).

## تحويل تنسيق القسم من ext4 إلى Btrfs

يمكن لـ [Btrfs](https://wikipedia.org/wiki/Btrfs) إجراء لقطات القرص، مما يسهل النسخ الاحتياطي.

أدخل أولاً إلى نظام الإنقاذ

![](https://i-01.eu.org/1718300448.avif)

اختر إنقاذ Debian

![](https://i-01.eu.org/1718300753.avif)

انتظر حتى يتم إعادة تشغيل الخادم

عند استخدام SSH، قد يظهر تغيير في بصمات الأصابع، استخدم الأمر `ssh-keygen -R ip` لتنظيفها

ثم قم بتشغيل البرنامج النصي التالي لتحويل القسم

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

سيقوم البرنامج النصي بإعادة تشغيل النظام تلقائيًا بعد نجاح التحويل.

### عرض نسبة ضغط Btrfs

يتم الضغط تلقائيًا بشكل افتراضي. الأمر للتحقق من نسبة الضغط هو كما يلي:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### الضغط اليدوي

إذا كنت من محبي الكمال، فيمكنك أيضًا إجراء ضغط كامل للقرص يدويًا، ويكون الأمر كما يلي:

```bash
btrfs filesystem defragment -r -czstd /
```

كما هو موضح في الشكل أدناه، يمكنك أن ترى أنه بعد الضغط اليدوي في النظام الجديد، تم ضغط ما يقرب من 100 ميجابايت إضافية.

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### إنشاء نسخة احتياطية باستخدام Snapper

[Snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html) هو أفضل شريك للنسخ الاحتياطي باستخدام Btrfs.

سيقوم البرنامج النصي للتهيئة بإنشاء الدليل `~/ubuntu`، فقط قم بتشغيل الأمر التالي:

```
~/ubuntu/snapper/init.sh
```

محتوى البرنامج النصي في [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

استخدم الأمر `snapper list-configs` لعرض المجلدات الفرعية التي تم تعيين اللقطات لها

![](https://i-01.eu.org/1718302689.avif)

استخدم الأمر `snapper -c mnt list` لعرض لقطات مجلد فرعي معين

يمكنك عرض إعدادات النسخ الاحتياطي المحددة لهذا المجلد الفرعي في `'/etc/snapper/configs/mnt'`

المزيد من المراجع:

[استخدام Snapper لتشغيل وظيفة اللقطة في Btrfs بسهولة](https://zhuanlan.zhihu.com/p/31082518)

## لقطة خادم Contabo

تقدم Contabo وظيفة لقطة الخادم، لكنها لا تدعم اللقطات المجدولة. لقد كتبت [برنامج نصي لـ GitHub Actions](https://github.com/i18n-cron/cron/actions/workflows/contabo.snapshot.yml) لالتقاط لقطات يومية.

![](https://i-01.eu.org/1718304206.avif)

لإعادة استعادة لقطة الخادم، انقر أولاً على "Crontol" في القائمة الإدارية

![](https://i-01.eu.org/1718304275.avif)

ثم انقر فوق "Snapshots" لاستعادة اللقطة.

![](https://i-01.eu.org/1718304063.avif)

ومع ذلك، فإن دقة هذه اللقطة كبيرة جدًا، وبمجرد استعادتها، سيتم فقدان البيانات المضافة حديثًا بالكامل، لذا يرجى استخدامها بحذر.

عادةً ما يكون استخدام لقطات Btrfs أكثر ملاءمة.

## تفاصيل نظام الإنقاذ

أدخل إلى نظام الإنقاذ، بعد [tcping](https://github.com/paradiseduo/tcping) على المنفذ 22، لن تتمكن من الوصول إليه باستخدام كلمة المرور المحددة لمدة 2-3 دقائق. انتظر لحظة وسيكون الأمر على ما يرام.

```
tcping ip 22 -c 9999
```

يمكنك مراقبة إمكانية الوصول إلى المنفذ.

### كيفية الوصول إلى القرص الصلب بعد الدخول إلى الإنقاذ وإنعاش اللقطة

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

يمكنك عرض اللقطة، على سبيل المثال:

```
snapper -c etc list
```

عرض تغييرات ملف اللقطة

```
snapper -c etc status 5..6
```

عرض نص الفروق المحددة

```
snapper -c etc diff 3..5
```

استعادة اللقطة: على سبيل المثال، التراجع عن جميع التغييرات بعد اللقطة 5

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### تمكين استمرارية السجل

يتطلب تمكين استمرارية السجل في نظام الإنقاذ إنشاء الدليل المقابل يدويًا وتنفيذ بعض الخطوات الإضافية، لأنه قد لا يكون لديك خدمة systemd قيد التشغيل. فيما يلي الخطوات التي يمكنك تجربتها:

قم بالجذر أولاً، ثم

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

إعادة التشغيل إلى نظام Ubuntu الخاص بك: بعد إجراء التغييرات المذكورة أعلاه، تحتاج إلى إعادة تشغيل جهازك بشكل طبيعي ودخول نظام Ubuntu الخاص بك بدلاً من نظام الإنقاذ.

بهذه الطريقة، سيبدأ systemd-journald بالتكوين الجديد ويبدأ في تخزين السجلات باستمرار على القرص الصلب.

ضع في اعتبارك أن العمليات المذكورة أعلاه يتم تنفيذها على نظام الإنقاذ، وبالتالي فإن هذه التغييرات لن تصبح سارية المفعول حتى يتم إعادة تشغيل النظام ولم يعد في وضع الإنقاذ.

وهذا يعني أنك لن ترى على الفور أي تغييرات في السجلات الموجودة والتي تصبح مستمرة، ولكن جميع السجلات التي تم إنشاؤها بعد ذلك ستستمر.

بهذه الطريقة يمكنك عرض سجلات خدمة SSH

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## ملاحظات على المشكلات الصعبة والمعقدة

### حل المشكلة الغريبة: الخادم لا يمكنه استخدام SSH

استخدمت إجراء GitHub لإنشاء برنامج نصي للتجميع والنشر التلقائي من قبل، لكنني وجدت أن خدمة النشر لا يمكن أن تكون SSH.

الدخول إلى نظام الإنقاذ والنظر في السجلات يظهر أن المشكلة تتعلق بالإذن:

`Missing privilege separation directory: /run/sshd`

يمكنك إنشاء هذا الدليل باستخدام الأمر التالي:

```
chown root:root /
systemd-tmpfiles --create
```

بعد التحقيقات المتكررة، يكون السبب هو أن المستخدم `uid` المستخدم في tar من GitHub Actions هو `1001`

إذا لم يكن هناك معلمة `--no-same-owner` عند فك ضغط tar، سيتم فك ضغطه إلى مستخدم `uid` `1001`

نتيجة لذلك، سيقوم rsync بتغيير مجموعة مستخدمي الدليل الجذر، وبعد ذلك لن تتمكن من تسجيل الدخول إلى SSH بعد إعادة التشغيل.

قم بتعديل البرنامج النصي لإلغاء الضغط كما يلي:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
