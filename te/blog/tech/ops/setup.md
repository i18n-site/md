# సర్వర్ని ప్రారంభించండి (BTRFS విభజనను ఉపయోగించండి, స్నాపర్ షెడ్యూల్ చేసిన స్నాప్షాట్)

## contabo కొనుగోలు సర్వర్

contabo అనేది సుదీర్ఘ చరిత్ర (20 సంవత్సరాల కంటే ఎక్కువ) మరియు చాలా మంది వినియోగదారులు (200,000 కంటే ఎక్కువ) కలిగిన జర్మనీలో చౌకైన VPS ప్రొవైడర్, కాబట్టి పారిపోవడం గురించి ఆందోళన చెందాల్సిన అవసరం లేదు.

కొనుగోలు VPS [contabo.com](//contabo.com) చౌకైన యూరో ధరను ఎంచుకోవాలని గుర్తుంచుకోండి.

![](https://i-01.eu.org/1718285782.avif)

కాంటాబో కోసం వన్-టైమ్ ఇన్స్టాలేషన్ ఫీజు ఉంది.
అయితే, ప్రతి నెలా కొన్ని రోజులు ఇన్స్టాలేషన్ రుసుము ఉచితం మరియు ప్రచార ఇమెయిల్లకు శ్రద్ధ వహించండి.

ఐరోపాలో ఉచిత ప్రాంతం రుసుముతో పాటు, ఇతర ప్రాంతాలలో ధరల పెరుగుదల యొక్క వివిధ స్థాయిలు ఉన్నాయి.

నేను యూరోపియన్ సర్వర్ని ఎంచుకున్నాను, క్లౌడ్ఫ్లేర్ని ఉపయోగించిన తర్వాత, సర్వర్ల మధ్య పెద్దగా తేడా లేదు.

చివరి కాన్ఫిగరేషన్ క్రింది విధంగా ఉంది, 16GB మెమరీ, 6CPU, 200 GB NVMe, నెలకు 9.50 యూరోలు, 74.43 RMBకి సమానం.

![](https://i-01.eu.org/1718286008.avif)

**ఈ ధర అలీబాబా క్లౌడ్ మరియు అమెజాన్ ధరలో దాదాపు పదో వంతు** .

![](https://i-01.eu.org/1718356956.avif)

పనితీరు మరియు పారామితులను అమలు చేయడానికి నేను [Yet-Another-Bench-Script](https://github.com/masonr/yet-another-bench-script) :

```
curl -sL yabs.sh | bash -s -- -s https://www.vpsbenchmarks.com/yabs/upload
```

మోడల్ `AMD EPYC 9224 24-Core Processor` , సింగిల్-కోర్ పనితీరు స్కోరు 1285 చూపుతోంది CPU
, మల్టీ-కోర్ స్కోర్ 4277 మరిన్ని ఫలితాలను చూడండి [vpsbenchmarks.com](https://www.vpsbenchmarks.com/yabs/contabo-6c-16gb-3fb7d8)

నేను ఒకేసారి మూడు సర్వర్లను కొనుగోలు చేసాను మరియు అదే సమయంలో వాటిని సక్రియం చేసాను, ఇది సర్వర్లు ఒకే కంప్యూటర్ గదిలో ఉండేలా చేస్తుంది, ఇది భవిష్యత్తులో అధిక లభ్యతను రూపొందించడం సులభం చేస్తుంది.

దీని సర్వర్లు అప్పుడప్పుడు హ్యాంగ్ అప్ అవుతాయి, అధిక లభ్యత ద్వారా వెబ్సైట్ ప్రాప్యతను నిర్ధారిస్తుంది.

కొత్త వినియోగదారులు కొనుగోలు చేసేటప్పుడు నిజమైన పేరు ప్రమాణీకరణ కోసం వారి ID కార్డ్లను సమర్పించాలి.

## పాస్వర్డ్ లేని లాగిన్ని కాన్ఫిగర్ చేయండి

కొన్ని నిమిషాల తర్వాత, సర్వర్లు అన్నీ యాక్టివేట్ చేయబడ్డాయి.

మారుపేరును సెట్ చేయడానికి స్థానిక యంత్రం యొక్క `~/.ssh/config` సవరించడం మొదటి IP , ఆపై పాస్వర్డ్ లేని లాగిన్ను కాన్ఫిగర్ చేయడానికి `ssh-copy-id root@c2` ఉపయోగించండి.

![](https://i-01.eu.org/1718287198.avif)

ప్రారంభ కాన్ఫిగరేషన్ను వేగంగా పూర్తి చేయడానికి మీరు ఈ [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) సవరించవచ్చు.

పాస్వర్డ్ రహిత కాన్ఫిగరేషన్ పూర్తయిన తర్వాత, మీరు దీన్ని [pdsh](https://github.com/chaos/pdsh) బ్యాచ్లలో, ఇది సౌకర్యవంతంగా ఉంటుంది.

మొదటిది ipv6 మరియు పరీక్షను మాన్యువల్గా ప్రారంభించడం

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

ఆపై ఒక క్లిక్తో సిస్టమ్ను ఉబుంటు 24కి అప్గ్రేడ్ చేయండి మరియు సాధారణంగా ఉపయోగించే సాఫ్ట్వేర్ను ఇన్స్టాల్ చేయండి.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 అంటే భాషను చైనీస్కి మరియు టైమ్ జోన్ని బీజింగ్కి మార్చడం.

[direnv](https://direnv.net/) ఇన్స్టాల్ చేయబడిన సాఫ్ట్వేర్ నా [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) వినియోగ అలవాట్లకు అనుగుణంగా కాన్ఫిగర్ చేయబడింది, [mise](https://github.com/jdx/mise)

ఈ దశ అరగంట వేచి ఉన్న తర్వాత, సాధారణంగా ఉపయోగించే సాఫ్ట్వేర్ను అప్గ్రేడ్ చేయడం మరియు ఇన్స్టాల్ చేయడం విజయవంతమవుతుంది. కింది ఆదేశంతో మీరు దీన్ని నిర్ధారించవచ్చు:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## మోష్ sshని వేగవంతం చేస్తుంది

యూరప్ నుండి చైనాకు నెట్వర్క్ స్థిరంగా లేదు `ssh` [mosh](//mosh.org)

కింది విధంగా స్థానిక డైరెక్టరీ `~/.bin` (ప్రధానంగా ఎన్విరాన్మెంట్ వేరియబుల్ `PATH` కి జోడించబడింది)లో `c0` స్క్రిప్ట్లను సృష్టించండి:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

సృష్టించిన తర్వాత, ఎక్జిక్యూటబుల్ అనుమతులు మరియు సాఫ్ట్-లింక్ `c1` మరియు `c2` మంజూరు చేయండి.

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

భవిష్యత్తులో, మీరు `mosh` హై-స్పీడ్ డైరెక్ట్ కనెక్షన్ని ఉపయోగించడానికి మరియు ఎంటర్ చేయడానికి నేరుగా `c0` మరియు `c1` నమోదు చేయవచ్చు [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## విభజన ఆకృతిని ext4 నుండి btrfsకి మార్చండి

[btrfs](https://wikipedia.org/wiki/Btrfs) సులభమైన బ్యాకప్ కోసం డిస్క్ స్నాప్షాట్లను తీసుకోవచ్చు.

ముందుగా రెస్క్యూ సిస్టమ్లోకి ప్రవేశించండి

![](https://i-01.eu.org/1718300448.avif)

డెబియన్ రెస్క్యూ ఎంచుకోండి

![](https://i-01.eu.org/1718300753.avif)

సర్వర్ పునఃప్రారంభించబడే వరకు వేచి ఉండండి

ssh వేలిముద్ర మార్పులు ఉంటాయి, దానిని శుభ్రం చేయడానికి `ssh-keygen -R ip` ఉపయోగించండి

విభజనను మార్చడానికి క్రింది స్క్రిప్ట్ను అమలు చేయండి

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

మార్పిడి విజయవంతం అయిన తర్వాత స్క్రిప్ట్ స్వయంచాలకంగా సిస్టమ్ను పునఃప్రారంభిస్తుంది.

### btrfs కుదింపు నిష్పత్తిని వీక్షించండి

ఇది డిఫాల్ట్గా స్వయంచాలకంగా కుదించబడుతుంది, కుదింపు నిష్పత్తిని తనిఖీ చేయడానికి ఈ క్రింది విధంగా ఉంటుంది

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### మాన్యువల్ కుదింపు

మీరు పర్ఫెక్షనిస్ట్ అయితే, మీరు పూర్తి డిస్క్ కంప్రెషన్ను మాన్యువల్గా కూడా చేయవచ్చు:

```bash
btrfs filesystem defragment -r -czstd /
```

దిగువ చిత్రంలో చూపిన విధంగా, కొత్త సిస్టమ్లో మాన్యువల్ కంప్రెషన్ తర్వాత, దాదాపు 100MB ఎక్కువ కంప్రెస్ చేయబడిందని మీరు చూడవచ్చు.

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### స్నాపర్తో Btrfs స్నాప్షాట్ బ్యాకప్ను సృష్టించండి

[snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html) Btrfs బ్యాకప్ కోసం ఉత్తమ భాగస్వామి.

ప్రారంభ స్క్రిప్ట్ `~/ubuntu` డైరెక్టరీని సృష్టిస్తుంది, కింది ఆదేశాన్ని అమలు చేయండి:

```
~/ubuntu/snapper/init.sh
```

స్క్రిప్ట్ కంటెంట్లో మిమ్మల్ని కలుద్దాం [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

స్నాప్షాట్లు సెట్ చేయబడిన సబ్వాల్యూమ్లను వీక్షించడానికి `snapper list-configs` ఉపయోగించండి.

![](https://i-01.eu.org/1718302689.avif)

సబ్వాల్యూమ్ యొక్క స్నాప్షాట్ను వీక్షించడానికి `snapper -c mnt list` ఉపయోగించండి

`/etc/snapper/configs/mnt` ఈ ఉపవాల్యూమ్ యొక్క నిర్దిష్ట బ్యాకప్ సెట్టింగ్లను వీక్షించగలదు

మరిన్ని సూచనలు:

[Btrfs యొక్క స్నాప్షాట్ ఫంక్షన్తో సులభంగా ఆడండి snapper](https://zhuanlan.zhihu.com/p/31082518)

## contabo సర్వర్ స్నాప్షాట్

contabo సర్వర్ స్నాప్షాట్ ఫంక్షన్ను అందిస్తుంది, కానీ షెడ్యూల్ చేసిన స్నాప్షాట్లకు [github action](https://github.com/i18n-cron/cron/actions/workflows/contabo.snapshot.yml) ఇవ్వదు.

![](https://i-01.eu.org/1718304206.avif)

సర్వర్ స్నాప్షాట్ను పునరుద్ధరించడానికి, ముందుగా మేనేజ్మెంట్ మెనులో Crontol క్లిక్ చేయండి

![](https://i-01.eu.org/1718304275.avif)

ఆపై, పునరుద్ధరించడానికి స్నాప్షాట్లను క్లిక్ చేయండి.

![](https://i-01.eu.org/1718304063.avif)

అయితే, ఈ స్నాప్షాట్ యొక్క గ్రాన్యులారిటీ చాలా పెద్దది అయిన తర్వాత, కొత్తగా జోడించిన డేటా పూర్తిగా పోతుంది, కాబట్టి దయచేసి దీన్ని జాగ్రత్తగా ఉపయోగించండి.

సాధారణంగా btrfs స్నాప్షాట్లను ఉపయోగించడం మరింత సౌకర్యవంతంగా ఉంటుంది.

## రెస్క్యూ సిస్టమ్ వివరాలు

రెస్క్యూ సిస్టమ్ను నమోదు 22 పోర్ట్ [tcping](https://github.com/paradiseduo/tcping) యాక్సెస్ చేయబడిన తర్వాత, మీరు సెట్ చేసిన పాస్వర్డ్తో 2-3 క్షణం వేచి ఉండలేరు మరియు అది బాగానే ఉంటుంది.

```
tcping ip 22 -c 9999
```

పోర్ట్ యాక్సెసిబిలిటీని పర్యవేక్షించవచ్చు.

### & స్నాప్షాట్లోకి ప్రవేశించిన తర్వాత హార్డ్ డ్రైవ్ను ఎలా యాక్సెస్ చేయాలి

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

మీరు స్నాప్షాట్ను చూడవచ్చు, ఉదాహరణకు

```
snapper -c etc list
```

స్నాప్షాట్ ఫైల్ మార్పులను వీక్షించండి

```
snapper -c etc status 5..6
```

నిర్దిష్ట తేడా వచనాన్ని వీక్షించండి

```
snapper -c etc diff 3..5
```

స్నాప్షాట్ని పునరుద్ధరించండి: ఉదాహరణకు, స్నాప్షాట్ 5 తర్వాత అన్ని మార్పులను రద్దు చేయండి

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### లాగ్ నిలకడను ప్రారంభించండి

రెస్క్యూ సిస్టమ్లో లాగ్ పెర్సిస్టెన్స్ని ఎనేబుల్ చేయడానికి మీరు సంబంధిత డైరెక్టరీని మాన్యువల్గా సృష్టించాలి మరియు కొన్ని అదనపు దశలను చేయవలసి ఉంటుంది, ఎందుకంటే మీరు సిస్టమ్డి సేవను అమలు చేయకపోవచ్చు. మీరు ప్రయత్నించగల దశలు ఇక్కడ ఉన్నాయి:

మొదటి chroot, తర్వాత

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

మీ ఉబుంటు సిస్టమ్లోకి రీబూట్ చేయండి: పై మార్పులు చేసిన తర్వాత, మీరు మీ మెషీన్ను సాధారణంగా రీబూట్ చేయాలి మరియు రెస్క్యూ సిస్టమ్కు బదులుగా మీ ఉబుంటు సిస్టమ్లోకి రీబూట్ చేయాలి.

ఈ విధంగా, systemd-journald కొత్త కాన్ఫిగరేషన్తో ప్రారంభమవుతుంది మరియు హార్డ్ డిస్క్లో లాగ్లను నిలకడగా నిల్వ చేయడం ప్రారంభిస్తుంది.

పై కార్యకలాపాలు రెస్క్యూ సిస్టమ్లో నిర్వహించబడతాయని గుర్తుంచుకోండి, కాబట్టి సిస్టమ్ రీబూట్ అయ్యే వరకు ఆ మార్పులు ప్రభావం చూపవు మరియు ఇకపై రెస్క్యూ మోడ్లో ఉండవు.

దీనర్థం మీరు ఇప్పటికే ఉన్న లాగ్లలో నిరంతరంగా మారే ఏవైనా మార్పులను వెంటనే చూడలేరు, కానీ ఆ తర్వాత రూపొందించబడిన అన్ని లాగ్లు కొనసాగించబడతాయి.

ఈ విధంగా మీరు ssh సేవ యొక్క లాగ్లను చూడవచ్చు

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## కష్టమైన మరియు సంక్లిష్ట వ్యాధులపై గమనికలు

### విచిత్రమైన సమస్యను పరిష్కరించండి: సర్వర్ ssh కాదు

నేను ఇంతకు ముందు ఆటోమేటిక్ కంపైలేషన్ మరియు డిప్లాయ్మెంట్ స్క్రిప్ట్ని రూపొందించడానికి github చర్యను ఉపయోగించాను, కానీ డిప్లాయ్మెంట్ సర్వీస్ ssh కాదని కనుగొన్నాను.

రెస్క్యూ సిస్టమ్లోకి ప్రవేశించడం మరియు లాగ్లను చూడటం అనేది అనుమతి సమస్య:

`Missing privilege separation directory: /run/sshd`

కింది ఆదేశంతో మీరు ఈ డైరెక్టరీని సృష్టించవచ్చు

```
chown root:root /
systemd-tmpfiles --create
```

పునరావృత పరిశోధనల తర్వాత, tar యొక్క github action `uid` `1001`

`--no-same-owner` పరామితి లేకుంటే అన్ప్యాకింగ్ చేయాలి, అది `1001` వినియోగదారులకు `uid` కి కుదించబడుతుంది tar

ఫలితంగా, rsync రూట్ డైరెక్టరీ యొక్క వినియోగదారు సమూహాన్ని మారుస్తుంది, ఆపై మీరు పునఃప్రారంభించిన తర్వాత sshకి లాగిన్ చేయలేరు.

డికంప్రెషన్ స్క్రిప్ట్ను ఈ క్రింది విధంగా సవరించండి:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
