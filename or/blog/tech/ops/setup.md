## ପାସୱାର୍ଡମୁକ୍ତ ଲଗଇନ୍ ବିନ୍ୟାସ କରନ୍ତୁ |

କିଛି ମିନିଟ୍ ପରେ, ସର୍ଭରଗୁଡ଼ିକ ସମସ୍ତ ସକ୍ରିୟ ହୋଇଗଲା |

ପ୍ରଥମ କଥା ହେଉଛି ସ୍ଥାନୀୟ ମେସିନ୍ ର ଏଡିଟ୍ କରିବା `~/.ssh/config` ପାଇଁ ଏକ ଛଦ୍ମନାମ ସେଟ୍ କରନ୍ତୁ IP ଏବଂ ତା’ପରେ ପାସୱାର୍ଡମୁକ୍ତ ଲଗଇନ୍ ବିନ୍ୟାସ କରିବାକୁ `ssh-copy-id root@c2` ବ୍ୟବହାର କରନ୍ତୁ |

<img alt="" src="https://i-01.eu.org/1718287198.avif">

ପ୍ରାରମ୍ଭିକରଣ ବିନ୍ୟାସକୁ ଶୀଘ୍ର ସମାପ୍ତ କରିବାକୁ ଆପଣ ଏହି [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) ପରିବର୍ତ୍ତନ କରିପାରିବେ |

ପାସୱାର୍ଡମୁକ୍ତ ବିନ୍ୟାସ ସମାପ୍ତ ହେବା ପରେ, ଆପଣ ଏହାକୁ ବ୍ୟବହାର କରିପାରିବେ [pdsh](https://github.com/chaos/pdsh) ଏହା ବ୍ୟାଚରେ ପରିଚାଳିତ ହୋଇପାରିବ, ଯାହା ସୁବିଧାଜନକ ଅଟେ |

ପ୍ରଥମଟି ହେଉଛି ipv6 କୁ ହସ୍ତକୃତ ଭାବରେ ସକ୍ଷମ କରିବା ଏବଂ ପରୀକ୍ଷା କରିବା |

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

ଏହି ପଦକ୍ଷେପଟି ଏକ ଦୀର୍ଘ ସମୟ ନେଇଥାଏ, ଅଧ ଘଣ୍ଟା ଅପେକ୍ଷା କରିବା ପରେ, ସାଧାରଣତ used ବ୍ୟବହୃତ ସଫ୍ଟୱେୟାରର ନବୀକରଣ ଏବଂ ସ୍ଥାପନ ସଫଳ ହେବ | ଆପଣ ନିମ୍ନଲିଖିତ ନିର୍ଦ୍ଦେଶ ସହିତ ଏହାକୁ ନିଶ୍ଚିତ କରିପାରିବେ:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## ମୋଶ୍ ସ୍ପିଡ୍ ଅପ୍ କରେ |

ୟୁରୋପରୁ ଚାଇନା ପର୍ଯ୍ୟନ୍ତ ନେଟୱାର୍କ ସ୍ଥିର ନୁହେଁ `ssh` ଏହାକୁ ବ୍ୟବହାର କରିବାକୁ ପରାମର୍ଶ ଦିଆଯାଇଛି [mosh](//mosh.org)

ନିମ୍ନଲିଖିତ ଭାବରେ ସ୍ଥାନୀୟ ଯନ୍ତ୍ରର ଏକ `c0` ସ୍କ୍ରିପ୍ଟ ସୃଷ୍ଟି `~/.bin` (ମୁଖ୍ୟତ the ପରିବେଶ ଭେରିଏବଲ୍ `PATH` ରେ ଯୋଗ କରାଯାଇଛି):

```bash
#! / usr / bin / env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

ସୃଷ୍ଟି ପରେ, ଏକଜେକ୍ୟୁଟେବଲ୍ ଅନୁମତି ଏବଂ ସଫ୍ଟ-ଲିଙ୍କ୍ ପ୍ରଦାନ କରନ୍ତୁ, `c2` `c1`

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

ଭବିଷ୍ୟତରେ, ଆପଣ ସିଧାସଳଖ ପ୍ରବେଶ କରିପାରିବେ `c0` `c1` ଏବଂ ଆପଣ `mosh` ହାଇ ସ୍ପିଡ୍ ସିଧାସଳଖ ସଂଯୋଗ ବ୍ୟବହାର କରିପାରିବେ ଏବଂ ପ୍ରବେଶ କରିପାରିବେ [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## Ext4 ରୁ btrfs କୁ ବିଭାଜନ ଫର୍ମାଟକୁ ରୂପାନ୍ତର କରନ୍ତୁ |

ସହଜ ବ୍ୟାକଅପ୍ ପାଇଁ ଡିସ୍କ ସ୍ନାପସଟ୍ ନେଇପାରେ [btrfs](https://wikipedia.org/wiki/Btrfs)

ପ୍ରଥମେ ଉଦ୍ଧାର ବ୍ୟବସ୍ଥାରେ ପ୍ରବେଶ କରନ୍ତୁ |

ଡେବିୟାନ୍ ରେସକ୍ୟୁ ବାଛନ୍ତୁ |

<img alt="" src="https://i-01.eu.org/1718300753.avif">

ସର୍ଭର ପୁନ rest ଆରମ୍ଭ ହେବା ପାଇଁ ଅପେକ୍ଷା କରନ୍ତୁ |

ଆଙ୍ଗୁଠି ଛାପ ପରିବର୍ତ୍ତନ ଘଟିବ ଯେତେବେଳେ ssh ଏହାକୁ ସଫା କରିବାକୁ ବ୍ୟବହାର କର `ssh-keygen -R ip`

ତାପରେ ବିଭାଜନକୁ ରୂପାନ୍ତର କରିବା ପାଇଁ ନିମ୍ନଲିଖିତ ସ୍କ୍ରିପ୍ଟକୁ ଚଲାନ୍ତୁ |

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

ରୂପାନ୍ତର ସଫଳ ହେବା ପରେ ସ୍କ୍ରିପ୍ଟ ସ୍ୱୟଂଚାଳିତ ଭାବରେ ସିଷ୍ଟମକୁ ପୁନ rest ଆରମ୍ଭ କରିବ |

### Btrfs ସଙ୍କୋଚନ ଅନୁପାତ ଦେଖନ୍ତୁ |

ଡିଫଲ୍ଟ ଭାବରେ ଏହା ସ୍ୱୟଂଚାଳିତ ଭାବରେ ସଙ୍କୋଚନ କରିବ |

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### ମାନୁଆଲ୍ ସଙ୍କୋଚନ |

ଯଦି ତୁମେ ଏକ ପରଫେକ୍ଟନିଷ୍ଟ, ତୁମେ ହସ୍ତକୃତ ଭାବରେ ଏକ ସମ୍ପୂର୍ଣ୍ଣ ଡିସ୍କ ସଙ୍କୋଚନ ମଧ୍ୟ କରିପାରିବ |

```bash
btrfs filesystem defragment -r -czstd /
```

ନିମ୍ନରେ ଥିବା ଚିତ୍ରରେ ଦେଖାଯାଇଥିବା ପରି, ଆପଣ ଦେଖିପାରିବେ ଯେ ନୂତନ ସିଷ୍ଟମରେ ମାନୁଆଲ୍ ସଙ୍କୋଚନ ପରେ ପ୍ରାୟ 100MB ଅଧିକ ସଙ୍କୋଚିତ ହୋଇଛି |

### ସ୍ନାପର୍ ସହିତ ଏକ Btrfs ସ୍ନାପସଟ୍ ବ୍ୟାକଅପ୍ ସୃଷ୍ଟି କରନ୍ତୁ |

Btrfs ବ୍ୟାକଅପ୍ ପାଇଁ ସର୍ବୋତ୍ତମ ଅଂଶୀଦାର ଅଟେ [snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html)

ପ୍ରାରମ୍ଭିକରଣ ସ୍କ୍ରିପ୍ଟ `~/ubuntu` ସୃଷ୍ଟି କରିବ, କେବଳ ନିମ୍ନ ନିର୍ଦ୍ଦେଶକୁ ଚଲାନ୍ତୁ:

```
~/ubuntu/snapper/init.sh
```

ସ୍କ୍ରିପ୍ଟ ବିଷୟବସ୍ତୁରେ ଆପଣଙ୍କୁ ଦେଖ [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

ଆପଣ ଏହି ଉପଭୋଲ୍ୟୁମର ନିର୍ଦ୍ଦିଷ୍ଟ ବ୍ୟାକଅପ୍ ସେଟିଂସମୂହ ଦେଖିପାରିବେ `/etc/snapper/configs/mnt`

## ଉଦ୍ଧାର କାର୍ଯ୍ୟ ବିବରଣୀ

ଉଦ୍ଧାର ପ୍ରଣାଳୀ ପ୍ରବେଶ କରନ୍ତୁ 22 ପୋର୍ଟଟି ଉପଲବ୍ଧ ହେବା ପରେ, ଆପଣ ଏହାକୁ 2-3 ମିନିଟ୍ ପାଇଁ ସେଟ୍ ପାସୱାର୍ଡ ସହିତ ପ୍ରବେଶ କରିବାକୁ ସମର୍ଥ ହେବେ ନାହିଁ ଏବଂ ଏହା ଠିକ୍ ହେବ [tcping](https://github.com/paradiseduo/tcping)

```
tcping ip 22 -c 9999
```

ପୋର୍ଟ ଆକ୍ସେସିବିଲିଟି ଉପରେ ନଜର ରଖାଯାଇପାରିବ |

### ଉଦ୍ଧାର କାର୍ଯ୍ୟରେ ପ୍ରବେଶ କରିବା ପରେ କିପରି ହାର୍ଡ ଡ୍ରାଇଭକୁ ପ୍ରବେଶ କରିବେ &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

ଉଦାହରଣ ସ୍ୱରୂପ, ଆପଣ ସ୍ନାପସଟ୍ ଦେଖିପାରିବେ |

```
snapper -c etc list
```

ସ୍ନାପସଟ୍ ଫାଇଲ୍ ପରିବର୍ତ୍ତନଗୁଡିକ ଦେଖନ୍ତୁ |

```
snapper -c etc status 5..6
```

ନିର୍ଦ୍ଦିଷ୍ଟ ଡିଫ୍ ଟେକ୍ସଟ୍ ଦେଖନ୍ତୁ |

```
snapper -c etc diff 3..5
```

ସ୍ନାପସଟ୍ ପୁନରୁଦ୍ଧାର କରନ୍ତୁ: ଉଦାହରଣ ସ୍ୱରୂପ, ସ୍ନାପସଟ୍ 5 ପରେ ସମସ୍ତ ପରିବର୍ତ୍ତନକୁ ପୂର୍ବବତ୍ କରନ୍ତୁ |

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### ଲଗ ସ୍ଥିରତା ସକ୍ଷମ କରନ୍ତୁ |

ଏକ ଉଦ୍ଧାର ପ୍ରଣାଳୀ ଅଧୀନରେ ଲଗ ସ୍ଥିରତା ସକ୍ଷମ କରିବା ଆପଣଙ୍କୁ ସଂପୃକ୍ତ ଡିରେକ୍ଟୋରୀକୁ ମାନୁଆଲୀ ସୃଷ୍ଟି କରିବା ଏବଂ କିଛି ଅତିରିକ୍ତ ପଦକ୍ଷେପ କରିବା ଆବଶ୍ୟକ କରେ, କାରଣ ଆପଣଙ୍କର ଏକ ଚାଲୁଥିବା systemd ସେବା ନ ଥାଇପାରେ | ଏଠାରେ ଆପଣ ପଦକ୍ଷେପଗୁଡିକ ଚେଷ୍ଟା କରିପାରିବେ:

ପ୍ରଥମେ କ୍ରୁଟ୍, ତାପରେ |

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

ତୁମର ଉବୁଣ୍ଟୁ ସିଷ୍ଟମରେ ପୁନ o ବୁଟ କର: ଉପରୋକ୍ତ ପରିବର୍ତ୍ତନ କରିବା ପରେ, ତୁମ ମେସିନକୁ ସାଧାରଣ ଭାବରେ ଏବଂ ଉଦ୍ଧାର ବ୍ୟବସ୍ଥା ପରିବର୍ତ୍ତେ ତୁମର ଉବୁଣ୍ଟୁ ସିଷ୍ଟମରେ ପୁନ bo ବୁଟ କରିବା ଆବଶ୍ୟକ |

ଏହିପରି, systemd-journald ନୂତନ ସଂରଚନା ସହିତ ଆରମ୍ଭ ହେବ ଏବଂ ହାର୍ଡ ଡିସ୍କରେ ଲଗାତାର ଲଗ ସଂରକ୍ଷଣ କରିବା ଆରମ୍ଭ କରିବ |

ମନେରଖନ୍ତୁ ଯେ ଉଦ୍ଧାର କାର୍ଯ୍ୟରେ ଉପରୋକ୍ତ କାର୍ଯ୍ୟଗୁଡ଼ିକ ସଂପାଦିତ ହୋଇଛି, ତେଣୁ ସେହି ପରିବର୍ତ୍ତନଗୁଡ଼ିକ କେବଳ ସିଷ୍ଟମ ପୁନ bo ବୁଟ ହେବା ପରେ କାର୍ଯ୍ୟକାରୀ ହେବ ଏବଂ ଉଦ୍ଧାର ମୋଡରେ ନାହିଁ |

ଏହାର ଅର୍ଥ ହେଉଛି ତୁମେ ତୁରନ୍ତ ବିଦ୍ୟମାନ ଲଗ୍ଗୁଡ଼ିକରେ କ changes ଣସି ପରିବର୍ତ୍ତନ ସ୍ଥାୟୀ ଦେଖାଯିବ ନାହିଁ, କିନ୍ତୁ ଏହା ପରେ ସୃଷ୍ଟି ହୋଇଥିବା ସମସ୍ତ ଲଗଗୁଡ଼ିକ ସ୍ଥାୟୀ ରହିବ |

ଏହି ଉପାୟରେ ଆପଣ ssh ସର୍ଭିସର ଲଗ୍ ଦେଖିପାରିବେ |

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## କଠିନ ଏବଂ ଜଟିଳ ରୋଗ ଉପରେ ଟିପ୍ପଣୀ |

### ଅଦ୍ଭୁତ ସମସ୍ୟାର ସମାଧାନ କରନ୍ତୁ: ସର୍ଭର ssh କରିପାରିବ ନାହିଁ |

ମୁଁ ପୂର୍ବରୁ ଏକ ସ୍ୱୟଂଚାଳିତ ସଂକଳନ ଏବଂ ନିୟୋଜନ ସ୍କ୍ରିପ୍ଟ ତିଆରି କରିବା ପାଇଁ ଗିଥବ୍ ଆକ୍ସନ୍ ବ୍ୟବହାର କରିଥିଲି, କିନ୍ତୁ ପାଇଲି ଯେ ନିୟୋଜନ ସେବା ssh ହୋଇପାରିବ ନାହିଁ |

ଉଦ୍ଧାର ବ୍ୟବସ୍ଥାରେ ପ୍ରବେଶ କରିବା ଏବଂ ଲଗଗୁଡ଼ିକୁ ଦେଖିବା ଏକ ଅନୁମତି ପ୍ରସଙ୍ଗ:

`Missing privilege separation directory: /run/sshd`

ଆପଣ ନିମ୍ନଲିଖିତ ନିର୍ଦ୍ଦେଶ ସହିତ ଏହି ଡିରେକ୍ଟୋରୀକୁ ସୃଷ୍ଟି କରିପାରିବେ |

```
chown root:root /
systemd-tmpfiles --create
```

ବାରମ୍ବାର ଅନୁସନ୍ଧାନ ପରେ, ଏହାର କାରଣ ହେଉଛି tar ଅପଲୋଡ୍ କରିଥିବା ଉପଭୋକ୍ତା `1001` `uid` github action

ପାରାମିଟର ବିନା ଅନ୍-ପ୍ୟାକ୍ କରିବା ଜରୁରୀ tar `--no-same-owner` ଏହା `1001` ବ୍ୟବହାରକାରୀଙ୍କ ପାଇଁ `uid` ରେ ସଙ୍କୋଚିତ ହେବ |

ଫଳସ୍ୱରୂପ, rsync ମୂଳ ଡିରେକ୍ଟୋରୀର ଉପଭୋକ୍ତା ଗୋଷ୍ଠୀକୁ ପରିବର୍ତ୍ତନ କରିବ, ଏବଂ ତାପରେ ଆପଣ ପୁନ rest ଆରମ୍ଭ କରିବା ପରେ ssh କୁ ଲଗ୍ ଇନ୍ କରିବାକୁ ସମର୍ଥ ହେବେ ନାହିଁ |

ନିମ୍ନଲିଖିତ ଭାବରେ ଡିକୋମ୍ରେସନ୍ ସ୍କ୍ରିପ୍ଟକୁ ପରିବର୍ତ୍ତନ କରନ୍ତୁ:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
