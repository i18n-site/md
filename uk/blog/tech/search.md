## Огляд безсерверних рішень для повнотекстового пошуку

Для невеликих веб-сайтів, таких як документи/особисті блоги, які є суто статичними, безсумнівно, надто важко створювати серверну систему повнотекстового пошуку самостійно, і повнотекстовий пошук без служб, безсумнівно, є кращою вагою.

Існуючі безсерверні рішення для повнотекстового пошуку можна розділити на дві великі категорії.

Перша категорія включає сторонніх постачальників пошукових послуг, таких як [algolia.com](//algolia.com), які пропонують компоненти для фронтенд пошуку.

Такі послуги зазвичай платні та недоступні для користувачів з материкового Китаю через проблеми з відповідністю веб-сайту.

Вони не можуть використовуватися офлайн або в інтрамережі, що обмежує їх використання. Ми не будемо детально обговорювати ці рішення.

Друга категорія — це чистий фронтенд пошук.

До більш [lunrjs](/0) інтерфейсних повнотекстових пошуків відноситься [ ElasticLunr.js ] [https://github.com/weixsong/elasticlunr.js](%E5%9F%BA%E4%BA%8E%60lunrjs%60%E4%BA%8C%E6%AC%A1%E5%BC%80%E5%8F%91) .

`lunrjs` має два методи побудови індексів, але обидва мають свої недоліки.

1. Попередньо створені файли індексу

   Оскільки індекс містить всі слова з документів, він займає багато місця.
   Кожен раз, коли документ змінюється або додається, потрібно завантажувати новий файл індексу.
   Це збільшує час очікування користувачів і споживає багато трафіку.

2. Завантаження документів та побудова індексів у режимі реального часу

   Побудова індексу є обчислювально інтенсивним завданням, і кожен раз, коли користувач звертається до нього, це може призвести до затримок і погіршення користувацького досвіду.

Крім `lunrjs`, існують інші рішення для повнотекстового пошуку, наприклад:

[fusejs](https://www.fusejs.io), який обчислює подібність між рядками для пошуку.

Це рішення має низьку продуктивність і не підходить для повнотекстового пошуку (див. [Fuse.js Довгий запит займає більше 10 секунд, як це оптимізувати?](https://stackoverflow.com/questions/70984437/fuse-js-takes-10-seconds-with-semi-long-queries)).

[TinySearch](https://github.com/tinysearch/tinysearch), який використовує фільтр Bloom для пошуку і не підтримує пошук за префіксом (наприклад, введення `goo` не знайде `good` або `google`), а також не підтримує автоматичне завершення.

Через незадоволеність існуючими рішеннями, `i18n.site` розробив нове рішення для чистого фронтенд пошуку з наступними особливостями:

1. Підтримує багатомовний пошук і розмір ядра пошуку `6.9KB` `gzip` для порівняння, розмір `lunrjs` дорівнює `25KB` ).
1. Побудова інвертованого індексу на основі `IndexedDB`, що забезпечує низьке використання пам'яті та високу швидкість
1. Коли документи додаються/змінюються, повторно індексуються лише додані чи змінені документи, що зменшує обсяг розрахунків
1. Підтримка пошуку за префіксом, можливість відображення результатів пошуку в режимі реального часу під час введення користувачем
1. Доступність офлайн

Детальніше про реалізацію `i18n.site` ми розповімо нижче.

## Багатомовна сегментація слів

Сегментація слів виконується за допомогою вбудованого інтерфейсу браузера `Intl.Segmenter`, який підтримується основними браузерами.

![](https://p.3ti.site/1727667759.avif)

Приклад коду на `coffeescript` для сегментації слів:

```coffee
SEG = new Intl.Segmenter 0, granularity: "word"

seg = (txt) =>
  r = []
  for {segment} from SEG.segment(txt)
    for i from segment.split('.')
      i = i.trim()
      if i and !'|`'.includes(i) and !/\p{P}/u.test(i)
        r.push i
  r

export default seg

export segqy = (q) =>
  seg q.toLocaleLowerCase()
```

де:

* `/\p{P}/` — регулярний вираз для знаків пунктуації, включаючи: `! " # $ % & ' ( ) * + , - . / : ; < = > ? @ [ \ ] ^ _ ` { | } ~. `.</p><ul><li> `split('.')` тому, що `Firefox` сегментація слова браузера не сегментує `.` .</li>


## Побудова Покажчика

5 таблиць зберігання об'єктів було створено в `IndexedDB` :

* У `IndexedDB` створено п'ять таблиць об'єктів:
* `word`: id - слово
* `doc`: id - url - версія документа
* `docWord`: id документа - масив id слів
* `prefix`: префікс - масив id слів

Передайте масив документа `url` і номер версії `ver` і знайдіть, чи існує документ у таблиці `doc` Якщо він не існує, створіть інвертований індекс. Водночас видаліть перевернутий індекс для тих документів, які не були передані.

Таким чином можна досягти поступової індексації та зменшити обсяг розрахунку.

### IndexedDB З Високим Рівнем Одночасного Запису

Проект [idb](https://www.npmjs.com/package/idb) на основі асинхронної інкапсуляції IndexedDB

Читання та запис IndexedDB є асинхронними. Під час створення індексу документи завантажуватимуться одночасно для створення індексу.

Читання та запис у `IndexedDB` є асинхронними. Під час створення індексу документи завантажуються одночасно. Щоб уникнути втрати даних через конкурентний запис, можна використовувати кешування між читанням та записом, як показано у наступному прикладі на `coffeescript`:

```coffee
pusher = =>
  ing = new Map()
  (table, id, val)=>
    id_set = ing.get(id)
    if id_set
      id_set.add val
      return

    id_set = new Set([val])
    ing.set id, id_set
    pre = await table.get(id)
    li = pre?.li or []

    loop
      to_add = [...id_set]
      li.push(...to_add)
      await table.put({id,li})
      for i from to_add
        id_set.delete i
      if not id_set.size
        ing.delete id
        break
    return

rindexPush = pusher()
prefixPush = pusher()
```

## Пошук за префіксом у режимі реального часу

Для відображення результатів пошуку під час введення користувачем, наприклад, коли введено `wor`, відображаються слова з префіксом `wor`, такі як `words` та `work`.

![](https://p.3ti.site/1727684944.avif)

Ядро пошуку використовує таблицю `prefix` для останнього слова після сегментації, щоб знайти всі слова з таким префіксом і здійснювати пошук послідовно.

Функція анти-тремтіння `debounce` також використовується у зовнішній взаємодії (реалізована таким чином), щоб зменшити частоту введення користувача, що запускає пошук, і зменшити кількість обчислень.

```js
export default (wait, func) => {
  var timeout;
  return function(...args) {
    clearTimeout(timeout);
    timeout = setTimeout(func.bind(this, ...args), wait);
  };
}
```

## Точність та повнота результатів

Пошук спочатку сегментує ключові слова, введені користувачем.

Припустимо, що після сегментації отримано `N` слів. Під час повернення результатів спочатку відображаються результати, що містять усі ключові слова, потім результати, що містять `N-1`, `N-2`, ..., `1` ключові слова.

Перші результати забезпечують високу точність запиту, а результати, завантажені пізніше (після натискання кнопки "Завантажити більше"), забезпечують повноту результатів.

![](https://p.3ti.site/1727684564.avif)

## Навантаження на вимогу

Для покращення швидкості відповіді пошук використовує генератор `yield` для навантаження результатів на вимогу, повертаючи результати після кожного `limit` запиту.

Зауважте, що після кожного `yield` потрібно знову відкривати транзакцію запиту `IndexedDB`.

## Доступність офлайн

Індексна таблиця не зберігає оригінальний текст, лише слова, що зменшує обсяг пам'яті.

Виділення результатів пошуку вимагає перезавантаження вихідного тексту, але за допомогою `service worker` можна уникнути повторних мережевих запитів.

Оскільки `service worker` кешує всі статті, після першого пошуку весь сайт, включаючи функцію пошуку, стає доступним офлайн.

## Оптимізація відображення документів Markdown

Рішення чистого зовнішнього пошуку `i18n.site` оптимізовано для документів `MarkDown` .

Під час відображення результатів пошуку відображається назва розділу, і після клацання користувач переходить до відповідного розділу.

## Підсумок

Чистий фронтенд пошук по повному тексту забезпечує швидку відповідь і не вимагає сервера.

Це рішення ідеально підходить для невеликих та середніх сайтів, таких як документи та особисті блоги.
