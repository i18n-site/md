## Огляд безсервісних рішень повнотекстового пошуку

Для суто статичних веб-сайтів невеликого та середнього розміру, таких як документи/особисті блоги, створення власноруч створеної системи повнотекстового пошуку є надто важким, і повнотекстовий пошук без сервісів є більш поширеним вибором.

Бессервісні рішення для повнотекстового пошуку розділяються на дві основні категорії:

По-перше, це сторонні постачальники послуг пошуку, подібні [algolia.com](//algolia.com), які надають інтерфейсні компоненти для повнотекстового пошуку.

Такі послуги вимагають оплати за обсяг пошуку та часто недоступні для користувачів у материковому Китаї через проблеми з відповідністю веб-сайту.

Вони не можуть бути використані офлайн, не можуть бути використані в інтрамережі та мають значні обмеження. Про це буде сказано далі.

Друга категорія — це чистий передовий повнотекстовий пошук.

На даний момент звичайний повнотекстовий пошук включає [lunrjs](/0) [ElasticLunr.js](/1) на основі `lunrjs` вторинної розробки).

`lunrjs` має два способи створення індексів, і у кожного є свої недоліки.

1. Попередньо створені файли індексу

   Оскільки індекс містить всі слова з усіх документів, він є досить великий за розміром.
   Кожен раз, коли додається або змінюється документ, необхідно завантажити новий файл індексу.
   Це може збільшити час очікування користувача та спожити велику пропускну здатність.

2. Завантаження документів та створення індексів в режимі реального часу

   Створення індексу є інтенсивним обчислювальним завданням. Перебудова індексу при кожному доступі може призвести до значних затримок та поганого користувацького досвіду.

---

Крім `lunrjs`, існують інші рішення для повнотекстового пошуку, наприклад:

[fusejs](//www.fusejs.io), що обчислює подібність між рядками для пошуку.

Це рішення має надзвичайно низьку продуктивність і не підходить для повнотекстового пошуку (див. [Fuse.js Довгий запит займає більше 10 секунд, як це оптимізувати?](//stackoverflow.com/questions/70984437/fuse-js-takes-10-seconds-with-semi-long-queries)).

[TinySearch](//github.com/tinysearch/tinysearch), що використовує фільтр Bloom для пошуку, не підтримує пошук за префіксом (наприклад, введіть `goo`, шукайте `good`, `google`) та не може забезпечити ефект автоматичного завершення.

Через недоліки існуючих рішень, `i18n.site` розробив нову систему повнотекстового пошуку, яка має наступні особливості:

1. Підтримує багатомовний пошук і розмір ядра пошуку `6.9KB` `gzip` для порівняння, розмір `lunrjs` дорівнює `25KB` ).
1. Створення інвертованого індексу на основі `IndexedDB`, що зменшує використання пам’яті та забезпечує швидкість
1. Коли документи додаються/змінюються, повторно індексуються лише додані чи змінені документи, що зменшує обсяг розрахунків
1. Підтримка пошуку за префіксом, що дозволяє відображати результати пошуку в режимі реального часу під час введення тексту користувача
1. Можливість використання офлайн

Нижче детально розглянемо технічні деталі реалізації повнотекстового пошуку на `i18n.site`.

## Багатомовна сегментація слів

Сегментація слів виконується за допомогою вбудованої функції браузера `Intl.Segmenter`, яка підтримується всіма основними браузерами.

![](//p.3ti.site/1727667759.avif)

Приклад коду сегментації на `coffeescript` виглядає так:

```coffee
SEG = new Intl.Segmenter 0, granularity: "word"

seg = (txt) =>
  r = []
  for {segment} from SEG.segment(txt)
    for i from segment.split('.')
      i = i.trim()
      if i and !'|`'.includes(i) and !/\p{P}/u.test(i)
        r.push i
  r

export default seg

export segqy = (q) =>
  seg q.toLocaleLowerCase()
```

В:

* `/\p{P}/` — це регулярний вираз, який відповідає знакам пунктуації. Конкретні відповідні символи включають: `! " # $ % & ' ( ) * + , - . / : ; < = > ? @ [ \ ] ^ _ ` { | } ~. `.</p><ul><li> `split('.')` тому, що `Firefox` сегментація слова браузера не сегментує `.` .</li>


## Побудова індексу

В `IndexedDB` створено п’ять таблиць для зберігання об’єктів:

* `word` : id - слів
* `doc` : id - url - Номер версії документа
* `docWord` : Масив документа id - слово id
* `prefix` : Масив префікса - слово id
* `rindex` : Word id - Документ id : Масив номерів рядків

Передайте масив документа `url` і номер версії `ver`, знайдіть, чи існує документ у таблиці `doc`. Якщо він не існує, створіть інвертований індекс. Водночас видаліть перевернутий індекс для тих документів, які не були передані.

Таким чином можна досягти поступової індексації та зменшити обсяг обчислень.

### Підтримка високого рівня паралельного запису в IndexedDB

Проект розроблений на основі асинхронного обгортання IndexedDB [idb](//www.npmjs.com/package/idb).

Читання та запис в IndexedDB є асинхронними. Під час створення індексу документи завантажуватимуться паралельно для створення індексу.

Щоб уникнути часткової втрати даних через конкурентний запис, можна скористатися кодом `coffeescript` нижче та додати кеш `ing` між читанням і записом для перехоплення конкурентних записів.

```coffee
pusher = =>
  ing = new Map()
  (table, id, val)=>
    id_set = ing.get(id)
    if id_set
      id_set.add val
      return

    id_set = new Set([val])
    ing.set id, id_set
    pre = await table.get(id)
    li = pre?.li or []

    loop
      to_add = [...id_set]
      li.push(...to_add)
      await table.put({id,li})
      for i from to_add
        id_set.delete i
      if not id_set.size
        ing.delete id
        break
    return

rindexPush = pusher()
prefixPush = pusher()
```

## Пошук префіксів у режимі реального часу

Для відображення результатів пошуку під час введення користувачем, наприклад, коли введено `wor`, відображаються слова з префіксом `wor`, наприклад `words` і `work`.

![](//p.3ti.site/1727684944.avif)

Ядро пошуку використовуватиме таблицю `prefix` для останнього слова після сегментації слова, щоб знайти всі слова з префіксом і здійснювати послідовний пошук.

Функція анти-тремтіння `debounce` також використовується у зовнішній взаємодії (реалізована таким чином), щоб зменшити частоту введення користувача, що запускає пошук, і зменшити кількість обчислень.

```js
export default (wait, func) => {
  var timeout;
  return function(...args) {
    clearTimeout(timeout);
    timeout = setTimeout(func.bind(this, ...args), wait);
  };
}
```

## Точність та повнота пошуку

Пошук спочатку сегментує ключові слова, введені користувачем.

Припустимо, що після сегментації слів `N` слів. Під час повернення результатів спочатку будуть повернуті результати, що містять усі ключові слова, а потім результати, що містять `N-1`, `N-2`, ..., `1` ключових слів.

Результати пошуку, які відображаються спочатку, забезпечують точність запиту, а результати, завантажені пізніше (натисніть кнопку «Завантажити більше»), забезпечують повноту запиту.

![](//p.3ti.site/1727684564.avif)

## Навантаження на вимогу

Щоб покращити швидкість відповіді, пошук використовує генератор `yield` для реалізації навантаження на вимогу та повертає `limit`, коли запитується результат.

Зауважте, що кожного разу, коли ви шукаєте знову після `yield`, вам потрібно знову відкрити транзакцію запиту `IndexedDB`.

## Можливість використання офлайн

Індексна таблиця не зберігає оригінальний текст, лише слова, що зменшує обсяг пам’яті.

Виділення результатів пошуку вимагає перезавантаження вихідного тексту, а відповідність `service worker` може уникнути повторних мережевих запитів.

У той же час, оскільки `service worker` кешує всі статті, коли користувач виконує пошук, весь веб-сайт, включаючи пошук, доступний офлайн.

## Оптимізація відображення документів MarkDown

Рішення повнотекстового пошуку від `i18n.site` оптимізовано для роботи з документами у форматі `Markdown`.

Під час відображення результатів пошуку буде відображатися назва розділу, а після клацання буде здійснюватися навігація по розділу.

## Висновок

Чистий передовий повнотекстовий пошук, реалізований виключно на інтерфейсі, не потребує сервера. Він ідеально підходить для невеликих та середніх веб-сайтів, таких як документи та особисті блоги.

`i18n.site` розробив власний інтерфейсний пошук з відкритим кодом, малий розмір і швидка відповідь, що усуває недоліки поточного повнотекстового пошуку в інтерфейсі та забезпечує кращий користувацький досвід.
