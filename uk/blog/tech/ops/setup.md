## Купівля сервера в Contabo

Contabo — це німецький постачальник недорогих VPS з багаторічною історією (більше 20 років) та великою кількістю користувачів (більше 200 000). Не потрібно хвилюватися про те, що вони зникнуть.

При купівлі VPS на [contabo.com](//contabo.com) пам'ятайте вибрати ціну в євро, це буде дешевше.

![](https://i-01.eu.org/1718285782.avif)

Contabo має одноразову плату за встановлення.
Проте, щомісяця є кілька днів, коли встановлення є безкоштовним. Після реєстрації слідкуйте за промо-листами.

Крім безкоштовної зони в Європі, в інших регіонах існують різні рівні надбавок.

Я вибрав європейський сервер, оскільки після підключення до Cloudflare розташування сервера не має великого значення.

Остаточна конфігурація: 16 ГБ оперативної пам'яті, 6 CPU, 200 ГБ NVMe, 9,50 євро на місяць, що еквівалентно 74,43 юанів.

## Налаштування безпарольного входу

Через кілька хвилин всі сервери будуть активовані

Перш за все, відредагуйте файл `~/.ssh/config` на вашій локальній машині, щоб встановити псевдонім для IP-адреси, а потім використовуйте команду `ssh-copy-id root@c2` для налаштування безпарольного входу.

![](https://i-01.eu.org/1718287198.avif)

Ви можете змінювати цей сценарій [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) для швидшого завершення налаштування ініціалізації.

Після налаштування безпарольного входу ви можете використовувати [pdsh](https://github.com/chaos/pdsh) для масових операцій, що дуже зручно.

Спочатку вручну увімкніть IPv6 та перевірте його роботу

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

Потім оновіть систему до Ubuntu 24 одним клацанням миші та встановіть необхідне програмне забезпечення.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 означає перемикання мови на китайську та встановлення часового поясу на Пекін.

Програмне забезпечення, встановлене за замовчуванням, налаштоване відповідно до моїх особистих звичок користування, наприклад [mise](https://github.com/jdx/mise), [direnv](https://direnv.net/) тощо. За потреби ви можете скопіювати репозиторій [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) та внести необхідні зміни.

Цей крок займає тривалий час. Після очікування протягом півгодини оновлення та встановлення програмного забезпечення, яке часто використовується, буде успішним. Ви можете підтвердити це за допомогою такої команди:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh для прискорення ssh

Мережа з Європи до Китаю не є стабільною. Рекомендується використовувати для прискорення `ssh` інструмент [mosh](//mosh.org).

Створіть скрипт `c0` у каталозі `~/.bin` (додайте цей каталог до змінної середовища `PATH`) наступним чином:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

Після створення надайте дозволи на виконання файлу та створіть символічні посилання `c1` та `c2`.

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

Після цього ви можете безпосередньо вводити `c0` або `c1`, щоб використовувати `mosh` для швидкого підключення та входу до [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html).

## Перетворення розділу з ext4 на btrfs

[btrfs](https://wikipedia.org/wiki/Btrfs) дозволяє створювати знімки диска для зручного резервного копіювання.

Спочатку увійдіть до системи порятунку

![](https://i-01.eu.org/1718300448.avif)

Виберіть Debian Rescue

![](https://i-01.eu.org/1718300753.avif)

Зачекайте, поки сервер перезавантажиться

Під час підключення ssh можуть змінюватися відбитки пальців. Використовуйте команду `ssh-keygen -R ip`, щоб очистити їх

Потім запустіть наступний скрипт для перетворення розділу

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

Скрипт автоматично перезапустить систему після успішного перетворення.

### Перегляд ступеня стиснення btrfs

За замовчуванням буде автоматично стиснуто. Команда для перегляду ступеня стиснення:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### Ручне стиснення

Якщо ви перфекціоніст, ви можете виконати ручне повне стиснення диска:

```bash
btrfs filesystem defragment -r -czstd /
```

Як показано на малюнку нижче, після ручного стиснення в новій системі було стиснуто ще майже 100 МБ.

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### Створення резервної копії знімка Btrfs за допомогою snapper

[snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html) є ідеальним партнером для резервного копіювання знімків Btrfs.

Ініціалізаційний скрипт створить каталог `~/ubuntu`. Виконайте наступну команду:

```
~/ubuntu/snapper/init.sh
```

Зміст скрипту дивіться на [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

Використовуйте команду `snapper list-configs`, щоб переглянути підтоми, для яких встановлено знімки

![](https://i-01.eu.org/1718302689.avif)

Використовуйте команду `snapper -c mnt list`, щоб переглянути знімки певного підтому

Файл `/etc/snapper/configs/mnt` містить конкретні параметри резервного копіювання цього підтому

Додаткові посилання:

[Легко користуватися функцією знімків Btrfs за допомогою snapper](https://zhuanlan.zhihu.com/p/31082518)

Contabo надає функцію знімків сервера, але не підтримує планування знімків. Я написав [github action](https://github.com/i18n-cron/cron/actions/workflows/contabo.snapshot.yml) для створення щоденних знімків.

![](https://i-01.eu.org/1718304206.avif)

Щоб відновити знімок сервера, спочатку перейдіть до меню керування та натисніть Crontol

![](https://i-01.eu.org/1718304275.avif)

Потім натисніть Snapshots, щоб відновити.

![](https://i-01.eu.org/1718304063.avif)

Однак цей знімок має велику деталізацію. Після відновлення всі нові дані будуть втрачені, тому використовуйте цю функцію з обережністю.

Зазвичай зручніше використовувати знімки btrfs.

## Деталі системи порятунку

Після входу до системи порятунку на порту 22 можна [tcping](https://github.com/paradiseduo/tcping). Після цього протягом 2-3 хвилин ви не зможете отримати доступ до системи з встановленим паролем. Зачекайте трохи.

```
tcping ip 22 -c 9999
```

Можна контролювати доступність портів.

### Як отримати доступ до жорсткого диска та відновити знімок після входу до системи порятунку

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Ви можете переглянути знімки, наприклад:

```
snapper -c etc list
```

Переглянути зміни файлів у знімку

```
snapper -c etc status 5..6
```

Переглянути конкретний текст відмінностей

```
snapper -c etc diff 3..5
```

Відновити знімок: наприклад, скасувати всі зміни після знімка 5

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Увімкнення збереження журналів

У системі порятунку для увімкнення збереження журналів потрібно вручну створити відповідний каталог та виконати додаткові кроки, оскільки у вас може не бути запущеної служби systemd. Ось кроки, які ви можете спробувати:

Спочатку виконайте chroot, а потім:

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Перезавантажте систему Ubuntu: після внесення змін вам потрібно перезавантажити комп'ютер у звичайному режимі та увійти до системи Ubuntu замість системи порятунку.

Таким чином, systemd-journald запуститься з новою конфігурацією та почне постійно зберігати журнали на жорсткому диску.

Майте на увазі, що наведені вище операції виконуються в системі порятунку, тому ці зміни не набудуть чинності, доки система не перезавантажиться та не перейде в режим відновлення.

Це означає, що ви не відразу побачите зміни в існуючих журналах, які стають постійними, але всі журнали, створені після цього, будуть збережені.

Таким чином ви можете переглянути журнали служби ssh

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Замітки про важкі та ускладнені проблеми

### Рішення дивної проблеми: сервер не може використовувати ssh

Раніше я використовував дію github для автоматичної компіляції та сценарію розгортання, але виявив, що після розгортання служба не може бути підключена через ssh.

Після входу до системи порятунку та перегляду журналів виявлено проблему з дозволами:

`Missing privilege separation directory: /run/sshd`

Ви можете створити цей каталог за допомогою такої команди:

```
chown root:root /
systemd-tmpfiles --create
```

Після повторних досліджень виявлено, що користувач `uid` з tar у github action дорівнює `1001`

tar Розпакування має бути, якщо немає параметра `--no-same-owner`, воно буде розпаковане до користувача з `uid` `1001`

У результаті rsync змінить групу користувачів кореневого каталогу, і після перезапуску ви не зможете увійти через ssh.

Змініть сценарій розпакування таким чином:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
