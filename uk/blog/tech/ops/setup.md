## Налаштувати Вхід Без Пароля

Через кілька хвилин усі сервери були активовані.

Перше, що потрібно відредагувати на локальному комп’ютері `~/.ssh/config` встановити псевдонім для IP а потім використовувати `ssh-copy-id root@c2` , щоб налаштувати вхід без пароля.

<img alt="" src="https://i-01.eu.org/1718287198.avif">

Ви можете змінити цей [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) , щоб швидше завершити налаштування ініціалізації.

Після завершення конфігурації без пароля ви можете використовувати її [pdsh](https://github.com/chaos/pdsh) Це зручно.

Перший — вручну ввімкнути ipv6 і перевірити

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

<img alt="" src="https://i-01.eu.org/1718287969.avif">

Потім оновіть систему до ubuntu 24 одним клацанням миші та інсталюйте програмне забезпечення, яке часто використовується.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 означає перемикання мови на китайську та часового поясу на Пекін.

Програмне забезпечення, встановлене за замовчуванням, налаштоване відповідно до моїх особистих звичок користування, наприклад [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) тощо. Якщо необхідно, ви можете скопіювати склад [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) і змінити його за потреби.

Цей крок займає багато часу. Після півгодини очікування оновлення та інсталяція програмного забезпечення, що часто використовується, буде успішним. Ви можете підтвердити це за допомогою такої команди:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh Прискорює ssh

Мережа з Європи в Китай не стабільна `ssh` Рекомендовано використовувати [mosh](//mosh.org)

`PATH` сценарій `c0` у каталозі `~/.bin`

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

Після створення надайте дозволи на виконання та програмне `c2` `c1`

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

У майбутньому ви можете безпосередньо ввести `c0` , `c1` і ви можете використовувати `mosh` високошвидкісне пряме з'єднання та ввести [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

[btrfs](https://wikipedia.org/wiki/Btrfs) Може робити знімки диска для легкого резервного копіювання.

Зміни відбитків пальців відбудуться `ssh-keygen -R ip` коли ssh

Потім запустіть наступний сценарій, щоб перетворити розділ

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

Сценарій автоматично перезапустить систему після успішного перетворення.

### Переглянути Ступінь Стиснення btrfs

За замовчуванням буде автоматично стиснуто таку команду:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### Ручне Стиснення

Якщо ви перфекціоніст, ви також можете виконати повне стиснення диска вручну

```bash
btrfs filesystem defragment -r -czstd /
```

Як показано на малюнку нижче, ви можете бачити, що після ручного стиснення в новій системі було стиснуто ще майже 100 МБ.

### Створіть Резервну Копію Знімка Btrfs За Допомогою snapper

## Деталі Системи Порятунку

Увійдіть у систему порятунку 22 Після того, як порт стане [tcping](https://github.com/paradiseduo/tcping) , ви не зможете отримати до нього доступ із встановленим паролем протягом 2-3 хвилин, і все буде добре.

```
tcping ip 22 -c 9999
```

Доступність портів можна контролювати.

### Як Отримати Доступ До Жорсткого Диска Після Входу В Режим Відновлення &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Ви можете переглянути, наприклад, знімок

```
snapper -c etc list
```

Перегляньте зміни файлу знімка

```
snapper -c etc status 5..6
```

Переглянути конкретний текст відмінностей

```
snapper -c etc diff 3..5
```

Відновити знімок: наприклад, скасувати всі зміни після знімка 5

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Увімкнути Збереження Журналу

Щоб увімкнути збереження журналу в системі відновлення, потрібно вручну створити відповідний каталог і виконати деякі додаткові кроки, оскільки у вас може не бути запущеної служби systemd. Ось кроки, які ви можете спробувати:

Спочатку chroot, потім

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Перезавантажте свою систему Ubuntu: після внесення вищезазначених змін вам потрібно перезавантажити комп’ютер у звичайному режимі та увійти до системи Ubuntu замість системи відновлення.

Таким чином, systemd-journald запуститься з новою конфігурацією та почне постійно зберігати журнали на жорсткому диску.

Майте на увазі, що наведені вище операції виконуються в системі відновлення, тому ці зміни набудуть чинності лише після перезавантаження системи, а не в режимі відновлення.

Це означає, що ви не відразу побачите, що будь-які зміни в існуючих журналах стануть постійними, але всі журнали, створені після цього, збережуться.

Таким чином ви можете переглянути журнали служби ssh

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Замітки Про Важкі Та Ускладнені Захворювання

### Вирішіть Дивну Проблему: Сервер Не Може Використовувати ssh

Раніше я використовував дію github для автоматичної компіляції та сценарію розгортання, але виявив, що служба розгортання не може використовувати ssh.

Вхід у систему відновлення та перегляд журналів є проблемою дозволу:

`Missing privilege separation directory: /run/sshd`

Ви можете створити цей каталог за допомогою такої команди

```
chown root:root /
systemd-tmpfiles --create
```

Після повторних досліджень це сталося тому github action що користувач, який завантажив `1001` tar `uid`

tar Розпаковування має виконуватися без параметрів `--no-same-owner` воно буде розпаковано в `uid` для користувача `1001`

У результаті rsync змінить групу користувачів кореневого каталогу, і тоді ви не зможете увійти в ssh після перезапуску.

Змініть сценарій декомпресії таким чином:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
