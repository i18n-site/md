# .i18n/conf.yml

Файл конфігурації для `i18n.site` — це `.i18n/conf.yml`, а його вміст такий:

```yaml
i18n:
  fromTo:
    en:
upload:
  ext:
    - md
nav:
  - i18n: home
    use: Toc
    url: /
  - i18n: doc
    menu: NB demo1,demo2
    use: Doc
  - i18n: blog
    use: Blog
addon:
  - i18n.addon/toc
```

Серед них параметр `upload` з `ext:` означає, що під час публікації буде завантажено лише файли з розширенням `.md`.

## Верхня навігаційна панель

Параметри конфігурації `nav:` відповідають навігаційному меню на верхівці головної сторінки.

<img src="//p.3ti.site/1721051426.avif" style="width:320px">

Серед них `i18n: home` відповідає `home: Home` у файлі `en/i18n.yml` (де `en` — вихідна мова проекту для перекладу).

Вміст файлу `en/i18n.yml` — це текст, що відображається в меню навігації, який буде перекладено відповідно до параметра `fromTo` у налаштуваннях, наприклад, на `zh/i18n.yml`.

<img src="//p.3ti.site/1721051689.avif" style="width:320px">

Після завершення перекладу ви можете змінити значення перекладу в `yml`, але не додавайте та не видаляйте ключі перекладу `yml`.

### `use: Toc` — шаблон для файлу з контуром

У конфігурації `nav`:

```
  - i18n: home
    use: Toc
    url: /
```

`use: Toc` означає, що буде використовуватися шаблон `Toc` для рендерингу, тобто для відображення одного файлу `Markdown`.

`TOC` — це абревіатура від `Table of Contents`. Коли цей шаблон рендериться, контур файлу `Markdown` буде відображений на бічній панелі.

`url:` вказує на шлях до файлу `Markdown` ( `/` відповідає кореневому каталогу `/README.md`, ім'я файлу повинно починатися з великої літери та закінчуватися малою).

### `use: Md` — шаблон для файлу без контуру

Шаблони `Md` та `Toc` є однаковими та використовуються для рендерингу одного файлу `Markdown`, але шаблон `Md` не відображає контур на бічній панелі.

Ви можете змінити `use: Toc` на `use: Md` у наведеній вище конфігурації, потім знову запустити `i18n.site` в каталозі `md` та відвідати URL для попереднього перегляду, щоб спостерігати за змінами на головній сторінці.

### `use: Blog` — шаблон для блогу

Шаблон блогу відображає список статей (заголовки та анотації) у хронологічному порядку.

[→ Натисніть тут, щоб дізнатися про конкретну конфігурацію](/i18n.site/conf/blog)

### `use: Doc` — багатофайловий шаблон документації

У файлі конфігурації:

```
  - i18n: doc
    menu: NB demo1,demo2
    use: Doc
```

Вказує на використання `Doc` для рендерингу шаблону.

Шаблон `Doc` підтримує інтеграцію кількох файлів `Markdown` для генерації контенту для одного або кількох проектів.

#### Багатопроектний багатофайловий режим

Конфігурація `.i18n/conf.yml` для `i18n:doc` — це багатопроектний режим рендерингу кількох файлів.

Тут `menu: NB demo1,demo2` означає використання шаблону `NB` для рендерингу спадного меню.

<img src="//p.3ti.site/1721275191.avif" width="320px">

`NB` — це абревіатура від `Name Brief`, що означає, що в спадному меню можна відображати назву та слоган проекту.

`NB` супроводжується параметром `demo1,demo2`.

Зверніть увагу: **не повинно бути пробілів** перед і після коми `,` в `demo1,demo2`.

Для наведених вище параметрів відповідний файл індексу каталогу:

* `en/demo1/TOC`
* `en/demo2/TOC`

#### Один проект з кількома файлами

Якщо у вас лише один проект, ви можете налаштувати його наступним чином.

```
  - i18n: xxx
    use: Doc
```

##### Якщо URL-Адреса Порожня, За Умовчанням Використовується Значення i18n

Якщо не вказано `url`, за замовчуванням `url` має значення `i18n`. Це правило також застосовується до інших шаблонів.

Наведена вище запись еквівалентна `url: xxx`, і відповідний файл — це `en/xxx/TOC`.

#### Індекс каталогу TOC

`i18n.site` викличе `js` плагін `.i18n/hook/after.tran/TOC.js` у демонстраційному сховищі, щоб прочитати файл індексу `TOC` каталогу, який відповідає конфігурації шаблону `doc`, і створити структуру `json` каталогу.

Якщо ви використовуєте шаблон `doc`, вам необхідно мати цей плагін.

Якщо ви ініціалізуєте проект `i18n.site` з порожньої папки, не забудьте скопіювати демонстраційний проект `.i18n` у ваш каталог.

Шаблон `Doc` відображатиме структуру змісту на основі згенерованого `json`.

##### Детальний опис вмісту

Вміст `en/blog/TOC` такий:

```
README.md

news/README.md
  news/begin.md
```

##### Використовуйте відступи для позначення рівнів

`README.md` у першому рядку `en/blog/TOC` відповідає `i18n.site`, тобто назві проекту.

Наступні два рядки, як показано на знімку екрана нижче.

<img src="https://p.3ti.site/1721097381.avif" style="width:320px">

`news/README.md` відповідає `News`
`news/begin.md` відповідає `Our Product is Online !`

Файли `TOC` використовують відступи для позначення ієрархії вмісту, підтримують багаторівневі відступи та коментарі, що починаються з `#`.

##### Батьківський рівень записує лише назву, а не вміст

У разі багаторівневих відступів батьківський рівень записує лише назву, а не вміст, щоб уникнути типографічних помилок.

##### Проект README.md

Вміст можна записати в файл `README.md`, наприклад `en/demo2/README.md`.

Зверніть увагу, що вміст цього файлу не містить структуру змісту, тому рекомендується обмежити його довжину та написати короткий вступ.

###### Слоган проекту

Ви можете побачити, що `Deme Two` має свій слоган під спадним меню та назву проекту `Your Project slogan`.

![](https://p.3ti.site/1721276842.avif)

Це відповідає першому рядку `en/demo2/README.md`:

```
# Demo Two : Your Project slogan
```

Вміст після першої двокрапки `:` у заголовку першого рівня файлу `README.md` буде розглядатися як слоган проекту.

Користувачі з Китаю, Японії та Кореї, зверніть увагу, що ви повинні використовувати половинну двокрапку `:` замість повної.

##### Як масово перемістити TOC?

Файли `TOC` повинні розміщуватися в каталозі вихідної мови.

Наприклад, якщо вихідною мовою є китайська, то `TOC` означає `zh/blog/TOC`.

Якщо вихідну мову змінено, вам потрібно масово перемістити `TOC` файли певної мови в проекті на іншу мову.

Ви можете скористатися такими командами:

```
rsync -av --remove-source-files --include='*/' \
--include='TOC' --exclude='*' en/ zh/
```

Будь ласка, змініть `en/` та `zh/` у наведеній вище команді на код вашої мови.

### Завантаження за умовчанням без конфігураційного шляху

Для певного шляху, до якого здійснюється доступ, якщо префікс шляху не налаштовано в `nav:`, файл `Markdown`, що відповідає шляху, буде завантажено за умовчанням і відтворено за допомогою шаблону `Md`.

Наприклад, якщо здійснюється доступ до `/test`, а `nav:` не налаштовано для цього шляху, і поточна мова веб-перегляду — англійська (код `en`), `/en/test.md` буде завантажено за умовчанням і відображено за допомогою шаблону `Md`.

Якщо `/en/test.md` цей файл не існує, буде відображено стандартну сторінку `404`.
