# Встановити &

## Налаштування маркера доступу

`i18n.site` має вбудований інструмент перекладу `i18`, [натисніть тут, щоб ознайомитися з документацією `i18` для налаштування маркера доступу](/i18/use).

## Встановити

```sh
bash <(curl -sS https://i.i18n.site) i18n.site
```

## Демонстраційний проект

Давайте розпочнемо з демонстраційного проекту, щоб дізнатися, як використовувати `i18n.site`.

Спочатку клонуйте демонстраційний репозиторій і виконайте наступну команду:

```
git clone https://github.com/i18n-site/demo.i18n.site.git md
git clone https://github.com/i18n-site/demo.i18n.site.docker.git docker
```

Користувачі в Китаї можуть:

```
git clone https://atomgit.com/i18n/demo.i18n.site.git md
git clone https://atomgit.com/i18n/demo.i18n.site.docker.git docker
```

Ім'я каталогу з клонованим кодом `demo.i18n.site` має бути `md`, щоб полегшити локальний попередній перегляд за допомогою `docker`.

### Опублікування вмісту

`i18n.site` використовує архітектуру [односторінкової програми](https://developer.mozilla.org/docs/Glossary/SPA), а сторінка входу на веб-сайт і вміст веб-сайту розгортаються незалежно.

Після виконання перекладу, каталоги `htm` і `v` будуть створені в каталозі `md/out/dev`.

Тут `dev` означає, що це створено на основі файлу конфігурації `.i18n/htm/dev.yml`.

`dev` каталог:

Каталог `htm` містить сторінку входу на сайт.

Каталог `v` містить вміст веб-сайту з версійними номерами.

Локальний попередній перегляд не враховує версійні номери та копіює всі файли до каталогу `out/dev/v/0.1.0`.

Для офіційного випуску змінені файли будуть скопійовані до каталогу з новим номером версії.

#### Використовуйте -C, Щоб Вказати Файл Конфігурації

Різні конфігураційні файли створять відповідні каталоги в каталозі `out`.

Наприклад, `.i18n/htm/ol.yml` створить каталог `out/ol`.

`dev.yml` і `ol.yml` є стандартними конфігураціями.

`dev` — скорочення від `development`, що вказує на середовище розробки, яке використовується для локального попереднього перегляду, а також є стандартним файлом конфігурації.
`ol` — скорочення від `online`, що вказує на онлайн-середовище, яке використовується для офіційного випуску. Це також стандартний файл конфігурації при використанні командного рядка `-n` для випуску через `npm`.

Ви також можете створити інші файли конфігурації, вказавши назву файлу конфігурації в командному рядку `--htm_conf`

наприклад:
```
i18n.site --htm_conf yourConfig --save
```

Тут `--save` означає оновлення версії випуску.

#### <a rel=id href="#npm" id="npm"></a> Публікування вмісту на npmjs.com

Публікація вмісту [в](/1) [npmjs.com](/0)

##### npm login & Реліз

Встановіть `nodejs` і увійдіть за допомогою `npm login`.

Редагуйте `md/.i18n/htm/ol.yml` і змініть `i18n.site` в `v: //unpkg.com/i18n.site` на свою назву пакета `npm`.

Просто використовуйте вільне ім'я пакета на [npmjs.com](//npmjs.com).

Під час публікації на основі пакета `npm` **обов’язково використовуйте `//unpkg.com/` для** префікса `v:` , `i18n.site` спеціально оптимізував час кешу `/.v` під цим префіксним шляхом, щоб забезпечити своєчасний перегляд нових випусків.

Запустіть `i18n.site --npm` або `i18n.site -n` у каталозі `md`, щоб перекласти та опублікувати.

Якщо ви використовуєте середовище безперервної інтеграції для публікації, немає необхідності встановлювати `nodejs`. Просто скопіюйте дозволи для входу та публікації `~/.npmrc` до цього середовища.

Якщо ви змінюєте назву пакета `v:` в `ol.yml`, будь ласка, **спочатку видаліть `.i18n/v/ol`**, а потім опублікуйте його.

##### Проксі-сервер для публікації npm

Якщо користувачі в материковому Китаї стикаються з проблемами мережі та не можуть опублікувати `npm` пакети, вони можуть встановити змінну середовища `https_proxy` для налаштування проксі-сервера.

Припускаючи, що порт вашого проксі-сервера дорівнює `7890`, ви можете написати:

```
https_proxy=http://127.0.0.1:7890 i18n.site -n
```

#### Самостійне розміщення вмісту

Якщо ви хочете самостійно розмістити вміст, спочатку відредагуйте `md/.i18n/htm/ol.yml` і змініть `v: //unpkg.com/i18n.site` на префікс URL-адреси, наприклад `v: //i18n-v.xxx.com`.

Перейдіть до каталогу `md` і запустіть

```
i18n.site --htm_conf ol --save
```

або скорочення

```
i18n.site -c ol -s
```

Потім налаштуйте вміст у каталозі `md/out/ol/v` на шлях префікса URL-адреси, встановленого у `v:`.

Нарешті, **налаштуйте час кешу шляху, що закінчується на `/.v`, до `1s`**, інакше до нещодавно випущеного вмісту буде неможливо отримати доступ негайно.

Час кешу для інших шляхів можна встановити на один рік або більше, щоб зменшити непотрібні запити.

##### Розміщення вмісту на S3

Для самостійного розміщення вмісту, крім використання власного сервера, поширеним варіантом є використання `S3` + `CDN`.

Ви можете використовувати [rclone](https://rclone.org) для входу на сервер `S3`, потім переглянути та змінити наступний сценарій та скопіювати лише додаткові зміни до `S3` для кожного випуску.

```bash
i18n.site -c ol -s
s3=your-s3
bucket=your-bucket
ver=$(head -1 .i18n/v/ol/v.hash | cut -c 2-)
rclone copy --overwrite-dir out/ol/htm/v/$ver $s3:/$bucket/$ver
rclone copy out/ol/v/.v "$s3:/$bucket/"
```

Не забувайте налаштувати `CDN`, щоб час кешу шляху, що закінчується на `/.v`, становив `1s`, інакше доступ до нещодавно випущеного вмісту буде неможливим негайно.

### Опублікування веб-сайту

Веб-сайт можна розгорнути де завгодно, [github page](https://pages.github.com) і [cloudflare page](https://pages.cloudflare.com) є хорошими варіантами.

Оскільки веб-сайт використовує архітектуру [односторінкової програми](https://developer.mozilla.org/docs/Glossary/SPA), не забудьте переписати URL-шляхи, які не містять `.`, до `index.html`.

Сторінка входу веб-сайту потрібно розгортати лише один раз, і немає необхідності повторно розгортати сторінку входу веб-сайту для наступних оновлень вмісту.

Потім створіть склад `i18n-demo.github.io` у цій організації (замініть `i18n-demo` на назву організації, яку ви створили):

#### Оптимізація розгортання веб-сайту в Китаї

Якщо ви хочете отримати кращу доступність у мережевому середовищі материкового Китаю, спочатку [зареєструйте доменне ім’я](//beian.aliyun.com).

Потім скористайтеся сховищем об’єктів хмарних постачальників у материковому + `CDN` для розгортання наступного вмісту `out/ol/htm`.

Ви можете використовувати периферійні обчислення для перепису шляхів для адаптації до односторінкових програм. Наприклад, [Baidu Smart Cloud `CDN`](//cloud.baidu.com/product/cdn.html) можна налаштувати так:

```js
var uri=r.uri,p=uri.lastIndexOf('.');
if(
  p<0 || !'|avif|css|html|ico|js|json|png|svg|txt|webmanifest|xml|'.includes('|'+uri.slice(p+1)+'|')
){
  r.uri='/index.html'
}
r.respHeader(()=>{
var t = [];
r.rawHeadersOut.forEach((i)=>{
    var out = r.headersOut;
    var key = i[0].toLowerCase();
    if(key.startsWith('x-')||key.startsWith('ohc-')){
        delete out[key]
    }
    out['Cache-Control']='max-age='+9e5;
    ['Content-MD5','Age','Expires','Last-Modified'].forEach((i)=>delete out[i])
})
})
```
![](https://p.3ti.site/1721121273.avif)

Оскільки запис `MX` і запис `CNAME` не можуть співіснувати, якщо ви хочете отримувати електронні листи з доменними іменами одночасно, вам потрібно співпрацювати зі сценарієм [cname_flatten](https://github.com/i18n-site/lib/tree/main/cname_flatten) для перетворення `CNAME` у запис `A`.

Крім того, оскільки плата за закордонний трафік постачальників хмарних технологій у материковому Китаї є відносно високою, якщо ви хочете оптимізувати витрати, ви можете скористатися [безкоштовною географічною роздільною здатністю Huawei DNS](https://support.huaweicloud.com/usermanual-dns/dns_usermanual_0041.html) і спеціальним доменним іменем [Cloudflare for SaaS](https://developers.cloudflare.com/cloudflare-for-platforms/cloudflare-for-saas) (як показано нижче) для перенаправлення трафіку ── трафік у материковому Китаї через Baidu Cloud `CDN`, міжнародний трафік через cloudflare.

![](https://p.3ti.site/1721119788.avif)

Ці рішення для оптимізації розгортання є досить складними та будуть детально розглянуті в окремих главах у майбутньому.

#### Переспрямування усього доменного імені

Якщо ви використовуєте `i18n.site` для створення вашого основного веб-сайту, зазвичай необхідно налаштувати переспрямування для усього доменного імені, тобто спрямувати відвідування `*.xxx.com` (включаючи `www.xxx.com`) на `xxx.com`.

Цю вимогу можна реалізувати за допомогою `EdgeScript` у Alibaba Cloud `CDN` ( [англійська документація](https://www.alibabacloud.com/help/en/cdn/developer-reference/how-edgescript-works) / [китайська документація](https://help.aliyun.com/zh/cdn/developer-reference/edgescript) )

Додайте доменне ім'я до [Alibaba Cloud CDN](https://cdn.console.aliyun.com/domain/list) та вкажіть `*.xxx.com` на `CNAME` Alibaba Cloud `CDN`.

![](https://p.3ti.site/1721122000.avif)

Наприклад, конфігурація для переспрямування усього доменного імені `*.i18n.site` виглядає так:

```
rewrite(concat('https://i18n.site/',$uri), 'redirect',301)
```

![](https://p.3ti.site/1721121934.avif)

#### Розгортання за допомогою nginx

Будь ласка, додайте наступну конфігурацію до розділу `server` у nginx, змінивши `/root/i18n/md/out/ol/htm` на шлях до вашого проекту `out/ol/htm`:

```
location / {
  root  /root/i18n/md/out/ol/htm;
  add_header Cache-Control "max-age=9999999";
  if ($uri !~* \.(avif|css|html|ico|js|json|png|svg|txt|webmanifest|xml)$) {
     rewrite ^ /index.html last;
  }
}
```

### Структура каталогу

#### `public`

Статичні файли веб-сайту, такі як `favicon.ico`, `robots.txt` тощо.

Значки можна згенерувати за допомогою [realfavicongenerator.net](https://realfavicongenerator.net).

#### `.i18n`

У каталозі `.i18n` зберігаються конфігураційні файли та кеш перекладів для `i18n.site`. Докладнішу інформацію див. у наступній главі [«Налаштування»](/i18n.site/conf).

#### `en`

Каталог вихідної мови, що відповідає значенню `en` у параметрі `fromTo` конфігураційного файлу `.i18n/conf.yml`

```yaml
i18n:
  fromTo:
    en: zh
```

Для налаштування конфігурації перекладу зверніться до [i18](/i18/use).
