## پاس ورڈ سے پاک لاگ ان کو ترتیب دیں۔

چند منٹوں کے بعد، تمام سرورز ایکٹیویٹ ہو گئے۔

پہلی چیز IP کہ مقامی مشین کے `~/.ssh/config` ایک عرف سیٹ کرنے کے لیے ترمیم کریں، اور پھر پاس ورڈ سے پاک لاگ ان کو ترتیب دینے کے لیے `ssh-copy-id root@c2` استعمال کریں۔

![](https://i-01.eu.org/1718287198.avif)

ابتدائی کنفیگریشن کو تیزی سے مکمل کرنے کے لیے آپ اس [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) ترمیم کر سکتے ہیں۔

پاس ورڈ سے پاک کنفیگریشن مکمل ہونے کے بعد، آپ اسے استعمال کر سکتے ہیں اسے بیچوں میں چلایا جا سکتا ہے، جو کہ آسان ہے [pdsh](https://github.com/chaos/pdsh)

سب سے پہلے ipv6 اور ٹیسٹ کو دستی طور پر فعال کرنا ہے۔

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

پھر ایک کلک کے ساتھ سسٹم کو اوبنٹو 24 میں اپ گریڈ کریں اور عام طور پر استعمال ہونے والا سافٹ ویئر انسٹال کریں۔

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 کا مطلب ہے زبان کو چینی میں اور ٹائم زون کو بیجنگ میں تبدیل کرنا۔

بطور ڈیفالٹ انسٹال کردہ سافٹ ویئر میری ذاتی استعمال کی عادات کے مطابق ترتیب دیا گیا ہے، جیسے کہ [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) ، وغیرہ [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu)

اس قدم میں کافی وقت لگتا ہے آدھے گھنٹے کے انتظار کے بعد عام طور پر استعمال ہونے والے سافٹ ویئر کی اپ گریڈیشن اور انسٹالیشن کامیاب ہو جائے گی۔ آپ درج ذیل کمانڈ سے اس کی تصدیق کر سکتے ہیں:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

![](https://i-01.eu.org/1718290346.avif)

## mosh ssh کو تیز کرتا ہے۔

یورپ سے چین تک نیٹ ورک مستحکم نہیں ہے `ssh` تیز کرنے کے لئے استعمال کیا جاتا ہے [mosh](//mosh.org)

مقامی ڈائرکٹری `~/.bin` میں `c0` اسکرپٹس بنائیں (بنیادی طور پر ماحولیاتی متغیر `PATH` میں شامل)

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

تخلیق کے بعد، قابل عمل اجازتیں اور نرم لنک `c1` اور `c2` دیں۔

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

مستقبل میں، آپ `mosh` تیز رفتار براہ راست کنکشن استعمال کرنے کے لیے `c0` اور `c1` کو براہ راست درج کر سکتے ہیں اور درج کر سکتے ہیں [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## پارٹیشن فارمیٹ کو ext4 سے btrfs میں تبدیل کریں۔

آسانی سے بیک اپ کے لیے ڈسک کے اسنیپ شاٹس لے سکتے ہیں [btrfs](https://wikipedia.org/wiki/Btrfs)

پہلے ریسکیو سسٹم میں داخل ہوں۔

![](https://i-01.eu.org/1718300448.avif)

ڈیبین ریسکیو کا انتخاب کریں۔

![](https://i-01.eu.org/1718300753.avif)

سرور کے دوبارہ شروع ہونے کا انتظار کریں۔

فنگر پرنٹ میں تبدیلیاں ہوں گی، اسے صاف کرنے کے لیے `ssh-keygen -R ip` استعمال کریں ssh

پھر تقسیم کو تبدیل کرنے کے لیے درج ذیل اسکرپٹ کو چلائیں۔

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

تبدیلی کے کامیاب ہونے کے بعد اسکرپٹ خود بخود سسٹم کو دوبارہ شروع کردے گا۔

### btrfs کمپریشن تناسب دیکھیں

یہ خود بخود کمپریشن ریشو کو چیک کرنے کی کمانڈ مندرجہ ذیل ہے:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### دستی کمپریشن

اگر آپ پرفیکشنسٹ ہیں تو آپ دستی طور پر مکمل ڈسک کمپریشن بھی کر سکتے ہیں۔

```bash
btrfs filesystem defragment -r -czstd /
```

جیسا کہ نیچے دی گئی تصویر میں دکھایا گیا ہے، آپ دیکھ سکتے ہیں کہ نئے سسٹم میں دستی کمپریشن کے بعد، تقریباً 100MB مزید کمپریس کیا گیا تھا۔

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### سنیپر کے ساتھ ایک Btrfs سنیپ شاٹ بیک اپ بنائیں

Btrfs بیک اپ کے لیے بہترین پارٹنر ہے [snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html)

ابتداء کا اسکرپٹ `~/ubuntu` ڈائرکٹری بنائے گا، صرف درج ذیل کمانڈ کو چلائیں:

```
~/ubuntu/snapper/init.sh
```

اسکرپٹ کے مواد میں ملتے ہیں [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

ذیلی حجم دیکھنے کے لیے `snapper list-configs` استعمال کریں جن کے لیے سنیپ شاٹس سیٹ کیے گئے ہیں۔

![](https://i-01.eu.org/1718302689.avif)

ذیلی حجم کا سنیپ شاٹ دیکھنے کے لیے `snapper -c mnt list` استعمال کریں۔

`/etc/snapper/configs/mnt` اس ذیلی حجم کی مخصوص بیک اپ سیٹنگز دیکھ سکتا ہے۔

مزید حوالہ جات:

[آسانی سے Btrfs کے سنیپ شاٹ فنکشن کے ساتھ کھیلیں snapper](https://zhuanlan.zhihu.com/p/31082518)

## ریسکیو سسٹم کی تفصیلات

ریسکیو سسٹم داخل کریں 22 پورٹ [tcping](https://github.com/paradiseduo/tcping) قابل رسائی ہونے کے بعد، آپ 2-3 منٹ کے لیے اس تک رسائی حاصل نہیں کر پائیں گے اور یہ ٹھیک ہو جائے گا۔

```
tcping ip 22 -c 9999
```

بندرگاہ تک رسائی کی نگرانی کی جا سکتی ہے۔

### ریسکیو ریسٹور اسنیپ شاٹ میں داخل ہونے کے بعد ہارڈ ڈرائیو تک کیسے رسائی حاصل کریں &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

مثال کے طور پر آپ سنیپ شاٹ دیکھ سکتے ہیں۔

```
snapper -c etc list
```

سنیپ شاٹ فائل میں تبدیلیاں دیکھیں

```
snapper -c etc status 5..6
```

مخصوص فرق متن دیکھیں

```
snapper -c etc diff 3..5
```

سنیپ شاٹ بحال کریں: مثال کے طور پر، سنیپ شاٹ 5 کے بعد تمام تبدیلیوں کو کالعدم کریں۔

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### لاگ استقامت کو فعال کریں۔

ریسکیو سسٹم کے تحت لاگ پرسٹینس کو فعال کرنے کے لیے آپ کو متعلقہ ڈائرکٹری کو دستی طور پر تخلیق کرنے اور کچھ اضافی اقدامات کرنے کی ضرورت ہوتی ہے، کیونکہ ہو سکتا ہے کہ آپ کے پاس سسٹم ڈی سروس نہ ہو۔ یہاں وہ اقدامات ہیں جنہیں آپ آزما سکتے ہیں:

پہلے چروٹ، پھر

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

اپنے Ubuntu سسٹم میں ریبوٹ کریں: مندرجہ بالا تبدیلیاں کرنے کے بعد، آپ کو اپنی مشین کو عام طور پر اور ریسکیو سسٹم کے بجائے اپنے Ubuntu سسٹم میں دوبارہ شروع کرنے کی ضرورت ہے۔

اس طرح سے، systemd-journald نئی کنفیگریشن کے ساتھ شروع ہو جائے گا اور ہارڈ ڈسک پر لاگز کو مستقل طور پر ذخیرہ کرنا شروع کر دے گا۔

ذہن میں رکھیں کہ اوپر کی کارروائیاں ریسکیو سسٹم پر کی جاتی ہیں، لہٰذا یہ تبدیلیاں اس وقت تک اثر انداز نہیں ہوں گی جب تک کہ سسٹم دوبارہ شروع نہ ہو اور ریسکیو موڈ میں نہ ہو۔

اس کا مطلب ہے کہ آپ کو فوری طور پر موجودہ لاگز میں کوئی ایسی تبدیلی نظر نہیں آئے گی جو مستقل ہو جائیں، لیکن اس کے بعد پیدا ہونے والے تمام لاگز برقرار رہیں گے۔

اس طرح آپ ssh سروس کے لاگز دیکھ سکتے ہیں۔

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## مشکل اور پیچیدہ بیماریوں پر نوٹس

### عجیب مسئلہ حل کریں: سرور ssh نہیں کر سکتا

میں نے اس سے پہلے خودکار تالیف اور تعیناتی اسکرپٹ بنانے کے لئے گیتھب ایکشن کا استعمال کیا تھا ، لیکن پتہ چلا کہ تعیناتی سروس ssh نہیں ہوسکتی ہے۔

ریسکیو سسٹم میں داخل ہونا اور لاگز کو دیکھنا اجازت کا مسئلہ ہے:

`Missing privilege separation directory: /run/sshd`

آپ اس ڈائرکٹری کو درج ذیل کمانڈ سے بنا سکتے ہیں۔

```
chown root:root /
systemd-tmpfiles --create
```

بار بار کی چھان بین کے بعد، اس کی وجہ یہ ہے کہ tar کا صارف `uid` `1001` ہے github action

پیک کھولنا ضروری ہے اگر کوئی پیرامیٹر `--no-same-owner` نہ ہو تو اسے `1001` صارفین کے لیے `uid` پر ڈیکمپریس کیا جائے گا tar

نتیجے کے طور پر، rsync روٹ ڈائرکٹری کے صارف گروپ کو تبدیل کر دے گا، اور پھر آپ دوبارہ شروع کرنے کے بعد ssh میں لاگ ان نہیں ہو سکیں گے۔

ڈیکمپریشن اسکرپٹ میں اس طرح ترمیم کریں:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
