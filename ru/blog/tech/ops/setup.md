## Настроить Вход Без Пароля

Через несколько минут все серверы были активированы.

Первым делом нужно отредактировать `~/.ssh/config` на локальном компьютере. Установите псевдоним для IP а затем используйте `ssh-copy-id root@c2` для настройки входа без пароля.

<img alt="" src="https://i-01.eu.org/1718287198.avif">

Вы можете изменить этот [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) , чтобы ускорить настройку инициализации.

После завершения настройки без пароля вы можете использовать ее [pdsh](https://github.com/chaos/pdsh) Ее можно использовать в пакетном режиме, что удобно.

Первый — вручную включить ipv6 и протестировать

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

<img alt="" src="https://i-01.eu.org/1718287969.avif">

Затем обновите систему до Ubuntu 24 одним щелчком мыши и установите часто используемое программное обеспечение.

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 означает переключение языка на китайский и часового пояса на Пекин.

Программное обеспечение, установленное по умолчанию, настроено в соответствии с моими личными привычками использования, такими как [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) и т. д. При необходимости вы можете скопировать хранилище [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) и изменить его по мере необходимости.

Этот шаг занимает много времени. После получасового ожидания обновление и установка часто используемого программного обеспечения пройдут успешно. Вы можете подтвердить это с помощью следующей команды:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## mosh Ускоряет ssh

Сеть из Европы в Китай не стабильна. Рекомендуется использовать [mosh](//mosh.org) Приходите, чтобы ускориться `ssh`

Создайте сценарий `c0` в каталоге `~/.bin` на локальном компьютере (в основном добавляемом в переменную среды `PATH` ) следующим образом:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

После создания предоставьте разрешения на исполняемый файл и программную ссылку `c1` , `c2` .

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

В дальнейшем вы можете напрямую вводить `c0` , `c1` и использовать высокоскоростное прямое соединение `mosh` и вводить [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

### Посмотреть Степень Сжатия btrfs

По умолчанию он автоматически сжимается. Команда для проверки степени сжатия выглядит следующим образом:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

Если вы перфекционист, вы также можете выполнить полное сжатие диска вручную. Команда выглядит следующим образом.

```bash
btrfs filesystem defragment -r -czstd /
```

### Как Получить Доступ К Жесткому Диску После Входа В Режим Восстановления &

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

Вы можете просмотреть снимок, например

```
snapper -c etc list
```

Просмотр изменений файла снимка

```
snapper -c etc status 5..6
```

Просмотр конкретного текста различий

```
snapper -c etc diff 3..5
```

Восстановить снимок: например, отменить все изменения после снимка 5.

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### Включить Сохранение Журнала

Для включения сохранения журнала в системе восстановления необходимо вручную создать соответствующий каталог и выполнить некоторые дополнительные действия, поскольку у вас может не быть работающей службы systemd. Вот шаги, которые вы можете попробовать:

Сначала chroot, затем

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

Перезагрузитесь в систему Ubuntu. После внесения вышеуказанных изменений вам необходимо перезагрузить компьютер в обычном режиме и в системе Ubuntu вместо системы восстановления.

Таким образом, systemd-journald запустится с новой конфигурацией и начнет постоянно хранить журналы на жестком диске.

Имейте в виду, что вышеуказанные операции выполняются в аварийной системе, поэтому эти изменения вступят в силу только после перезагрузки системы и выхода из режима восстановления.

Это означает, что вы не сразу увидите, что какие-либо изменения в существующих журналах станут постоянными, но все журналы, созданные после этого, будут сохранены.

Таким образом вы можете просмотреть логи службы ssh.

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## Заметки О Сложных И Сложных Заболеваниях

### Решите Странную Проблему: Сервер Не Может Подключиться По ssh.

Раньше я использовал действие github для создания сценария автоматической компиляции и развертывания, но обнаружил, что служба развертывания не может быть ssh.

Вход в систему спасения и просмотр логов — это проблема с разрешением:

`Missing privilege separation directory: /run/sshd`

Вы можете создать этот каталог с помощью следующей команды

```
chown root:root /
systemd-tmpfiles --create
```

После неоднократных расследований это произошло потому github action что tar загрузил `1001` `uid`

tar Распаковка должна производиться без параметров `--no-same-owner` она будет распакована в `uid` для `1001` user

В результате rsync изменит группу пользователей корневого каталога, и тогда вы не сможете зайти по ssh после перезапуска.

Измените сценарий декомпрессии следующим образом:

```bash
tar --no-same-owner -xvf $TZT
rm $TZT

if [ -f "init.sh" ]; then
  ./init.sh
  rm init.sh
fi

rsync --remove-source-files -av . /
```
