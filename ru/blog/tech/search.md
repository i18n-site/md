## Обзор Решений Для Бессерверного Полнотекстового Поиска

Для небольших и средних чисто статических веб-сайтов, таких как документы/личные блоги, создание самостоятельного механизма полнотекстового поиска слишком сложно, и более распространенным выбором является полнотекстовый поиск без обслуживания.

Решения для безсервисного полнотекстового поиска делятся на две основные категории:

Первая категория — это такие сторонние��索-провайдеры, как [algolia.com](//algolia.com), которые предоставляют фронтенд-компоненты для полнотекстового поиска.

За такие услуги необходимо платить в зависимости от объёма поиска, и они часто недоступны для пользователей из материкового Китая из-за проблем с соответствием требованиям.

Его нельзя использовать в автономном режиме, нельзя использовать в интрасети и он имеет большие ограничения. В этой статье многого не обсуждается.

Вторая категория — это чисто фронтендный полнотекстовый поиск.

В настоящее время常见的纯前端全文搜索工具包括 [lunrjs](//lunrjs.com) и [ElasticLunr.js](//github.com/weixsong/elasticlunr.js) (разработан на основе `lunrjs`).

`lunrjs` имеет два метода создания индексов, и у обоих есть свои проблемы.

1. Готовые индексные файлы

   Поскольку индекс содержит слова из всех документов, он имеет большой размер.
   При каждом добавлении или изменении документа необходимо загрузить новый индексный файл.
   Это увеличит время ожидания пользователя и потребует много трафика.

2. Загружайте документы и создавайте индексы на лету

   Создание индекса — это трудоемкая задача, требующая больших вычислительных ресурсов. Перестроение индекса каждый раз, когда вы к нему обращаетесь, приведет к очевидным задержкам и ухудшению взаимодействия с пользователем.

---

Помимо `lunrjs`, существуют и другие решения для полнотекстового поиска, например:

[fusejs](//www.fusejs.io) использует вычисление相似ности строк для поиска.

Этот метод имеет крайне низкую производительность и не годится для全文ового поиска (см. [Fuse.js: запросы longer_than 10 секунд, как оптимизировать?](//stackoverflow.com/questions/70984437/fuse-js-takes-10-seconds-with-semi-long-queries)).

[TinySearch](//github.com/tinysearch/tinysearch) использует фильтр Блума для поиска, но не поддерживает搜索 по префиксу (например, при вводе `goo` не найдет `good` или `google`), и не может обеспечить эффект автоматического дополнения.

Из-за недостатков существующих решений `i18n.site` разработал全新的 чисто фронтендный метод полнотекстового поиска с следующими особенностями:

1. Поддерживает многоязычный поиск и имеет небольшой размер. Размер поискового ядра после упаковки `gzip` составляет `6.9KB` (для сравнения, размер `lunrjs` составляет `25KB`)
1. Создает инвертированный индекс на основе `indexedb`, который занимает меньше памяти и работает быстрее
1. При добавлении/изменении документов переиндексируются только добавленные или измененные документы, что сокращает объем вычислений
1. Поддерживает поиск по префиксам и может отображать результаты поиска в реальном времени по мере ввода пользователем
1. Доступно офлайн

Ниже详细介绍ены `i18n.site` технические детали реализации.

## Многоязычная сегментация слов

Сегментация слов использует встроенную в браузер функцию `Intl.Segmenter`, которую поддерживают все основные браузеры.

![](//p.3ti.site/1727667759.avif)

Пример кода на `coffeescript` для сегментации слов:

```coffee
SEG = new Intl.Segmenter 0, granularity: "word"

seg = (txt) =>
  r = []
  for {segment} from SEG.segment(txt)
    for i from segment.split('.')
      i = i.trim()
      if i and !'|`'.includes(i) and !/\p{P}/u.test(i)
        r.push i
  r

export default seg

export segqy = (q) =>
  seg q.toLocaleLowerCase()
```

Где:

* `/\p{P}/` — регулярное выражение, соответствующее знакам препинания. К специальным символам соответствия относятся: `! " # $ % & ' ( ) * + , - . / : ; < = > ? @ [ \ ] ^ _ ` { | } ~. `.</p><ul><li> `split('.')` — это потому, что сегментация `Firefox` слова браузера не сегментирует `.` .</li>


## Построение индекса

В `IndexedDB` созданы пять таблиц объектов:

* `word`: id - слово
* `doc`: id - URL документа - версия документа
* `docWord`: id документа - массив id слов
* `prefix`: префикс - массив id слов
* `rindex`: id слова - id документа : массив номеров строк

Передается массив `url` документа и версия `ver`, и в таблице `doc` проверяется, существует ли документ. Если нет, создается инвертированный индекс. В то же время удаляются инвертированные индексы для документов, которые не были переданы.

Таким образом, достигается Incremental Indexing, что уменьшает объем вычислений.

В интерфейсе взаимодействия можно отображать индикатор прогресса загрузки индекса, чтобы избежать sensation замедления при первой загрузке. Смотри «Анимированный индикатор прогресса с использованием одного progress + Pure css» [английский](//dev.to/i18n-site/a-single-progress-uses-pure-css-to-achieve-animation-effects-2oo) / [китайский](//juejin.cn/post/7413586285954154522).

### IndexedDB с высокой производительностью при одновременной записи

Проект основан на асинхронной обертке IndexedDB [idb](//www.npmjs.com/package/idb).

Чтение и запись IndexedDB являются асинхронными. При создании индекса документы будут загружаться одновременно для создания индекса.

Чтобы избежать частичной потери данных из-за конкурентных записей, можно использовать следующий код на `coffeescript`, добавив кэш `ing` между чтением и записью для перехвата конкурентных записей.

```coffee
pusher = =>
  ing = new Map()
  (table, id, val)=>
    id_set = ing.get(id)
    if id_set
      id_set.add val
      return

    id_set = new Set([val])
    ing.set id, id_set
    pre = await table.get(id)
    li = pre?.li or []

    loop
      to_add = [...id_set]
      li.push(...to_add)
      await table.put({id,li})
      for i from to_add
        id_set.delete i
      if not id_set.size
        ing.delete id
        break
    return

rindexPush = pusher()
prefixPush = pusher()
```

## Поиск по префиксу в реальном времени

Чтобы отображать результаты поиска по мере ввода пользователем, например, при вводе `wor`, показываются слова с префиксом `wor`, такие как `words` и `work`.

![](//p.3ti.site/1727684944.avif)

Ядро поиска использует таблицу `prefix` для последнего слова после сегментации, чтобы найти все слова с его префиксом и выполнить поиск последовательно.

Функция `debounce` также используется во внешнем взаимодействии (реализация ниже) для уменьшения частоты ввода данных пользователем, запускающих поиск, и уменьшения объема вычислений.

```js
export default (wait, func) => {
  var timeout;
  return function(...args) {
    clearTimeout(timeout);
    timeout = setTimeout(func.bind(this, ...args), wait);
  };
}
```

## Точность и полнота

Поиск сначала сегментирует введенные пользователем ключевые слова.

Предположим, что после сегментации получено `N` слов. При возвращении результатов сначала отображаются результаты, содержащие все ключевые слова, затем результаты с `N-1`, `N-2`, ..., `1` ключевыми словами.

Первоначальные результаты поиска обеспечивают точность запроса, а последующие результаты (при нажатии на кнопку «Загрузить еще») обеспечивают полноту.

![](//p.3ti.site/1727684564.avif)

## Загрузка по требованию

Для повышения скорости ответа используется генератор `yield`, который реализует загрузку по требованию, возвращая результат `limit` раз при запросе.

Обратите внимание, что после `yield` при повторном поиске необходимо заново открывать транзакцию запроса `IndexedDB`.

## Доступно офлайн

В индексной таблице не хранится исходный текст, а только слова, что уменьшает объем памяти.

Для выделения результатов поиска требуется перезагрузка исходного текста, а соответствие `service worker` позволяет избежать повторных сетевых запросов.

В то же время, поскольку `service worker` кэширует все статьи, как только пользователь выполняет поиск, весь веб-сайт, включая搜索, становится доступен в автономном режиме.

## Итог

Полностью фронтендный инвертированный полнотекстовый поиск, не требующий сервера. Идеально подходит для документов, личных блогов и других малых и средних сайтов.

`i18n.site` предлагает открытый исходный код для собственного чисто фронтендного поиска, который компактен и быстр в ответе, решая многие проблемы текущих фронтендных полнотекстовых поисков и предоставляя лучший пользовательский опыт.
