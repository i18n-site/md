## पासवर्ड मुक्त लॉगिन कॉन्फ़िगर करीं

कुछ मिनट बाद सर्वर सब सक्रिय हो गईल।

सबसे IP स्थानीय मशीन के `~/.ssh/config` संपादित क के एगो उपनाम सेट कईल जाला, अवुरी ओकरा बाद पासवर्ड मुक्त लॉगिन के कॉन्फ़िगर करे खाती `ssh-copy-id root@c2` इस्तेमाल कईल जाला।

![](https://i-01.eu.org/1718287198.avif)

रउआँ एह [ssh-copy-id.sh](https://github.com/i18n-ops/ubuntu/blob/main/ssh-copy-id.sh) संशोधित क के इनिशियलाइजेशन कॉन्फ़िगरेशन के तेजी से पूरा क सकत बानी।

पासवर्ड मुक्त कॉन्फ़िगरेशन पूरा होखला के बाद, आप एकर इस्तेमाल कर सकतानी एकरा के बैच में संचालित कईल जा सकता, जवन कि सुविधाजनक बा [pdsh](https://github.com/chaos/pdsh)

पहिला बा कि मैन्युअल रूप से ipv6 के सक्षम कइल जाव आ परीक्षण कइल जाव

```bash
pdsh -w 'c0 c1 c2' 'bash -i -c enable_ipv6'
pdsh -w 'c0 c1 c2' 'curl -s 6.ipw.cn;echo'
```

![](https://i-01.eu.org/1718287969.avif)

एकरा बाद एक क्लिक से सिस्टम के उबंटू 24 में अपग्रेड करीं आ आमतौर पर इस्तेमाल होखे वाला सॉफ्टवेयर इंस्टॉल करीं।

```bash
pdsh -w 'c0 c1 c2' 'curl -s https://raw.githubusercontent.com/i18n-ops/ubuntu/main/boot.sh | CN=1 bash'
```

CN=1 के मतलब होला भाषा चीनी में आ समय क्षेत्र के बीजिंग में बदलल।

डिफ़ॉल्ट रूप से इंस्टॉल सॉफ्टवेयर हमरा निजी उपयोग के आदत के अनुसार कॉन्फ़िगर कइल गइल बा, जइसे कि [mise](https://github.com/jdx/mise) , [direnv](https://direnv.net/) , आदि जरूरत पड़ला पर रउआ [i18n-ops/ubuntu](https://github.com/i18n-ops/ubuntu) गोदाम के कॉपी कर सकेनी आ जरूरत के हिसाब से संशोधित कर सकेनी।

एह कदम में बहुत समय लागेला आधा घंटा इंतजार कईला के बाद आमतौर प इस्तेमाल होखेवाला सॉफ्टवेयर के अपग्रेड अवुरी इंस्टॉलेशन सफल हो जाई। रउआँ एकर पुष्टि निम्नलिखित कमांड से कर सकत बानी:

```bash
pdsh -w 'c0 c1 c2' 'cat /etc/issue'
```

## मोश ssh के गति बढ़ावेला

यूरोप से चीन तक के नेटवर्क स्थिर नइखे एकर इस्तेमाल `ssh` गति बढ़ावे खातिर कइल जाला [mosh](//mosh.org)

स्थानीय डाइरेक्टरी `~/.bin` (मुख्य रूप से पर्यावरण चर `PATH` में जोड़ल गइल) में `c0` स्क्रिप्ट निम्नलिखित तरीका से बनाईं:

```bash
#!/usr/bin/env bash

if [ -z "$1" ]; then
arg=/root/.tmux_default
else
arg=${@:1:$#}
fi

vps=$(basename $0)
set -ex
mosh $vps -p 60000:60009 $arg
```

बनावे के बाद, एक्जीक्यूटिव अनुमति आ सॉफ्ट-लिंक `c1` आ `c2` प्रदान करीं।

```bash
chmod +x c0
ln -s c0 c1
ln -s c0 c2
```

भविष्य में रउआ सीधे `c0` आ `c1` डाल के `mosh` हाई स्पीड डायरेक्ट कनेक्शन के इस्तेमाल कर सकेनी आ [tmux](https://www.ruanyifeng.com/blog/2019/10/tmux.html)

## विभाजन प्रारूप के ext4 से btrfs में बदलीं

[btrfs](https://wikipedia.org/wiki/Btrfs) आसान बैकअप खातिर डिस्क स्नैपशॉट ले सकत बानी।

पहिले बचाव प्रणाली में प्रवेश करीं

![](https://i-01.eu.org/1718300448.avif)

डेबियन बचाव के चयन करीं

![](https://i-01.eu.org/1718300753.avif)

सर्वर के रिस्टार्ट होखे के इंतजार करीं

फिंगरप्रिंट में बदलाव ssh , एकरा के साफ करे खातिर `ssh-keygen -R ip` इस्तेमाल करीं

एकरा बाद विभाजन के बदले खातिर निम्नलिखित स्क्रिप्ट चलाईं

```bash
nohup bash -c "bash <(curl -s https://raw.githubusercontent.com/i18n-ops/ops/main/contabo/ext4_btrfs.sh) >/tmp/nohup.out 2>&1 && reboot" &
sleep 1 && tail -f /tmp/nohup.out
```

रूपांतरण सफल होखला के बाद स्क्रिप्ट अपने आप सिस्टम के रिस्टार्ट क दिही।

### btrfs संपीड़न अनुपात देखल जाव

ई डिफ़ॉल्ट रूप से स्वचालित रूप से संपीड़ित हो जाई संपीड़न अनुपात के जांच करे खातिर कमांड निम्नलिखित बा:

```bash
if ! command -v compsize &>/dev/null; then
apt install -y  btrfs-compsize
fi
compsize -x /
```

### मैनुअल संपीड़न के काम कइल जाला

अगर रउआँ परफेक्शनिस्ट बानी त मैन्युअल रूप से भी पूरा डिस्क संपीड़न क सकत बानी कमांड निम्नलिखित बा:

```bash
btrfs filesystem defragment -r -czstd /
```

जईसे कि नीचे दिहल आकृति में देखावल गईल बा, आप देख सकतानी कि नाया सिस्टम में मैनुअल संपीड़न के बाद लगभग 100MB अवुरी संपीड़न भईल।

![](https://i-01.eu.org/2023/12/8NWlr4c.webp)

### स्नैपर के साथ Btrfs स्नैपशॉट बैकअप बनाईं

Btrfs बैकअप खातिर सबसे बढ़िया साथी बा [snapper](https://manual.siduction.org/sys-admin-btrfs-snapper_en.html)

इनिशियलाइजेशन स्क्रिप्ट `~/ubuntu` डाइरेक्टरी बनाई, बस निम्नलिखित कमांड चलाईं:

```
~/ubuntu/snapper/init.sh
```

स्क्रिप्ट के सामग्री में मिलब [github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh](https://github.com/i18n-ops/ubuntu/blob/main/snapper/init.sh)

`snapper list-configs` इस्तेमाल ओह सबवॉल्यूम सभ के देखे खातिर करीं जिनहन खातिर स्नैपशॉट सेट कइल गइल बा।

![](https://i-01.eu.org/1718302689.avif)

उपखंड के स्नैपशॉट देखे खातिर `snapper -c mnt list` इस्तेमाल करीं

`/etc/snapper/configs/mnt` एह सबवॉल्यूम के विशिष्ट बैकअप सेटिंग देख सकेला

अउरी संदर्भ दिहल गइल बा:

[Btrfs के स्नैपशॉट फंक्शन के साथ आसानी से खेलल जा सकेला snapper](https://zhuanlan.zhihu.com/p/31082518)

## बचाव प्रणाली के विवरण दिहल गइल बा

बचाव 22 में प्रवेश करीं पोर्ट के [tcping](https://github.com/paradiseduo/tcping) सुलभ होखला के बाद, रउआ एकरा के सेट पासवर्ड से 2-3 मिनट तक एक्सेस ना कर पइब बस एक पल इंतजार करीं आ ठीक हो जाई।

```
tcping ip 22 -c 9999
```

बंदरगाह के पहुँच पर नजर राखल जा सकेला.

### & में प्रवेश कइला के बाद हार्ड ड्राइव तक कइसे पहुँचल जाला रिस्टोर स्नैपशॉट

```
mkdir -p /mnt/os
mount  /dev/sda3 /mnt/os -t btrfs -o "defaults,ssd,discard,noatime,compress=zstd:3,space_cache=v2"
mount --bind /dev /mnt/os/dev
mount --bind /proc /mnt/os/proc
mount --bind /sys /mnt/os/sys
mount --bind /run /mnt/os/run
chroot /mnt/os zsh -c "mount -a && dbus-daemon --system 2>/dev/null; exec zsh"
```

उदाहरण खातिर रउआँ स्नैपशॉट देख सकत बानी

```
snapper -c etc list
```

स्नैपशॉट फाइल में बदलाव देखल जाव

```
snapper -c etc status 5..6
```

विशिष्ट अंतर पाठ देखल जाव

```
snapper -c etc diff 3..5
```

स्नैपशॉट के रिस्टोर करीं: उदाहरण खातिर, स्नैपशॉट 5 के बाद के सभ बदलाव के पूर्ववत करीं

```
snapper -c etc undochange 1..0
snapper -c disk undochange 1..0
```

### लॉग स्थायित्व सक्षम करीं

रेस्क्यू सिस्टम के तहत लॉग स्थायित्व सक्षम करे खातिर रउआँ के मैन्युअल रूप से संबंधित डाइरेक्टरी बनावे के पड़े ला आ कुछ अतिरिक्त स्टेप करे के पड़े ला, काहें से कि रउआँ के लगे सिस्टमड सेवा चलत ना हो सके ला। इहाँ स्टेप बा जवना के रउआ आजमा सकत बानी:

पहिले क्रोट, तब

```
mkdir -p /var/log/journal
chown root:systemd-journal /var/log/journal
chmod 2755 /var/log/journal
sed -i 's/^#\?\s*Storage=.*/Storage=persistent/' /etc/systemd/journald.conf
```

अपना उबंटू सिस्टम में रिबूट करीं: उपर दिहल बदलाव कइला के बाद, रउआ अपना मशीन के सामान्य रूप से आ रेस्क्यू सिस्टम के बजाय अपना उबंटू सिस्टम में रिबूट करे के पड़ी।

एह तरीका से, systemd-journald नया कॉन्फ़िगरेशन से शुरू करी आ हार्ड डिस्क पर लॉग के लगातार स्टोर करे शुरू करी।

ध्यान रहे कि उपर दिहल ऑपरेशन रेस्क्यू सिस्टम पर कइल जाला, एहसे ऊ बदलाव तबले प्रभावी ना होखी जबले सिस्टम रिबूट ना हो जाई आ अब रेस्क्यू मोड में ना रह जाई.

एकर मतलब ई बा कि रउआँ के तुरंत मौजूदा लॉग सभ में कौनों अइसन बदलाव ना लउकी जे लगातार हो जाई, बाकी एकरे बाद पैदा भइल सगरी लॉग सभ स्थायी होखी।

एह तरीका से रउआँ ssh सेवा के लॉग देख सकत बानी

```
journalctl --directory=/var/log/journal  _SYSTEMD_UNIT=ssh.service
```

## कठिन आ जटिल बेमारी पर नोट

### अजीब समस्या के हल करीं: सर्वर ssh नइखे कर सकत

हम पहिले एगो ऑटोमैटिक संकलन आ डिप्लोयमेंट स्क्रिप्ट बनावे खातिर github एक्शन के इस्तेमाल कइले रहीं, बाकिर पावल कि डिप्लोयमेंट सेवा ssh ना हो सके.

बचाव प्रणाली में प्रवेश आ लॉग देखल अनुमति के मुद्दा बा:

`Missing privilege separation directory: /run/sshd`

रउआँ निम्नलिखित कमांड से ई डाइरेक्टरी बना सकत बानी

```
chown root:root /
systemd-tmpfiles --create
```

बार-बार जांच के बाद एकर कारण बा github action tar के यूजर `uid` `1001` बा

अनपैकिंग होखे के tar अगर कवनो पैरामीटर `--no-same-owner` ना होखे, एकरा के `1001` यूजर खातिर `uid` पर डिकम्प्रेस कइल जाई

एकरे परिणाम के रूप में, rsync रूट डाइरेक्टरी के यूजर ग्रुप में बदलाव करी, आ फिर रउआँ रिस्टार्ट कइला के बाद ssh में लॉग इन ना कर पइब।

डिकम्प्रेसन स्क्रिप्ट के निम्नलिखित तरीका से संशोधित करीं:

